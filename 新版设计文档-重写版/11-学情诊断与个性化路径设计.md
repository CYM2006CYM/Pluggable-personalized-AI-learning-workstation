## 1. 阅读规则与状态标签

本文同时描述当前基础框架和六周新增设计，必须使用以下标签区分，不能把规划接口写成当前已有能力。

- **[已有]**：`pi-study-helper` 当前源码和测试已经具备。
- **[六周新增]**：六周最小可行产品必须实现。
- **[候选]**：只有原型或调研通过后才采用，不阻塞主闭环。
- **[延期]**：六周内不实现。

正文使用中文。真实目录、文件名、接口、类型、方法、字段和错误码保留英文，以便与实现精确对应；首次出现时给出中文解释。

本文反复使用的四个版本标识不能混淆：

| 字段 | 中文含义 | 何时变化 | 解决的问题 |
|---|---|---|---|
| `profileRevision` | 资料包修订号 | 维护者启用新版资料包时递增 | 防止进行中的会话混用新旧知识点、题目或规则 |
| `sessionVersion` | 会话比较写版本 | 每次成功修改会话后递增 | 防止网页、终端或重复请求覆盖较新的会话状态 |
| `evidenceVersion` | 正式学习证据版本 | 每写入一条新的有效正式证据后递增 | 标明知识状态是根据哪一批事实计算的 |
| `pathVersion` | 路径快照版本 | 首次确认或正式重算路径时递增 | 保留每次路径变化，避免原地改写历史路径 |

`mastery` 表示根据有效证据估算的知识点掌握度，范围为 `0..1`；`confidence` 表示这项估算有多少证据支撑，范围同样为 `0..1`。高掌握度不自动等于高置信度：只做对一道题的学习者可以答对，但证据仍然不足。

## 2. 目标、用户和非目标

### 2.1 要回答的三个问题

系统必须在少量、可复核的证据下回答：

1. 学习者对目标所需的哪些知识点已经有直接证据；
2. 哪些知识点需要支持，哪些只是尚未验证；
3. 在目标、先修关系、时间预算和资料包能力范围内，下一步应该学习或练习什么。

输出是可解释、可恢复的 `LearnerDiagnostic`（学习者诊断快照）、`KnowledgeState`（知识点状态）和 `LearningPath`（学习路径快照），不是心理测量结果，也不声称精确预测长期能力。

### 2.2 六周范围

- Pandas 数据清洗是唯一完整示范领域。
- Profile 第2版提供固定诊断、知识点有向无环先修图、活动、目标和评分资产。
- 章节学习和系统推荐共用同一个路径引擎；目标学习、快速实战和查漏补缺保留为未来入口扩展，不进入Pandas六周主界面。
- 路径、评分和状态更新由确定性 TypeScript 代码完成。
- 诊断智能体和学习决策智能体只生成解释；失败时使用固定中文模板。

### 2.3 明确不做

- **[延期]** 不训练或微调模型，不建设图数据库，不做心理学量表或正式能力认证。
- **[六周新增]** CIDPP用于评价讲义、普通题、代码注释和任务设计质量，不替换知识点状态；评分主要服务内容优化和人工参考。
- **[六周新增][已冻结]** 学习者Pandas五维在事实掌握度底座上正式运行；诊断后生成初始画像，学习单元完成后异步更新。AI直接评分可观察维度，程序校验范围、证据引用、版本和幂等；失败保留旧画像。
- **[延期]** 不允许智能体自动修改知识点先修图、权重、阈值和评分规则。
- **[延期]** 不建立另一套“推荐服务”或并行智能体路由。

## 3. 与当前基础框架的适配边界

### 3.1 当前可复用内容

当前 `pi-study-helper` 已有：

- `StudySession`（学习会话）：保存学科、模式、范围、题目统计和生命周期；
- `Attempt`（作答记录）：保存题目、用户提交、订正历史、最终结果和知识点；
- `SessionEvidence`（会话证据）：由会话和作答确定性提取观察事实、最终答对证据和未验证主题；
- `LearningProfile`（长期学习画像）：跨会话保存总题数、准确率、强弱项和建议；
- `PrivateMemoryRepository`（私有学习记录仓储）：保存 pending 批次、作答、总结和长期画像；
- `practice`、`card_practice`、`chapter_study` 三种当前学习模式；
- 单选、判断和简答的现有展示/判题基础，以及本地原子写和恢复基础。

当前代码**没有**固定初始诊断、知识点掌握度、置信度、路径引擎、路径快照、路径重算或网页会话版本控制。所有这些都必须标为[六周新增]。

还需特别注意：当前单选、判断和简答均有展示基础，当前旧流程的已提交答案统一调用 `study_grade_answer` 语义判定；[六周新增]新版固定诊断和Pandas学习后单选/判断改为服务端比较 `correct_answer`，简答暂不进入Pandas v2新主链。当前 `Attempt` 会在补充讨论摘要后覆盖同一个题目文件，`SessionEvidence` 也是运行时派生投影，二者都不是本文规划的不可变 Evidence 事实仓。当前 `/study-recover` 只能处理未完成学习批次的补总结或中断，不能恢复新版诊断和路径检查点。

### 3.2 类型适配矩阵

| 当前类型或能力 | [六周新增] 类型或能力 | 适配方法 | 不能直接复用的内容 |
|---|---|---|---|
| `StudySession` | 新版会话快照 | 保留旧字段，在新版会话仓储中补充四个版本、目标和当前阶段 | 旧会话没有诊断、路径和比较写版本 |
| `Attempt` | `Evidence` | 通过明确适配器把旧版最终结果转换为较低权重兼容证据 | 不能把总正确率平均分配给所有知识点 |
| `SessionEvidence` | 通用证据集合 | 复用确定性摘录和知识点映射原则 | `final_answer_accepted` 不足以表达代码、实操、提示和部分得分 |
| `LearningProfile` | `KnowledgeState[]` | 继续保存为学习者可读摘要；知识状态单独计算和保存 | 强弱项自然语言不能作为路径引擎事实输入 |
| 三种 `ReviewMode` | 章节学习、系统推荐及未来入口 | Pandas六周只开放章节学习和系统推荐，不直接破坏旧枚举和测试 | 卡片回忆是活动偏好，不是另一套路径算法 |
| Profile 第1版 | Profile 第2版诊断/路径资产 | 第1版继续旧流程；第2版声明能力后进入新闭环 | 不能假设第1版有目标、诊断、活动和先修图 |
| `PrivateMemoryRepository` | `LearningSessionRepository` | 保留旧 `_user/summaries/`；新增仓储管理新版会话 | 不把新版文件强塞进旧批次事务导致旧恢复失效 |

### 3.3 新旧画像的关系

`LearningProfile` 是兼容摘要，适合回答“最近总体表现怎样”；`KnowledgeState` 是确定性派生数据，适合回答“某个知识点依据哪些证据处于什么状态”。应用层可以根据知识状态生成新的画像摘要，但路径引擎只能读取知识状态及其证据版本，不能读取智能体写出的 `weak_points` 后直接排路径。

### 4.1 诊断蓝图结构

以下是[六周新增]的实现契约，关键字段必须在资料包启用前校验：

```ts
type DiagnosticQuestionKind =
  | "single_choice"      // 单选题；代码按固定答案判分
  | "judgment"           // 判断题；代码按固定布尔答案判分
  | "code_reading"       // 代码阅读题；不执行用户代码
  | "output_prediction"; // 固定输出预测题；不依赖Python执行器

interface DiagnosticBlueprint {
  blueprintId: string;          // 蓝图稳定编号；同一revision内唯一
  profileRevision: number;      // 蓝图所属资料包修订号
  goalIds: string[];            // 适用的学习目标编号；至少一个
  estimatedMinutes: number;     // 预计完成分钟数；Pandas默认不超过10
  minimumCoverage: number;      // 完成诊断所需题目覆盖比例，范围0..1
  questions: DiagnosticQuestion[]; // 固定题目；Pandas为8至12题
  scoringVersion: string;       // 评分规则版本，例如diagnostic-v1
}

interface DiagnosticQuestion {
  questionId: string;           // 题目稳定编号；也用于幂等保存答案
  knowledgePointId: string;     // 本题主要测量的一个知识点
  kind: DiagnosticQuestionKind; // 题型；六周不开放multi_choice
  difficulty: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 沿用当前五档难度
  prompt: string;               // 可发送给学习者的安全题干
  options?: string[];           // 仅单选题需要；不包含答案标记
  maxScore: number;             // 原始满分；必须大于0
  required: boolean;            // 是否属于完成诊断所需的关键题
  evaluatorRef: string;         // 服务端评分资产引用；不得下发前端
  sourceAnchorIds: string[];    // 题目事实所依据的资料包来源锚点
}

interface DiagnosticAnswerRecord {
  questionId: string;           // 对应蓝图中的固定题目编号
  requestId: string;            // 幂等请求编号；网络重试不得更换
  status: "answered" | "skipped"; // answered已判分，skipped仅表示未验证
  submittedAnswer?: string;     // 学习者私有答案；不进入Agent解释上下文
  normalizedScore?: number;     // 服务端确定性得分0..1；跳过时省略
  evidenceId?: string;          // answered成功落盘后关联的正式证据
  evaluatorVersion?: string;    // 实际使用的固定评分器版本
  createdAt: string;            // 服务端生成的ISO 8601时间
}
```

`single_choice`、`judgment`、`code_reading` 和 `output_prediction` 分别表示单选、判断、代码阅读和固定输出预测。当前 `QuestionType` 虽预留 `multi_choice`，但生成、资料构建和终端没有完整支持，因此诊断蓝图不得使用多选题。

知识点可选字段 `requiresCodeEvidence: boolean` 表示“这个知识点只有在存在正式代码提交或实操证据时，才允许压缩先修”。该字段属于知识点本身，不随学习目标变化；未声明时默认为 `false`。这样同一个 `KnowledgeState` 不会因为目标不同产生两套 `skipEligible`。

## 5. 完整诊断流程

### 5.1 主调用链

```text
startSession()                 # 创建会话，锁定profileRevision和初始sessionVersion
  → saveDiagnosticDraft()      # 保存背景问卷、当前题号和未完成答案草稿
  → submitDiagnosticAnswer()   # 服务端确定性判分，幂等保存答案和正式证据
  → completeDiagnostic()       # 检查最低覆盖，聚合KnowledgeState并保存诊断快照
  → buildPath()                # 根据目标、先修图、状态和预算生成候选路径
  → confirmPath()              # 保存不可变正式路径并进入首个available节点
```

任何一步失败都不能假装下一步已经完成。诊断解释失败不等于诊断计算失败；前者使用固定模板，后者必须返回错误并保留已保存答案。

### 5.2 背景问卷

背景问卷只收集：Python 经验、Pandas 经验、学习目标、可用分钟数和学习偏好。每项允许选择“不确定”。这些值用于：

- 决定是否显示更详细的解释；
- 在多个同样合法的活动间选择呈现方式；
- 约束时间预算；
- 选择系统推荐或章节学习入口；快速实战、查漏补缺等入口留作未来扩展。

自评不是直接能力证据，不参与 `mastery`。未知值保存为明确的 `unknown`，不能用数字 `0` 代替，否则会把“没有回答”误写成“能力为零”。

### 5.3 固定诊断题

- Pandas 示例诊断为8至12题，总时长不超过10分钟。
- 一题一页，支持上一题、保存、跳过和退出后恢复。
- 题目优先使用确定性单选、判断、代码阅读和固定输出预测。
- 跳过保存为诊断答案状态 `skipped`，表示未验证，不生成负分证据。
- `required=true` 的关键题可以跳过，但诊断结果必须标记相应知识点证据不足。
- 浏览器只取得题干、选项和安全知识点标签；正确答案、分值映射、权重、阈值和评分资产只在 Node.js 服务端读取。
- 同一 `questionId + requestId` 重复提交返回第一次保存的结果，不重复生成证据。

### 5.4 诊断完成条件

`completeDiagnostic()` 只有在以下条件满足后才能生成 `completed` 快照：

1. 所有已提交答案都通过结构校验和确定性判分；
2. 已完成题目数达到蓝图 `minimumCoverage`；
3. 每个缺失的关键知识点都明确记录为 `unverified`；
4. 所用蓝图和当前会话的 `profileRevision` 一致；
5. 所有正式证据已经原子写入并具有连续 `evidenceVersion`。

诊断状态与含义：

| 状态 | 中文含义 | 允许操作 |
|---|---|---|
| `not_started` | 尚未开始固定诊断 | 填问卷或开始第一题 |
| `in_progress` | 已保存部分答案 | 继续、返回、跳过或退出 |
| `ready_to_complete` | 达到最低覆盖且答案已判定 | 完成聚合或继续答题 |
| `completed` | 已保存不可变诊断快照 | 生成路径；重新诊断需创建新版本 |
| `interrupted` | 写入或进程中断但草稿仍可恢复 | 从最后检查点恢复 |

诊断不完整时可以保存退出，但不能生成“完整诊断”标签。关键证据不足时，路径引擎插入短验证节点或采用保守基础路径。

### 5.5 诊断智能体边界

Diagnosis Agent（诊断解释智能体）只接收已经裁剪的知识状态、证据类型、原因码和安全来源摘要，用于生成：

- 学习者能理解的暂时结论；
- “为何这样判断”的证据说明；
- 最多两个不影响分数的澄清问题。

它不能看到正确答案和隐藏评分资产，不能输出新的 `mastery`、`confidence` 或路径节点。超时、拒答或结构无效时，应用层使用固定模板：“根据已完成的固定诊断，你在……已有直接证据；……仍需练习；……目前证据不足。”

## 6. 统一学习证据模型

### 6.1 类型定义

```ts
type EvidenceKind =
  | "diagnostic"        // 固定初始诊断；不是LearningActivity
  | "mcq"               // 正式活动：客观题
  | "code_completion"   // 正式活动：代码补全
  | "coding_practical"  // 正式活动：独立实操
  | "explain"           // 正式活动：解释题或迁移说明
  | "debug"             // 正式活动：代码调试
  | "legacy"            // 当前Attempt转换的兼容证据
  | "interaction";      // 提问、求助、提示、自评或公开运行

type EvidenceSource =
  | "fixed_diagnostic"          // 固定诊断的确定性判分
  | "deterministic_quiz"        // 客观题的确定性判分
  | "code_submit"               // 正式代码提交及公开/隐藏测试
  | "practical_rubric"          // 独立实操的Rubric评分
  | "legacy_final_answer"       // 当前Attempt最终答对的兼容映射
  | "self_report"               // 背景问卷自评
  | "context_question"          // 上下文提问
  | "hint_view"                 // 查看提示或示例
  | "public_run"                // 浏览器公开运行；不计掌握度
  | "evaluator_error";          // 评测器异常；不归因于学习者

type EvidenceImpact =
  | "mastery"  // 允许参与掌握度和置信度计算
  | "soft"     // 只影响讲解、偏好或后续验证，不直接改分
  | "none";    // 仅用于运行审计，不影响学情

type EvidenceForm =
  | "selected_response" // 单选、判断等选定答案形式
  | "code_reasoning"    // 代码阅读、输出预测或解释形式
  | "code_execution"    // 正式代码提交形式
  | "practical_rubric"  // 独立实操Rubric形式
  | "legacy_attempt"    // 当前Attempt兼容形式
  | "interaction";      // 软交互或审计形式；不计证据多样性

interface Evidence {
  evidenceId: string;            // 证据唯一编号；创建后不可修改
  requestId: string;             // 创建本记录的幂等请求编号；用于崩溃补提交
  sessionId: string;             // 产生证据的学习会话编号
  knowledgePointId: string;      // Profile内稳定知识点编号
  profileRevision: number;       // 产生证据时锁定的资料包修订号
  evidenceVersion?: number;      // 仅正式mastery证据提交后分配；soft/none省略
  kind: EvidenceKind;            // 教学活动类型
  source: EvidenceSource;        // 确定性判定来源
  form: EvidenceForm;            // 作答/验证形式；防止只换来源刷证据多样性
  impact: EvidenceImpact;        // 是否参与掌握度计算
  outcome: "correct" | "partial" | "incorrect" | "unverifiable"; // 观察结果；跳过不建Evidence
  score?: number;                // 标准化得分0..1；soft/none可以省略
  difficulty?: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 无难度的软交互可省略
  independence: "independent" | "hinted" | "worked_example" | "answer_exposed"; // 最高帮助等级
  activityId?: string;           // 来源活动编号；背景问卷等无活动时省略
  attemptId?: string;            // 正式提交幂等编号；同一编号只计一次
  evaluatorVersion?: string;     // 判题器、测试或Rubric版本
  createdAt: string;             // ISO 8601时间；由服务端生成
}
```

`mastery`、`soft`、`none` 分别表示正式计分证据、软证据和纯审计记录。`Evidence` 是统一观察记录，不等于“每条都计分”；只有 `impact="mastery"`、`score` 有效且来源权重大于零的记录才进入聚合器。`mcq/code_completion/coding_practical/explain/debug` 是与12号文档一致的五类正式 `LearningActivity`；`reading/quiz/code/practice` 是02号文档中的资料包能力模态，不是另一套活动枚举。卡片、无分自测、提问和提示也不能伪装成新的正式计分活动。

### 6.2 证据来源和权重

| `source` | 中文含义 | 默认权重 | 对掌握度的影响 |
|---|---|---:|---|
| `fixed_diagnostic` | 固定诊断 | 1.0 | 正式直接证据 |
| `deterministic_quiz` | 正式客观题 | 1.0 | 正式直接证据 |
| `code_submit` | 正式代码提交 | 1.2 | 正式强证据 |
| `practical_rubric` | 独立实操评分 | 1.2 | 正式强证据 |
| `legacy_final_answer` | 旧版最终答对记录 | 0.6 | 兼容证据；不能单独形成高置信度掌握 |
| `self_report` | 学习者自评 | 0 | 只影响讲解偏好 |
| `context_question` | 普通提问 | 0 | 不能因提问推断能力弱 |
| `hint_view` | 查看提示 | 0 | 本身不改分；正式提交通过独立性折扣反映 |
| `public_run` | 公开预览运行 | 0 | 只保存可选运行轨迹 |
| `evaluator_error` | 评测器异常 | 0 | 不产生学习者负证据 |

旧 `Attempt` 只有在 `outcome="correct"` 且有明确知识点时，才可映射成 `legacy_final_answer`。`gave_up` 明确表示学习者主动放弃，只形成未验证记录；旧结构没有独立、可计分的正式错误结果，不能把放弃或答错后订正的过程直接转换成零分掌握证据。

### 6.3 不可变、幂等和安全规则

1. 正式证据写入后不可原地修改；勘正只能写新证据并引用被替代编号。
2. 同一 `attemptId` 只允许产生一条正式计分证据；重复请求返回原结果。
3. 诊断跳过只写 `DiagnosticAnswerRecord.status="skipped"` 和诊断快照的未回答知识点，不创建 Evidence。旧版明确放弃可以写 `outcome="unverifiable"` 的兼容软记录，但不产生零分。
4. `public_run`、自评、提问、查看提示和 `evaluator_error` 可按需要写 `impact="soft"/"none"` 的审计记录，但不分配 `evidenceVersion`、不改变掌握度。
5. 正确答案、隐藏测试、完整 Rubric、用户完整私密回答和应用程序接口密钥不进入前端 DTO、智能体输入或普通轨迹。
6. 证据必须携带 `profileRevision`；旧修订证据可用于历史展示，是否迁移到新版知识点必须有明确映射，不能按标题猜测。

## 7. 掌握度与置信度计算

### 7.1 版本与输入过滤

聚合规则版本固定为 `knowledge-state-v1`。第2周只能使用开发集冻结参数；正式60案例不得用于反复调参。

计算前依次过滤：知识点编号匹配、资料包修订兼容、`impact="mastery"`、`score` 在 `0..1`、判题成功、来源权重大于零、独立性权重大于零、未被同一 `attemptId` 去重。越界分数返回 `evidence_invalid`，不能静默截断。过滤后按 `createdAt` 降序排列，同一时间使用 `evidenceId` 稳定排序，只取最近7条作为统一计算集合；`consideredEvidenceIds`、掌握度、有效证据数、一致性/冲突窗口和证据形式数都只能使用这一集合。完整历史仍保留在 `evidenceIds`；`answer_exposed` 仅用于审计，不占最近7条名额。若过滤后没有证据，`mastery=null` 而不是 `0`，状态必须为 `unverified`。

### 7.2 掌握度公式

```text
mastery
  = Σ(score × sourceWeight × independenceWeight × recencyWeight)
    / Σ(sourceWeight × independenceWeight × recencyWeight)
```

变量解释：

- `score`：单条正式证据的标准化结果，范围为 `0..1`；
- `sourceWeight`：表6.2中的来源可靠度；
- `independenceWeight`：独立完成程度的折扣；
- `recencyWeight`：旧证据的有限时间折扣；
- `mastery`：加权估计，不是考试原始分，也不能脱离置信度展示为精确能力。

独立性权重：

| `independence` | 中文含义 | 权重 |
|---|---|---:|
| `independent` | 未查看提示或示例独立完成 | 1.0 |
| `hinted` | 查看局部提示后完成 | 0.8 |
| `worked_example` | 查看完整范例后完成 | 0.5 |
| `answer_exposed` | 已看到答案或参考实现 | 0；本次提交不计掌握度 |

时间权重按 `ageDays` 确定：`0≤ageDays≤30` 为 `1.0`，`31≤ageDays≤90` 为 `0.9`，`ageDays≥91` 为 `0.8`。六周演示中的证据通常都处于30天内，但实现仍需固定规则，不能由智能体自由衰减。

计算快照必须保存服务端 `asOf`（计算基准时间）。`ageDays=floor((asOf-createdAt)/86400000)`；`createdAt` 只能由服务端生成。损坏或未来时间戳隔离并记录 `evidence_invalid`，不静默修正后参加正式聚合。内部使用未四舍五入的 JavaScript `number` 比较阈值，序列化至少保留6位小数；只有界面把结果转换成百分数。

### 7.3 置信度公式

```text
base = min(0.75, 0.25 × log2(1 + min(validEvidenceCount, 7)))
confidence = clamp(base + consistencyBonus - conflictPenalty, 0, 1)
```

- `validEvidenceCount`：最近7条中通过过滤、幂等去重且独立性权重大于零的有效直接证据数量；避免刷题无限抬高置信度。
- `consistencyBonus`：有效证据至少两条时，取最近 `min(3, validEvidenceCount)` 条；最高分与最低分之差不超过 `0.2` 时取 `0.1`，否则为 `0`。
- `conflictPenalty`：同一最近窗口中任意两条有效证据分差达到 `0.5` 时取 `0.2`，否则为 `0`。冲突优先，不能同时获得一致性奖励。
- `clamp`：把结果限制在 `0..1`。
- 无有效直接证据时不计算上述奖励，直接令 `confidence=0`。

虽然字段类型允许 `confidence` 到1，但 `knowledge-state-v1` 的 `base` 最高0.75、奖励最高0.10，因此当前算法实际最高值为0.85。这是有意保留的不确定性，不得额外归一化为1。CIDPP只评价教学内容质量，不进入本公式；Pandas学习者五维方案B另行作为第3周候选上层建议，不生成第二份权威掌握度。

置信度只说明证据充足程度，不是掌握度的第二种叫法。高分但证据少应显示“表现很好，但仍需一次验证”。

### 7.4 五个可手工复算的示例

以下示例均假定证据在30天内；因此 `recencyWeight=1.0`。

**示例一：没有有效证据。** 学习者只填写自评，其来源权重为0。过滤后分子和分母均为空，不能执行除法，因此 `mastery=null`；有效证据数为0，所以 `confidence=0`、状态为 `unverified`、`skipEligible=false`。不能把空分母改写成0分。

**示例二：单条固定诊断答对。** `score=1.0`、来源权重1.0、独立权重1.0。分子和分母均为1，所以 `mastery=1.00`；`base=0.25×log2(2)=0.25`，无一致性奖励和冲突惩罚，所以 `confidence=0.25`。结果状态为 `ready`，证据形式数只有1，`skipEligible=false`；含义是表现很好但仍需一次不同形式验证。

**示例三：两种形式结果一致。** 一条固定诊断选定答案得0.8，来源权重1.0；一条正式代码提交得0.9，来源权重1.2；两条均独立完成。分子为 `0.8×1.0+0.9×1.2=1.88`，分母为 `1.0+1.2=2.2`，所以 `mastery≈0.854545`。`base=0.25×log2(3)≈0.396241`；分差0.1，增加0.1，因此 `confidence≈0.496241`。两条分别属于 `selected_response` 和 `code_execution`；对允许压缩的代码知识点，状态为 `mastered` 且 `skipEligible=true`。

**示例四：两条证据冲突。** 固定诊断得1.0、来源权重1.0；正式代码提交得0.2、来源权重1.2；两条均独立完成。分子为 `1.0×1.0+0.2×1.2=1.24`，分母为2.2，因此 `mastery≈0.563636`。`base≈0.396241`；分差0.8触发0.2惩罚，不加一致性奖励，所以 `confidence≈0.196241`。状态为 `learning`、`skipEligible=false`，并安排验证或回补活动。

**示例五：评测器异常。** 原有一条固定题得0.6，因此分子0.6、分母1.0、`mastery=0.6`，`base=0.25`、`confidence=0.25`、状态为 `learning`。随后代码提交返回 `evaluator_error`；该记录权重为0且 `impact="none"`，不进入分子、分母或有效证据数。最终仍为 `mastery=0.6`、`confidence=0.25`、`learning`，且不得触发路径重算。

## 8. KnowledgeState 与低样本处理

### 8.1 数据结构

```ts
type KnowledgeStatus =
  | "unverified"      // 没有足够直接证据
  | "support_needed"  // 已有证据但掌握度较低
  | "learning"        // 正在形成能力
  | "ready"           // 可进入目标活动，但尚不足以压缩关键先修
  | "mastered";       // 掌握度、置信度和证据多样性均达标

interface KnowledgeState {
  knowledgePointId: string;       // Profile中的稳定知识点编号
  profileRevision: number;        // 对应资料包修订号
  evidenceVersion: number;        // 本状态使用的最后证据版本
  aggregationVersion: "knowledge-state-v1"; // 聚合规则版本
  mastery: number | null;         // 0..1；无有效证据时必须为null
  confidence: number;             // 0..1；证据充足度
  status: KnowledgeStatus;        // 内部知识状态
  validEvidenceCount: number;     // consideredEvidenceIds中的有效直接证据数量
  evidenceFormCount: number;      // consideredEvidenceIds中的不同直接证据形式数量
  evidenceIds: string[];          // 当前revision全部有效mastery证据编号；不含软/审计记录
  consideredEvidenceIds: string[]; // 本次实际参与聚合的最近最多7条证据
  asOf: string;                   // 本次计算的服务端基准时间，保证可复算
  skipEligible: boolean;          // 是否允许压缩这个先修节点
  lastUpdatedAt: string;          // ISO 8601更新时间
}
```

### 8.2 状态进入条件和教学动作

按以下顺序判定，先命中的规则生效：

| 状态 | 确定性进入条件 | 页面文案 | 可跳过先修 | 默认活动和难度 | 后续验证 |
|---|---|---|---|---|---|
| `unverified` | 有效直接证据为0 | 目前没有足够证据 | 否 | 5分钟验证题，S-R或S-U | 获取至少一条直接证据 |
| `support_needed` | 有证据且 `mastery<0.35` | 需要先补基础 | 否 | 讲解、无分自测、客观题，S-R至S-U | 两次低风险练习后再判定 |
| `learning` | `0.35≤mastery<0.70` | 正在学习，还需练习 | 否 | 卡片、解释题、代码补全，S-U至M-U | 增加不同形式证据 |
| `ready` | `mastery≥0.70`，但未满足 mastered 全部条件 | 可以进入目标活动 | 否 | 应用题或补全，M-U至M-A | 用正式代码/实操巩固 |
| `mastered` | `mastery≥0.85`、`confidence≥0.40`、至少两条且两种直接证据 | 已有较充分直接证据 | 视 `skipEligible` | 迁移或综合任务，M-A至C-A | 到期复习或目标任务验证 |

界面可以把五种内部状态折叠为三类：“已证明”对应 `mastered`，“正在学习”对应 `support_needed/learning/ready`，“未验证”对应 `unverified`。界面折叠不能改变内部路径规则。

### 8.3 skipEligible 规则

默认同时满足以下条件才为 `true`：

1. `mastery≥0.85`；
2. `confidence≥0.40`；
3. 至少两条直接证据；
4. 至少两种 `EvidenceForm`，例如选定答案和正式代码执行；诊断单选与普通单选仍是同一种形式；
5. 该知识点不是资料包声明的不可跳过安全/核心门禁；
6. Profile 对该知识点声明 `requiresCodeEvidence=true` 时，最近7条计算证据中至少包含一条 `code_submit` 或 `practical_rubric`。

达到 `mastered` 不等于删除知识点。路径引擎只压缩可选讲解和重复练习；最终任务、资料包明确不可跳过节点和目标验证仍然保留。

不变量：`skipEligible=true` 时 `status` 必须是 `mastered`；反过来不一定成立，因为 `mastered` 知识点仍可能是不可跳过门禁，或缺少资料包要求的代码证据。

## 9. LearnerDiagnostic 诊断快照

```ts
interface LearnerDiagnostic {
  diagnosticId: string;          // 诊断版本唯一编号
  sessionId: string;             // 所属学习会话
  profileRevision: number;       // 锁定的资料包修订号
  diagnosticVersion: number;     // 同一会话重新诊断时递增
  evidenceVersion: number;       // 完成诊断时使用的证据版本
```

重新诊断必须新建 `diagnosticVersion`，旧快照和旧路径保留为历史。`agentExplanation` 失败或缺失不影响 `states` 的有效性。

## 10. 统一入口和旧模式映射

五种[六周新增]入口使用同一 `PathEngine`（路径引擎）：

| 新入口模式 | 中文含义 | 必要输入 | 引擎差异 |
|---|---|---|---|
| `chapter` | 章节学习 | 章节或知识点 | 指定起点并检查先修，不绕过知识图 |
| `recommended` | 系统推荐 | 目标、诊断或已有状态 | 平衡基础、目标和时间预算 |
| `goal` | 目标学习 | 目标编号和期限 | 从目标反向取得全部先修闭包 |
| `task_first` | 快速实战 | 可用时间和目标任务 | 未来扩展；Pandas六周不开放 |
| `gap_review` | 查漏补缺 | 已有直接证据 | 未来扩展；Pandas六周不开放 |

与当前 `ReviewMode` 的适配：

- `chapter_study` 映射到 `chapter`；仍可按原章节菜单开始。
- `practice` 映射到带范围约束的 `recommended` 或 `goal`。
- `card_practice` 表示先进行卡片回忆的活动呈现偏好，不建立第二套路径算法。

Profile 第1版没有新能力声明时，继续调用旧控制器；Profile 第2版通过统一应用入口调用同一证据、路径和活动系统。章节入口和路径入口共享知识点编号、活动、证据和长期画像，不能建设两份数据。

## 11. PathEngine 确定性路径算法

### 11.1 构建步骤

`PathEngine.buildPath()` 严格按以下顺序运行：

1. 校验会话、`profileRevision`、目标、知识点和活动引用；
2. 从目标取得全部核心知识点，并计算完整先修闭包；
3. 根据 `skipEligible` 标记可以压缩的先修，但不删除不可跳过门禁；
4. 为未满足先修或 `unverified` 节点插入验证/回补节点；
5. 对知识点图执行稳定拓扑排序，保证先修始终在后置节点前；
6. 在同一拓扑层内依据目标、重要度、错误证据和入口模式排序；
7. 从 active Profile 选择真实存在且匹配知识点和难度的活动；
8. 根据知识状态决定目标难度、脚手架和提示退坡；
9. 按时间预算压缩可选活动；
10. 校验核心知识覆盖、活动引用、无重复节点和最终任务；
11. 返回不可变路径草稿、原因码，或返回结构化不可行结果。

模型和 Agent 不参与以上任何排序、过滤和合法性判断。学习决策智能体只能把 `reasonCodes` 翻译成自然语言。

### 11.2 稳定排序

相同资料包修订、证据版本、目标、模式、预算和用户约束必须得到相同路径。多个节点同样合法时依次比较：

1. 核心目标节点优先；
2. 已暴露的先修缺口优先；
3. `importance` 高者优先；
4. 资料包声明顺序靠前者优先；
5. `knowledgePointId` 字典序靠前者优先。

不能使用随机数、当前时间或智能体措辞作为排序输入。

### 11.3 候选路径与用户选择

系统可以用同一引擎生成系统推荐路径；未来扩展可增加其他侧重点候选。候选路径只是不同约束下的合法结果，不是三套手写路径。Pandas六周用户不能选择未开放的快速实战或查漏补缺入口，也不能直接上传节点顺序或解锁后置知识点。

所有候选共享相同专业事实、核心目标、最终任务和通过标准。差异只能出现在顺序、可选活动、难度、脚手架和预计时间。

## 12. 路径数据结构、状态和原因码

### 12.1 公共类型

```ts
type PathMode = "chapter" | "recommended" | "goal" | "task_first" | "gap_review";
type PathStatus = "draft" | "confirmed" | "superseded";
type PathNodeStatus = "locked" | "available" | "in_progress" | "completed" | "skipped";
type ScaffoldLevel = "none" | "hint" | "worked_example";

type PathReasonCode =
  | "prerequisite_gap"      // 存在未满足先修
  | "low_mastery"           // 已有直接低分证据
  | "goal_required"         // 当前目标必须覆盖
  | "review_due"            // 已掌握但达到复习条件
  | "user_selected"         // 用户选择的合法侧重点
  | "error_remediation"     // 正式错误暴露了需要回补的知识
  | "time_compressed"       // 因时间预算压缩了可选活动
  | "evidence_insufficient";// 证据不足，先安排短验证

interface PathBuildRequest {
  requestId: string;             // 写请求幂等编号
  sessionId: string;             // 所属会话
  sessionVersion: number;        // 预期会话比较写版本
  profileRevision: number;       // 预期资料包修订号
  evidenceVersion: number;       // 本次构建所依据的证据版本
  goalId: string;                // 资料包内学习目标
  mode: PathMode;                // 统一入口模式
  availableMinutes: number;      // 可用分钟数；必须为正整数
  selectedKnowledgePointIds: string[]; // 用户合法选择；仅作为约束
  lockedNodeIds: string[];       // 重算时不得移动的节点
}

interface LearningPathNode {
  nodeId: string;                // 路径版本内稳定节点编号
  knowledgePointId: string;      // 对应知识点
  activityIds: string[];         // 真实存在的Profile活动编号
  status: PathNodeStatus;        // 当前节点运行状态
  positionLocked: boolean;       // 用户/运行规则是否禁止重算时移动位置
  required: boolean;             // 核心目标/门禁节点不能因预算删除
  difficulty: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 目标活动难度
  scaffold: ScaffoldLevel;       // none无脚手架、hint局部提示、worked_example完整范例
  estimatedMinutes: number;      // 预计分钟数；正整数
  reasonCodes: PathReasonCode[]; // 有限、可测试的安排原因
}

interface LearningPath {
  pathId: string;                // 路径身份编号
  sessionId: string;             // 所属学习会话
  profileRevision: number;       // 锁定资料包修订号
  evidenceVersion: number;       // 生成时使用的证据版本
  pathVersion: number;           // 同一会话单调递增的快照版本
  engineVersion: "path-engine-v1"; // 确定性算法版本
  status: PathStatus;            // 草稿、正式或已被替代
  mode: PathMode;                // 入口模式
  goalId: string;                // 统一目标编号
  availableMinutes: number;      // 用户确认时的时间预算
  estimatedMinutes: number;      // 路径活动预计总分钟数
  nodes: LearningPathNode[];     // 已通过先修和引用校验的节点
  positionLockedNodeIds: string[]; // 快照内禁止移动的节点；与先修locked状态不同
  changeReasons: PathReasonCode[]; // 相对旧版本的变化原因
  createdAt: string;             // 服务端生成时间
}

interface PathFailure {
  code: "path_infeasible";       // 统一不可行错误码
  missingPrerequisiteIds: string[]; // 未满足核心先修
  minimumRequiredMinutes: number;   // 合法最短路径预计分钟数
  suggestions: Array<"increase_time" | "change_goal" | "run_probe">; // 可行的安全补救动作
}

type PathBuildResult =
  | { status: "ok"; path: LearningPath }
  | { status: "infeasible"; failure: PathFailure };
```

### 12.2 不同层级的状态不能混用

- 会话状态回答“整个学习会话进行到哪一步”。
- 诊断状态回答“固定诊断是否可继续或已完成”。
- 知识状态回答“某知识点有多少直接证据”。
- 路径状态回答“某个路径版本是否正式生效”。
- 节点状态回答“某个路径节点现在能否进入”。

例如 `KnowledgeStatus="mastered"` 不等于 `PathNodeStatus="completed"`：前者来自跨活动证据，后者表示这条路径中的节点已经完成。

`PathNodeStatus="locked"` 表示先修尚未满足，`positionLocked=true` 表示重算时保持节点位置；二者完全不同。一个已完成节点的状态是 `completed`，但其 `positionLocked` 仍为 `true`。

原因码由服务端产生，前端只能按固定映射解释，不能根据自然语言猜测路径逻辑：

| 原因码 | 中文含义 | 建议展示文案 |
|---|---|---|
| `prerequisite_gap` | 存在未满足的先修知识 | 先补足这项基础，再进入后续任务 |
| `low_mastery` | 已有可靠证据显示掌握度较低 | 根据近期作答，先安排一次针对性练习 |
| `goal_required` | 当前目标必须覆盖该节点 | 这是完成当前目标所必需的内容 |
| `review_due` | 过去已掌握，但需要到期复习 | 你以前已有证据，现在用短任务复核 |
| `user_selected` | 来自用户选择的合法侧重点 | 根据你选择的学习侧重点加入此节点 |
| `error_remediation` | 正式错误暴露了先修缺口 | 刚才的错误与这项基础有关，先做短回补 |
| `time_compressed` | 时间预算压缩了可选活动 | 已按你的时间预算减少重复练习 |
| `evidence_insufficient` | 直接证据不足 | 先完成一个短验证，再决定是否压缩该节点 |

`PathFailure.suggestions` 中的 `increase_time`、`change_goal` 和 `run_probe` 分别表示增加时间、调整学习目标和先做短验证。前端只能提供这些安全操作，不能自行删除核心先修来“修复”不可行路径。

### 12.3 节点状态转换

```text
locked      # 先修未满足；不能直接进入
  → available   # 先修满足或验证通过；允许开始
  → in_progress # 已打开并产生草稿；提交期间禁止重算
  → completed   # 必需活动达到完成条件

available → skipped # 仅当规则允许压缩且节点非必需
```

前端不能自行改变状态。所有转换由服务端校验 `sessionVersion`、当前路径和先修条件后写入。

## 13. 时间预算与不可行路径

时间不足时按固定顺序压缩：

1. 删除可选的重复练习；
2. 把非核心讲解缩短为固定摘要或卡片；
3. 保留核心先修、核心目标、不可跳过门禁和最终任务；
4. 若仍超过预算，返回 `path_infeasible`，不伪造一条“不完整但已完成目标”的路径。

不可行结果必须向界面提供：缺失先修、合法最短路径预计时间、可增加的预算、可更换的目标，以及可先运行的5分钟验证活动。快速实战只承诺完成一次任务练习，不能把练习完成显示成整个学习目标达成。

## 14. 动态更新和路径重算

### 14.1 允许触发重算的事件

- 正式提交产生有效新证据，并使知识状态跨越表8.2阈值；
- 连续可靠失败或正式代码错误暴露新的先修缺口；
- 用户修改目标、入口模式或时间预算；
- 用户主动重新诊断或请求重新规划。

### 14.2 不触发重算的事件

- Pyodide 公开运行；
- 背景自评变化；
- 普通上下文提问；
- 单纯查看提示或范例；
- `evaluator_error`；
- 页面刷新、轮询或重复读取。

### 14.3 重算不变量

1. 当前活动提交期间不重算。
2. 已完成节点、当前 `in_progress` 节点和用户锁定节点不移动。
3. 只重新生成未完成后缀，旧路径保存为 `superseded`。
4. 每次正式提交最多生成一个新 `pathVersion`。
5. 新路径写入或校验失败时保留旧 confirmed 路径。
6. 变化必须保存 `changeReasons`，界面解释“哪条证据导致了什么改变”。
7. 使用状态阈值和连续证据条件，不能每次微小分数变化都重新排序。

默认防抖规则：只有知识状态变化、`skipEligible` 变化或出现 `error_remediation` 才自动重算。`mastery` 在同一状态区间内的小幅变化只更新画像，不改路径。

### 14.4 上下文提问

提问智能体只读取当前节点、学习者选中的安全文本、允许来源和裁剪后的知识状态。提问写入 `context_question` 软证据，只影响解释方式和后续验证建议，不降低掌握度、不满足先修、不直接插入或删除路径节点。

## 15. LearningRuntimeFacade 公共用例

`LearningRuntimeFacade`（统一学习运行入口）是[六周新增]应用层接口。网页和 Pi 终端必须调用相同用例，不能直接读写仓储。

09号文档中的 `LearningRuntimeFacade` 是统一规范名称；04号文档若出现 `StudyAppFacade`，视为同一拟新增接口的旧称，实施时不得创建第二个 facade。09号的聚合方法 `submitDiagnostic` 表示整套诊断用例能力，本文的 `saveDiagnosticDraft`、`submitDiagnosticAnswer` 和 `completeDiagnostic` 是其可恢复短用例拆分。

| 方法 | 中文职责 | 关键输入 | 成功输出 | 主要失败 |
|---|---|---|---|---|
| `startSession` | 创建会话并锁定资料包版本 | subjectId、mode、goalId、预算 | 安全会话视图 | Profile无效、写盘失败 |
| `saveDiagnosticDraft` | 保存问卷和诊断进度 | requestId、sessionVersion、profileRevision、草稿 | 新会话版本 | 版本冲突、输入无效 |
| `submitDiagnosticAnswer` | 判定并幂等保存一题 | requestId、sessionVersion、profileRevision、questionId、答案 | 安全判定状态 | 重复冲突、题目无效 |
| `completeDiagnostic` | 校验覆盖并生成诊断快照 | requestId、sessionVersion、profileRevision | 诊断安全视图 | 诊断不完整、修订冲突 |
| `buildPath` | 生成一条或多条合法候选路径 | 含requestId、sessionVersion、profileRevision的 `PathBuildRequest` | 路径或不可行结果 | 先修、预算、引用错误 |
| `confirmPath` | 原子确认路径快照 | requestId、sessionVersion、profileRevision、pathId、pathVersion | 正式路径视图 | 陈旧路径、会话冲突 |
| `getNextStep` | 读取当前可进入节点 | sessionId | 安全节点/活动视图 | 无正式路径、节点锁定 |
| `replanPath` | 重算未完成路径后缀 | requestId、sessionVersion、profileRevision、触发原因 | 新路径版本 | 当前提交中、计算失败 |
| `recoverSession` | 从检查点和证据恢复会话 | requestId、sessionVersion、profileRevision、sessionId | 最新安全会话视图 | 数据损坏、修订不可用 |

除 `getNextStep` 等纯读取方法外，所有会改变状态或恢复指针的方法都必须携带：

- `requestId`：请求幂等编号；网络重试时保持不变；
- `sessionVersion`：预期会话版本；比较失败不得覆盖；
- `profileRevision`：会话锁定的资料包修订号；不得由浏览器任意指定新值。

响应统一返回 `requestId`、最新 `sessionVersion`、业务状态、安全视图和可选 `errorCode`。04 号文档中的 `POST /api/sessions/:id/diagnostic` 和 `POST /api/sessions/:id/path` 只是 HTTP 传输层入口，内部仍分别调用上表的短用例。

## 16. 错误码与失败处理

| 错误码 | 触发条件 | 可否重试 | 是否已经写入 | 界面处理 |
|---|---|---|---|---|
| `profile_invalid` | active Profile结构、能力或引用无效 | 修复/换包后可重试 | 不创建会话 | 留在开始页并显示原因 |
| `profile_revision_mismatch` | 请求版本与会话锁定版本不同 | 刷新会话后处理 | 本次不写入 | 使用旧会话版本或新建会话 |
| `diagnostic_incomplete` | 未达到最低覆盖或答案尚未判定 | 完成剩余题后可重试 | 草稿保留 | 回到首个未完成题 |
| `diagnostic_answer_invalid` | 题号、答案结构或选项不合法 | 修改输入可重试 | 本题不写入 | 保留输入并标明问题 |
| `evidence_invalid` | 分数、来源、知识点或评测版本不合法 | 修复服务端问题后重试 | 不递增证据版本 | 不改变学情和路径 |
| `path_infeasible` | 核心先修或时间预算无法满足 | 改预算/目标后可重试 | 不替换当前路径 | 显示最低时间和验证建议 |
| `prerequisite_violation` | 用户尝试进入锁定后置节点 | 先完成验证/先修后重试 | 不改变节点 | 显示缺失先修 |
| `session_conflict` | `sessionVersion` 比较失败 | 获取最新视图后可重试 | 本次不写入 | 展示冲突页，不静默覆盖 |
| `storage_error` | 原子写、磁盘或目录操作失败 | 环境恢复后可重试 | 以最后完整检查点为准 | 不显示“已保存” |

诊断或学习决策智能体超时不使用以上业务错误替代确定性结果：程序保留已经计算的状态和路径，只把解释替换为固定模板。

## 17. 运行数据目录、读写者和恢复

以下目录全部是[六周新增]建议；真实代码实现时使用英文名称。每一项右侧注释说明内容、读写者、私有性和恢复用途。

```text
_user/                                      # 学习者私有数据根；仓储受控访问，不随Profile分发
└─ learning_sessions/                       # 新版会话集合；LearningSessionRepository读写，可逐会话恢复
   └─ <session-id>/                         # 单个会话目录；应用层独占写入，私有且支持断点恢复
      ├─ session.json                       # 会话阶段和四个版本；仓储原子写，是恢复入口
      ├─ diagnostic/                        # 当前会话诊断数据；诊断用例读写，不对其他学习者共享
      │  ├─ background.json                 # 背景问卷快照；软输入，不直接参与掌握度
      │  ├─ draft.json                      # 未完成题号和保存状态；刷新或退出后可恢复
      │  ├─ answers/                        # 逐题答案目录；服务端受控写入，前端不能直接读取文件
      │  │  └─ <question-id>.json           # 单题提交和判定引用；按requestId幂等保存
      │  └─ result.json                     # 完整诊断快照；完成后不可原地修改
      ├─ evidence/                          # 不可变Evidence与审计记录集合；mastery子集是状态重建事实源
      │  └─ <evidence-id>.json              # 单条裁剪记录；正式证据含evidenceVersion，禁止隐藏答案
      ├─ knowledge_state.json               # 当前派生知识状态；聚合器写，可由evidence/重算
      ├─ paths/                             # 全部不可变路径快照；路径用例写，用于恢复和审计
      │  └─ path-<version>.json             # 单个pathVersion；旧版本保留为superseded
      ├─ active_path.json                   # 当前confirmed路径指针；不复制路径正文，可原子替换
      └─ checkpoints/                       # 跨页面检查点；应用层写，恢复时验证后使用
         └─ latest.json                     # 最后完整业务阶段和版本；不能领先于已落盘事实
```

现有 `_user/summaries/` 和 `PrivateMemoryRepository` 保持不变；它们继续服务旧版会话、作答、总结和长期画像。新增 `LearningSessionRepository` 管理上述目录，不修改旧批次格式来冒充迁移完成。

### 17.1 写入顺序

```text
作答判定完成                         # 尚未改变正式学情
  → 预分配下一个evidenceVersion      # 只用于本次有效mastery证据，不先公开
  → 写入不可变Evidence候选           # Evidence自身保存预分配版本，尚未提交
  → 重算knowledge_state.json        # 派生数据，可由事实源恢复
  → 必要时写path-<version>.json     # 新路径先完整校验再落盘
  → 写入active_path.json候选        # 指向新路径，但尚不对读取者公开
  → 原子替换session.json候选        # 保存即将公开的会话状态和版本
  → 最后原子替换latest.json         # 唯一提交标记；一次公开所有新版本
```

`latest.json` 是唯一提交标记，保存最后已提交的 `evidenceVersion/sessionVersion/pathVersion`。所有读取者只接受不超过该标记的 Evidence、会话和路径；即使候选指针已经写入，也不能提前公开。进程在提交标记更新前中断时，恢复器扫描候选文件：内容完整且 `requestId` 匹配的候选可以安全补提交，否则隔离候选并重新使用该未公开版本；不得同时存在两个同版本正式 Evidence。提交标记更新后，证据才对聚合查询可见。派生知识状态或路径指针不一致时根据已提交证据重算，不重复创建 Evidence，也不能因界面已经显示结果就假定磁盘成功。

### 17.2 并发和恢复

- 同一会话只允许一个正式写入事务；网页轮询和终端读取不取得写锁。
- 比较写失败返回 `session_conflict`，调用者重新读取最新安全视图。
- `requestId` 已处理时返回原响应，避免网络重试重复计分。
- Profile active 更新不改变进行中会话的 `profileRevision`；新会话使用新版。
- 旧修订被归档后，进行中会话仍需能够读取其只读快照；如果快照损坏则安全中断，不用新版规则重判旧作答。

## 18. 安全、隐私和智能体权限

1. 浏览器不接收诊断正确答案、来源权重、状态阈值、隐藏测试或完整 Rubric。
2. Agent 不接收完整私人答案，只接收必要的裁剪证据和允许来源。
3. Agent 输出永远不能写 `Evidence`、`KnowledgeState`、先修图或 `LearningPath.nodes`。
4. 资料包外知识点、活动或来源不能进入路径。
5. 用户提问不能被标记成能力弱；背景问卷不收集姓名、学号等无关身份信息。
6. 运行轨迹可以记录版本、原因码、耗时和错误码，不能记录密钥、隐藏资产或完整答案。
7. 所有目录通过安全相对路径解析，拒绝绝对路径、`..` 和符号链接穿越。

## 19. 测试设计

### 19.1 单元测试

- 每种来源权重、独立性权重和时间权重；
- 无证据、跳过、部分得分、提示折扣和答案暴露；
- `attemptId` 幂等、无效分数和修订不匹配；
- 一致性奖励、冲突惩罚、置信度上下界；
- 五种知识状态、三类界面文案和 `skipEligible`；
- 五个手工示例的计算结果与文档一致；
- 时间预算压缩顺序和不可行结果。

### 19.2 属性与表格测试

- 随机生成合法有向无环先修图，所有输出路径无环、无重复且先修在前；
- 相同输入重复运行得到完全相同的节点、活动、难度和原因码；
- 核心目标、不可跳过门禁和最终任务始终存在；
- 无可行路径返回原因，不抛出无法解释的空结果；
- 60个人工标准案例逐项比较知识状态、允许难度、脚手架和路径约束。

### 19.3 集成与恢复测试

- 创建会话→保存问卷→逐题提交→退出→恢复→完成诊断；
- 诊断→知识状态→候选路径→用户确认→取得下一节点；
- 正式提交→Evidence→KnowledgeState→未完成后缀重算；
- 重复请求不重复计分，陈旧 `sessionVersion` 不覆盖新数据；
- 在写证据、写状态、写路径和换指针后分别模拟中断并恢复；
- 新路径失败继续使用旧 confirmed 路径；
- Web/TUI 对同一会话看到相同下一节点和版本。

### 19.4 安全和权限测试

- 前端 DTO、日志和 Agent 输入中不存在答案、权重、阈值、隐藏测试和密钥；
- 用户不能提交任意先修图、路径节点、难度或 `profileRevision`；
- 目录穿越、符号链接和非法编号被拒绝；
- `public_run`、自评、提问、提示和 `evaluator_error` 均不改变 `mastery`；
- Agent 伪造分数、路径或来源时被结构/业务校验拒绝。

### 19.5 兼容和效果验收

- Profile 第1版和现有三种 `ReviewMode` 继续通过原测试；
- 旧 `LearningProfile` 不会被错误转换为知识点状态；
- 60个正式案例路径合法率为 `100%`；
- 赛题难度适配准确率不低于 `85%`；
- 核心知识点覆盖率不低于 `90%`；
- 三类学习者在相同目标、修订、事实和最终标准下得到不同但合法的路径；
- 三个完整展示案例保存诊断输入、Evidence、KnowledgeState、候选路径、最终选择、路径变化和相同通过标准。

05号文档中的 `insufficient` 是正式案例的“证据不足判定标签”，用于检查系统是否正确识别低样本，不是第六种 `KnowledgeStatus`。三类正式画像各20例，必须分别报告结果；三个完整展示案例保持同一目标、核心事实、最终测试、Rubric和通过阈值，并至少在路径顺序、难度、示例、脚手架、提示和反馈六项中的三项形成有证据的差异。

## 20. 六周实施顺序、降级和完成定义

### 20.1 实施顺序

| 周次 | 学情与路径工作 | 完成门禁 |
|---|---|---|
| 第1周 | 冻结类型、字段、错误码、Profile诊断/目标引用 | 文档、资料包和接口名称一致 |
| 第2周 | 完成Pandas诊断蓝图、证据聚合器、知识状态和60案例人工标准 | 五个计算示例通过；随机路径合法率100% |
| 第3周 | 实现 `LearningSessionRepository`、`LearningRuntimeFacade`、`PathEngine` 和路径确认 | 诊断到确认路径端到端可恢复 |
| 第4周 | 接入诊断解释与学习决策智能体、固定模板和运行轨迹 | Agent失败不阻断确定性结果 |
| 第5周 | 接入正式活动证据、Web/TUI共享会话和动态重算 | Run不改分、Submit幂等、冲突可恢复 |
| 第6周 | 冻结参数，运行60案例、三类展示和全量回归 | 指标、限制和原始证据可复核 |

### 20.2 降级顺序

进度不足时按以下顺序削减：

1. 取消五维高级图形、复杂图动画和路径变化动画，但保留五维画像数据及“分数+置信度+证据卡”；
2. 关闭运行时CIDPP和复杂能力维度，保留构建期内容质量摘要；
3. 候选路径保持系统推荐单一路径；
4. 动态重算改为节点完成后显式触发；
5. 智能体解释全部降级为固定中文模板。

不得削减：固定诊断、证据不可变和幂等、低样本“未验证”、确定性路径合法性、关键先修、路径版本、错误隔离、恢复和60案例验证。

### 20.3 完成定义

本模块只有同时满足以下条件才算完成：

1. AI 或新开发者仅阅读本文即可解释每个类型、字段、状态、公式和写入顺序；
2. 相同输入得到相同知识状态和路径；
3. 无证据不会显示成低分，提问和评测器异常不会降低掌握度；
4. 章节和路径入口共用 Profile、Evidence、KnowledgeState、活动和仓储；
5. 所有路径满足先修、核心覆盖、引用和时间预算规则；
6. 刷新、重复提交、版本冲突和进程中断不会重复计分或丢失最后完整事实；
7. Profile 第1版和当前学习流程没有被破坏；
8. 自动化测试和05号文档规定的正式评测均能生成可复核证据。
