## 1. 本方案要证明什么

评测不是只跑几个单元测试，也不是只展示三张不同页面。本方案需要依次证明：

1. **工程上能运行**：从诊断到路径、资源、活动、反馈、更新和恢复没有断链。
2. **知识上可信**：生成内容有来源，事实和代码错误受到控制，幻觉率 `<5%`。
3. **个性化真实**：不同画像得到合理难度、顺序和脚手架，难度适配准确率 `≥85%`。
4. **领域内容完整**：核心知识点同时有讲解、练习和应用，覆盖率 `≥90%`。
5. **多智能体有作用**：串行审核相对同一原稿能消除问题，且额外延迟和成本可接受。
6. **学习过程会动态变化**：正式活动证据能更新学情和未完成路径，并给出理由。
7. **代码任务可判定**：Pandas 和通用 Python 题能运行，错误分类正确，隐藏测试不泄漏。
8. **人能正常使用**：界面、状态、恢复和智能体协作可视化不会阻塞学习者。
9. **材料可复核**：公司接入自己的 OpenAI 兼容应用程序接口后，能够按同一协议重跑并得到完整报告。

## 3. 指标分级和统一字段

### 3.0 学习者评价方案A/B的专项比较

学习者评价采用已冻结的双层方案，不把内容质量 CIDPP 分数当作学习效果：

| 方案 | 测量对象 | 计算方式 | 启用条件与失败回退 |
|---|---|---|---|
| 知识点底座 | 知识点掌握度与置信度 | 由固定诊断、客观题、代码测试和实操证据按11号规则确定性聚合 | 路径和先修唯一权威 |
| Pandas五维画像 | 底座之上的五维能力 | AI对可观察维度直接给出0—100分、0—1置信度、理由和证据引用；无证据为 `unverified` | 只用于展示和建议，不能改写底座、路径或代码分数 |

专项测试至少覆盖：同一证据重复评价的一致性、缺少维度时是否保持 `unverified`、AI是否试图修改客观对错或路径、五维建议是否被确定性规则拒绝非法跳过、方案B失败后是否完整回退方案A。报告同时给出两方案的延迟、人工复核量、路径合法率和错误案例，不宣称小样本原型证明因果效果。

### 3.1 指标等级

- **硬性门禁**：赛题硬指标或安全、合法性门禁；未达标不得宣称完成。
- **必须完成**：六周核心功能验收；未通过必须修复或执行既定降级。
- **证据性指标**：用于证明增益、体验或趋势；必须如实报告，未必单独阻塞发布。
- **诊断性指标**：用于定位成本、延迟和失败原因，不单独作为产品效果结论。

### 3.2 每条评测记录的公共字段

下列英文名称是程序实际使用的字段标识。文档、界面和报告使用字段右侧中文含义，不要求项目成员用英文交流。

```ts
interface EvaluationRecord {
  evaluationId: string;       // 评测运行编号
  metricId: string;           // 指标编号
  caseId: string;             // 案例编号
  profileId: string;          // 资料包编号
  profileRevision: number;    // 资料包修订版本
  modelProvider: string;      // 模型服务提供方
  modelId: string;            // 模型编号
  promptVersion: string;      // 提示模板版本
  evaluatorVersion: string;   // 评测器版本
  inputHash: string;          // 输入内容摘要
  outputArtifactIds: string[];// 输出产物编号
  startedAt: string;          // 开始时间
  endedAt: string;            // 结束时间
  status: "included" | "excluded" | "failed"; // 纳入、排除或失败
  exclusionReason?: string;   // 排除原因

```

缺少版本、输入哈希或排除原因的记录不能进入正式报告。

正文、界面和最终报告统一使用左侧中文名称；右侧标识只用于程序字段、接口返回值和自动测试，不能直接当作面向评委的解释。

| 中文业务名称 | 程序中的固定标识 | 保留原因 |

### 4.1 各目录之间的数据流

```text
personas/（学习者画像案例） + golden/（人工标准）
        → 运行诊断和路径
        → 产生难度适配与路径合法性结果

showcases/（完整差异化案例）
        → 导出学习资源和智能体审核记录
        → claims/（原子事实评测） + paired-review/（单/多智能体对照）

资料包核心知识点
        → knowledge-coverage/（知识覆盖评测）

学习活动和代码提交
        → runtime/（代码运行与判分） + failure-injection/（故障注入记录）

以上原始记录 + human-study/（真实学习者试测）
        → reports/（自动生成评测报告和证据索引）
```

### 4.2 敏感性和访问权限

- **普通可读**：`README.md`、`runtime/public/`、`runtime/environment-lock.json`、`reports/`，以及 `showcases/` 中已脱敏的输入输出。
- **项目组受限**：`personas/final-60.jsonl`、`golden/`、`claims/`、`paired-review/`、`human-study/anonymous-records.jsonl` 和故障原始记录。
- **仅评测器可读**：`runtime/hidden/`、参考实现和完整评分阻断规则。
- **禁止写入前端或智能体上下文**：隐藏测试、参考实现、应用程序接口密钥、真实身份、绝对文件路径。

提交给发榜公司的测试包应保留相同权限边界，并在说明文档中写清如何替换模型接口和隐藏测试，而不是将全部文件无差别公开给浏览器。

## 5. 指标一：专业知识幻觉率（硬性门禁，目标 `<5%`）

### 5.1 测什么

测最终交付给学习者的事实型内容是否错误、杜撰或与锁定版本不一致。评测对象包括：

- 学习卡片中的概念、程序接口行为和注意事项；
- 智能体回答中的事实结论；
- 题目解析与错误反馈；
- Python/Pandas 示例代码声称的运行结果；
- 路径解释中关于先修、活动和资料的陈述。

不把纯鼓励语、界面提示、明确标注为类比的非事实句放入分母。

### 5.2 原子事实怎样产生

1. 从 3 个完整差异化案例和分层抽取的 20 个生成运行中获得最终产物。
2. 由内容人员把每个可独立验证的事实拆为一条“原子事实”；一句有两个事实必须拆两条。
3. 保存原子事实编号、产物编号、知识点编号、事实正文、来源锚点编号和资料包版本。代码字段分别为 `claimId/artifactId/knowledgePointId/text/sourceAnchorIds/profileRevision`。
4. 原子事实拆分人员不能看到该内容是审核前初稿还是审核后终稿，降低主观偏差。

### 5.3 怎样判定

两名内容评审独立对照资料包允许来源、锁定的 Pandas 文档版本和实际代码运行结果，使用：

| 标签 | 含义 | 是否计入幻觉分子 |
|---|---|---:|
| 来源支持（代码值 `supported`） | 允许来源直接支持，范围和版本正确 | 否 |
| 外部正确但包内无依据（`correct_external`） | 事实正确，但允许来源不支持 | 否；计入无包内依据率 |
| 幻觉或事实错误（`hallucinated`） | 错误、杜撰、版本不符或与运行结果冲突 | 是 |
| 暂时无法核验（`unverifiable`） | 当前证据不足，无法判断 | 暂不进入已裁决分母；必须复核 |

两人分歧交给第三名裁决。正式报告前所有“暂时无法核验”项必须复核；仍无法判断时单独报告数量，不能偷偷排除。

### 5.4 公式和样例

```text
专业知识幻觉率 = 幻觉或事实错误数 / 已完成裁决的事实总数
无包内依据率   = 外部正确但包内无依据数 / 已完成裁决的事实总数
有效溯源率     = 有有效来源锚点的事实数 / 全部事实数
```

例：抽取 240 条原子事实，裁决后 230 条可判定，其中 8 条错误，则幻觉率 `8/230=3.48%`，达标；若 18 条虽正确但无允许来源，无包内依据率为 `18/230=7.83%`，仍需单独整改。

### 5.5 如何完成

- 第 2 周：内容负责人冻结来源锚点和原子事实标注说明。
- 第 4 周：自动导出审核前初稿和审核后终稿，试标 30 条并校准评审口径。
- 第 5 周：完成正式抽样和代码示例预运行。
- 第 6 周：双人盲标、第三人裁决、脚本汇总。

### 5.6 失败处理

任何核心原子事实错误、来源锚点摘要失效或代码示例不能运行，必须阻断相应产物；不能靠减少原子事实数量降低幻觉率。若整体 `≥5%`，回到来源、提示模板、猎手和裁判门禁修复后使用新的评测运行编号全量重跑，旧结果保留。

## 6. 指标二：学习者画像—资源难度适配准确率（硬性门禁，目标 `≥85%`）

### 6.1 测什么

测系统依据背景、固定诊断、目标和时间预算给出的资源难度与脚手架是否处于专家预先认可范围。不能只看一个全局“难/中/易”标签，最小判定单元是：

```text
学习者案例 × 路径节点 × 推荐难度 × 脚手架等级
```

同时记录路径先修是否合法；非法路径即使难度数值命中，也不能算成功。

### 6.2 60 个最终离线案例

| 画像组 | 数量 | 主要特征 | 应观察到的差异 |
|---|---:|---|---|
| 计算机/Python 基础较好 | 20 | 语法和抽象能力较强，Pandas程度不同 | 可压缩已证明先修，更多边界/综合题 |
| 非计算机/初学者 | 20 | 编程基础不稳、认知负担敏感 | 更细讲解、较低起始难度、较强脚手架 |
| 实践目标导向 | 20 | 时间有限、偏好先看任务 | 允许情境预览，失败后按错误回补 |

每组覆盖高/中/低诊断表现、矛盾证据、跳过题、不同时间预算、不同错误标签。60 条是团队正式目标，也满足赛题“测试用例不足 50 组会被扣分”所要求的规模余量。

### 6.3 人工标准如何标注

每个案例先由两名熟悉 Pandas 内容和教学目标的评审，在看系统输出之前独立标注：

- 每个必需节点允许的难度集合；
- 允许的脚手架范围；
- 可以跳过的节点；
- 必须出现的回补节点；
- 禁止行为，如跳过未证明的核心先修；
- 允许的多条路径集合或约束，而不是强制唯一顺序。

分歧由第三名裁决并写入“分歧裁决记录”。人工标准冻结后生成内容摘要；看过系统结果后不得修改人工标准来提高分数。

### 6.4 执行步骤

1. 使用全新私有学习记录目录载入固定资料包修订版本。
2. 对每个案例输入问卷、诊断答案、目标和预算。
3. 运行确定性的证据聚合器和路径引擎；智能体生成的路径解释不参与正确性判定。
4. 对每个可判定节点比较目标难度字段 `targetDifficulty` 和脚手架字段 `scaffoldLevel` 是否命中人工标准。
5. 先运行路径合法性检查；非法案例记失败并报告原因。
6. 输出逐案例结果和三类画像分组结果。

### 6.5 公式

```text
难度适配准确率
  = 同时命中允许难度和脚手架范围的节点数 / 全部可判定案例节点数

案例完整成功率
  = 路径合法且满足全部必需约束的案例数 / 全部可判定案例数
```

主赛题指标采用节点级准确率，并同时报告案例级成功率、三组各自准确率和 95% 威尔逊置信区间。不可判定项仅能因人工标准或评测资产缺陷排除，必须列原因；系统报错不能排除，按失败计。

### 6.6 如何完成和门禁

- 第 1 周：冻结字段和人工标准模板。
- 第 2 周：内容和学情负责人完成 60 个案例双人标注；自动脚本能输出分子分母。
- 第 3–5 周：只用调试集调整规则，不查看正式集结果反复调参。
- 第 6 周：在冻结的发布候选版和全新目录运行正式 60 案例。

门禁：总体 `≥85%`，三组均需报告；路径合法率必须 `100%`。总体达标但某组明显偏低时不能只报告总体值，必须在限制中披露并优先修复该组。

## 7. 指标三：核心知识点覆盖率（硬性门禁，目标 `≥90%`）

### 7.1 测什么

测 Pandas 资料包的核心知识点是否真正形成“教—练—用—验—溯源”闭环，而不是只在文档中出现名字。

资料包为每个“核心知识点”（代码字段 `core=true`）定义“必需证据”（字段 `requiredEvidence`）：

1. 至少一个有效来源锚点；
2. 至少一个讲解或卡片产物；
3. 至少一个可确定判分的活动；
4. 对代码型核心点至少一个代码补全或实操应用；
5. 至少一个通过验证的降级资源；
6. 所有引用编号、测试和评分规则有效。

### 7.2 怎样测

使用静态覆盖脚本扫描冻结的资料包：

```text
知识点编号
 → 来源锚点存在且内容摘要有效
 → 学习活动编号存在
 → 讲解、评测、代码等必需证据满足
 → 评分规则和测试引用有效并可运行
 → 降级资源存在
```

随后对所有代码/客观题资产进行可执行性检查：参考实现通过公开/隐藏测试；错误样例不会被误判通过；固定诊断题、动态题和学习兜底题的答案均可由代码比较或规定的评测器确定判分；动态题与正式效果评测保留题严格隔离。

### 7.3 公式

```text
核心知识点覆盖率
  = 满足全部必需证据的核心知识点数 / 核心知识点总数
```

只满足部分证据的知识点按未覆盖计，不能按字段数量给部分分。例：10 个核心点中 9 个全部满足，则为 90%；第 10 个只有卡片没有可判定活动，仍算未覆盖。

### 7.4 如何完成

- 第 1 周：冻结“必需证据”字段和结构规范。
- 第 2 周：Pandas 包内容人员给每个核心点登记资产。
- 第 3 周：测试人员实现覆盖脚本和损坏引用测试样例。
- 第 5 周：在锁定执行环境预运行所有活动和降级资源。
- 第 6 周：随资料包发布候选版冻结重跑。

任何来源锚点内容摘要失效、测试不能运行或降级资源缺失，会使对应知识点未覆盖。核心覆盖 `<90%` 时应补充资产或勘正未经确认的“核心”范围；不能在看结果后随意把重要知识点改成非核心。

## 8. 指标四：路径合法性与动态决策正确性（必须完成）

### 8.1 测什么

路径必须满足：节点存在、无环、先修已完成或有足够直接证据、目标核心点被覆盖、活动引用有效、时间预算只压缩可选活动、已完成节点重算后不移动。

动态决策测试覆盖：

- 连续可靠正确使难度/脚手架按规则变化；
- 连续失败或新的先修错误插入短回补；
- 评测器异常（代码值 `evaluator_error`）不产生负证据；
- 仅提问、查看提示或自评不直接降低掌握度；
- 矛盾证据降低置信度并安排验证；
- 用户改变目标或预算产生新的路径版本和变更原因。

### 8.2 怎样测

领域层属性测试生成随机合法的有向无环先修图和学习证据序列；集成测试运行“诊断→路径→活动→学习证据→重新规划”。每次比较路径快照、原因代码、证据版本和完成节点集合。

通过标准：60 个正式案例路径合法率 `100%`；全部状态转移符合冻结规则；相同输入得到相同路径；智能体不能构造或写入非法路径。

## 9. 指标五：三组完整差异化学习者案例（硬性证据）

### 9.1 测什么

证明“不同学习者输入→不同中间决策→不同资源/路径→同一专业标准”的完整链，而不是三张只换名字的页面。

### 9.2 三个案例必须保存什么

从三类画像各选 1 个，固定同一目标、资料包修订版本、核心事实、最终评分规则和测试：

1. 背景问卷和诊断答案；
2. 原始学习证据和学习者诊断结果；
3. 知识状态、置信度和证据不足项；
4. 合法候选路径、最终路径和原因代码；
5. 同一核心点的审核前初稿、猎手意见、条件辩护、裁判结论和审核后终稿；
6. 卡片、客观题、代码补全和实操；
7. 运行、正式提交、错误标签、评分规则和提示使用；
8. 新学习证据、路径变化、总结和下一步。

### 9.3 差异判定

三案在以下六项中至少三项产生可解释差异：顺序、难度、例子、脚手架、提示、反馈。差异必须能追溯到输入或学习证据；核心事实、先修、测试、评分规则和通过阈值不得因画像而变化。

验收由产品、内容、测试三方逐项签字，生成差异对照表、全链路结构化记录、截图或视频和产物索引。

## 10. 指标六：多智能体审核增益（证据性指标，20 组配对盲评）

### 10.1 测什么

比较同一生成者原稿经过串行“猎手核查→条件辩护→裁判定稿”后是否更正确、可溯源、适合学习和安全；同时测增加的延迟、令牌消耗和降级比例。

### 10.2 公平配对

每个生成运行编号固定资料包修订版本、画像、节点、提示版本、模型和随机配置：

- 甲、乙两侧中的一侧是第一次通过结构校验的审核前初稿；
- 另一侧是对该同一初稿完成审核后的终稿；
- 不能再次调用生成者来生成所谓“多智能体版原稿”；
- A/B 随机左右、隐藏身份，评审不知道哪份经审核。

20 组按三类画像、Pandas 六个模块、不同难度/脚手架分层抽取。

### 10.3 评分表

两名评审独立给 1–5 分：事实/代码正确性、来源一致性、个性化适配、教学清晰度、可执行性、答案泄漏安全。另标严重错误、偏好 A/B/相同和理由；第三人只处理重大分歧。

### 10.4 报告指标

```text
配对胜率 = 审核后终稿胜出数 / 排除平局后的有效配对数
严重错误消除率 = 被终稿消除的严重错误数 / 初稿严重错误总数
成本倍数 = 多智能体总令牌消耗 / 单智能体初稿令牌消耗
增加延迟 = 终稿可用时间 - 初稿可用时间
降级率 = 使用固定降级资源的审核运行数 / 全部审核运行数
```

同时报告胜、平、负，六个维度的平均差、评审一致率和条件辩护触发率。

最低验收：审核后终稿的严重错误不得多于审核前初稿；胜数大于负数；所有 20 对原始记录完整。若未体现增益，必须诚实报告并分析角色或门禁问题，不替换不利样本。

## 11. 指标七：串行智能体协作真实性与恢复（必须完成）

### 11.1 测什么

证明多智能体确实按顺序、隔离、有条件触发，而不是一个智能体改名或多个角色并行表演。

### 11.2 怎样测

- 运行轨迹检查“生成结束时间 ≤ 猎手开始时间 ≤ 条件辩护开始时间 ≤ 裁判开始时间”；
- 同一生成运行同时最多一个角色处于运行中；
- 每个角色获得新的隔离智能体会话，只收到显式结构化输入；
- 低风险运行不触发辩护角色，高风险运行必须触发；
- 在各阶段后注入故障，恢复只重跑未完成阶段，初稿内容摘要不变；
- 裁判和其他智能体不能修改资料包、评分规则、隐藏测试和路径引擎输出。

通过标准：全部合同、顺序和隔离测试通过，演示时间线与运行轨迹一致；不能用同时闪烁的界面状态暗示并行。

## 12. 指标八：代码任务与评测器正确性（必须完成）

### 12.1 测什么

测浏览器 Python 运行环境的公开“运行”和本地 Node/外部评测器的正式“提交”，是否能对 Pandas 和通用 Python 函数题稳定执行、正确分类并保护隐藏资产。实现名 Pyodide 和 Node 在首次出现后保留，避免与代码实现脱节。

### 12.2 测试矩阵

| 输入 | 预期分类 | 是否写正式学习证据 |
|---|---|---:|
| 正确代码 | 通过（`passed`） | 是，正证据 |
| 语法错误 | 学习者语法错误（`syntax_error`） | 正式提交时是 |
| 运行异常 | 学习者运行错误（`runtime_error`） | 正式提交时是 |
| 公开或隐藏断言失败 | 学习者测试失败（`test_failed`） | 正式提交时是 |
| 无限循环 | 学习者超时（`timeout`） | 正式提交时是 |
| 超大输出 | 学习者输出超限（`output_limit`） | 正式提交时是 |
| 评测进程崩溃 | 评测器异常（`evaluator_error`） | 否 |
| 测试资产损坏或缺库 | 评测器异常（`evaluator_error`） | 否 |
| 重复提交编号 | 返回原结果 | 不重复写 |

每个固定任务至少准备正确、部分正确、典型错误和恶意资源占用四类代码。锁定任务重复运行 3 次，结果和评分规则必须一致；浏览器运行环境与 Node 正式评测的差异要记录，正式提交结果为权威。

### 12.3 隐藏测试泄漏

自动扫描 React 构建产物、页面文档对象、浏览器响应、本地存储、智能体提示、运行轨迹、普通日志和错误信息。出现隐藏测试源码、参考实现、绝对路径或应用程序接口密钥即安全门禁失败。

## 13. 指标九：工程质量、兼容和失败恢复（必须完成）

| 层级 | 测试内容 | 完成标准 |
|---|---|---|
| 领域层 | 第1/2版结构规范、有向无环先修图、学习证据、路径、评分规则、泄漏状态 | 纯函数和边界值测试通过 |
| 仓储层 | 原子写、修订版本、待处理批次、检查点、归档、版本比较写入 | 故障后旧启用包和已保存提交不损坏 |
| 工作流图层 | 结构规范、重试、超时、隔离、降级 | 重试有上限，错误可分类 |
| 应用层 | 统一用例入口、幂等、终端和网页共用同一会话 | 不复制业务规则，无双写 |
| 界面层 | 空、加载、成功、错误、恢复、冲突 | 用户总有明确下一步 |
| 回归 | 现有五命令、三学习模式、终端材料、题目和恢复 | 既有测试和手工冒烟检查通过 |
| 安全 | 路径穿越、跨站脚本、提示注入、密钥和答案泄漏 | 敏感资产不可达 |

失败注入包括模型超时、无效结构化输出、错误来源锚点、裁判失败、资料包损坏、Pyodide 初始化失败、Node 子进程退出、浏览器刷新、双窗口版本冲突、终端和网页同时写入及磁盘失败。

## 14. 指标十：用户体验和任务完成（证据性指标/必须完成）

### 14.1 自动和人工走查

对 04 号文档所有页面执行组件和端到端测试：路由守卫、键盘主流程、状态文案、缩放、刷新、运行和正式提交、恢复及智能体时间线。核心页面不能黑屏或只显示内部错误。

### 14.2 任务完成指标

6–10 名真实试测者执行：开始学习、完成诊断、确认路径、完成卡片/客观题/代码补全、查看反馈和恢复一次会话。

```text
任务完成率 = 无协助完成全部指定任务人数 / 参与人数
阻塞错误率 = 因系统问题无法继续的任务数 / 全部任务数
```

报告完成时间、求助次数、误操作、主要阻塞点和简化系统可用性量表结果。样本小，不设虚假的统计显著性硬门槛；必须完成的门禁是不存在普遍性阻塞、评测器异常不被误导为用户错误、核心流程可完成。

## 15. 指标十一：探索性学习效果与迁移（证据性指标）

### 15.1 目的

观察短时使用后知识和迁移是否有正向趋势，不声称证明长期因果效果。

### 15.2 流程

1. 匿名编号和知情说明；
2. 15–20 分钟固定前测；
3. 45–60 分钟完成规定路径片段；
4. 同构但不重复题面的后测；
5. 24–72 小时后用新的合成表格数据完成迁移题；
6. 访谈学习负担、提示帮助和错误反馈。

前/后测使用同一知识蓝图、难度配平和确定性评分；A/B 卷随机顺序。迁移题不能是训练中原题或同一数据值。

```text
标准化学习增益 =（后测分数 - 前测分数）/（满分 - 前测分数）
迁移正确率 = 迁移题得分 / 迁移题总分
```

报告每人和总体中位数、范围，不只报告平均值；退出或缺少延迟迁移的样本单列。结果不理想时仍提交原始数据和限制。

## 16. 指标十二：性能、成本、缓存和降级（诊断性指标）

每个智能体或评测运行记录耗时、令牌数量、是否命中缓存、重试次数、是否使用降级资源和错误代码。对应程序字段为 `durationMs/token/cacheHit/retryCount/fallback/errorCode`。分场景报告：缓存卡片、普通问答、标准审核、高风险审核、浏览器运行、Node 正式提交。

关键指标：中位延迟、95百分位延迟、平均令牌数量、每完成节点调用次数、缓存命中率、降级率、智能体和评测器错误率。六周不承诺生产级服务指标，但普通低风险问答不得无条件走完整审核链；降级率过高说明动态能力不稳定，必须在演示中说明。

## 17. 真实学习者数据合规

- 招募前提供用途、时长、采集字段、退出和删除方式。
- 只保存匿名编号、作答、代码、任务时长、提示/点击事件和问卷；不采集姓名、学校、联系方式或屏幕外活动。
- 同意记录与学习记录分开保存；公开报告只给汇总/脱敏片段。
- 真实数据不进入模型提示上下文，除非知情说明明确覆盖且先脱敏；最小可行产品优先使用合成画像。
- 用户要求退出时删除未汇总的个人记录，并在报告中调整分母和说明。

## 18. 评测防污染和可复现

1. 调试集用于开发阶段调整，正式60案例、正式原子事实和迁移题在发布候选版冻结后才运行。
2. 冻结资料包修订版本、知识来源、模型和提示版本、执行器与库版本、随机种子、人工标准内容摘要。
3. 测试人员维护人工标准；开发人员不得看正式集输出后修改人工标准。
4. 隐藏测试不进入生成上下文，也不用于优化题面提示。
5. 每次运行从全新数据根目录开始，保存环境信息和输入/输出哈希。
6. 排除样本必须使用预定义原因：资产损坏、评审缺失、参与者退出；系统失败不能排除。
7. 指标脚本同时输出分子、分母、排除数、失败数和分组结果。
8. 修复后重跑使用新的运行编号，旧报告只读保留，禁止覆盖不利结果。

## 19. 一次正式评测怎样执行

### 19.1 冻结前检查

- 第1版和第2版资料包、Pandas 数据集、来源、任务、评分规则、降级资源校验通过；
- 模型和执行器配置可运行；
- 人工标准、原子事实和盲评表完成双人预标；
- 隐藏资产权限检查通过；
- 当前代码提交编号或压缩包内容摘要记录完成。

### 19.2 自动阶段

```text
新建评测数据目录
→ 运行现有回归和类型检查
→ 扫描资料包和知识覆盖
→ 运行正式60案例的路径与难度评测
→ 运行代码任务与失败注入
→ 运行三个完整差异化案例端到端测试
→ 导出单/多智能体配对和原子事实
→ 生成待人工标注表
```

### 19.3 人工阶段

内容评审标注原子事实；盲评员标注甲、乙两版；测试人员完成三个完整案例走查；6–10 人完成探索性试测。角色冲突要避免：生成内容的开发者不能独自判定该内容正确。

### 19.4 汇总阶段

脚本读取不可变的逐行结构化记录，生成“指标结果.json”和“指标报告.md”；负责人核对分子分母、排除原因、三类画像分组和失败案例；最终形成“证据索引.md”，将每条赛题要求链接到原始证据。

## 20. 责任分工

| 角色 | 主要责任 | 不得单独完成的判定 |
|---|---|---|
| 测试负责人 | 评测清单、运行脚本、报告、发布门禁 | 内容事实正确性 |
| 内容负责人 | 来源锚点、原子事实、核心覆盖、人工标准 | 自己生成内容的唯一裁决 |

| 智能体负责人 | 配对导出、运行轨迹、成本、降级 | 选择性删除负面配对 |
| 执行器负责人 | 代码任务、隐藏测试、错误分类、环境锁定 | 把评测器异常记成用户错误 |
| 产品和网页负责人 | 端到端测试、任务完成、可访问性、恢复 | 隐藏阻塞失败 |
| 第三评审 | 原子事实、人工标准、盲评分歧裁决 | 无依据强行选边 |

## 21. 六周完成计划

| 周次 | 必须完成的评测工作 | 周门禁 |
|---|---|---|
| 第1周 | 指标契约、目录、逐行记录结构、运行模板、责任人 | 每个必须功能都有判定依据和失败行为 |
| 第2周 | 60 个正式案例双标、核心覆盖清单、三个完整案例输入 | 人工标准冻结并有内容摘要；路径合法可计算 |
| 第3周 | 领域层、仓储层、路径和执行器测试，代码错误样本 | 三次重复结果一致；评测器异常独立分类 |
| 第4周 | 智能体配对导出、原子事实试标、串行和恢复测试 | 同一初稿可追踪；标注口径一致 |
| 第5周 | 三个完整案例、界面端到端测试、失败注入、安全扫描、招募试测者 | 干净目录无需手改即可跑通 |
| 第6周 | 正式60案例、正式原子事实和盲评、6–10 人试测、报告与证据索引 | 三项硬指标达标，全部分母可复算 |

内容和测试不能全部留到第六周；第2周必须完成人工标准和覆盖资产，第3–5周持续生成工程证据。

## 22. 发布门禁与未达标处理

### 22.1 必须全部满足

- 幻觉率 `<5%`；
- 难度适配准确率 `≥85%`；
- 核心知识覆盖率 `≥90%`；
- 60 个正式案例路径合法率 `100%`；
- 至少一个完整 Pandas 知识库、三类差异案例、三种资源可用；
- 现有第1版核心回归、必须完成的端到端测试、安全和隐藏资产检查通过；
- 评测器异常、智能体降级、磁盘失败、刷新和冲突均可恢复或安全终止；
- 原始逐行结构化记录、排除原因和重算脚本齐全。

### 22.2 未达标怎么办

| 未达标 | 优先排查 | 允许修复 | 禁止做法 |
|---|---|---|---|
| 幻觉率 | 来源、提示模板、猎手和裁判、代码预检 | 修复来源或门禁后全量重跑 | 删除错误原子事实 |
| 难度适配 | 人工标准、权重、阈值、脚手架规则 | 用调试集调整后冻结重跑 | 看正式结果后修改人工标准 |
| 覆盖率 | 缺讲义、活动、代码或降级资源 | 补充资产和引用 | 随意取消核心标记 |
| 多智能体无增益 | 角色重叠、审核弱、裁判越权改写 | 调整角色合同并保留旧结果 | 挑选有利配对 |
| 用户阻塞 | 页面、错误或恢复不清 | 简化交互、采用降级资源 | 隐藏失败案例 |
| 外部评测器失败 | 安装、接口或隔离问题 | 切换 Node 本地兜底 | 阻塞全部演示 |

无法在截止前达标时必须使用 06 号文档的降级方案，并在演示文稿、视频和“已知限制.md”中明确披露，不能把未完成能力写成已通过。
