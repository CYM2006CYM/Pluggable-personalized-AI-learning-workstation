## 1. 一页理解本项目

### 1.1 赛题希望解决什么问题

传统在线课程通常给所有学习者相同的讲义、题目和顺序；直接使用单个大模型生成学习材料，又容易出现三个问题：

1. 它并不真正了解学习者已经会什么、缺什么；
2. 生成内容可能不准确，尤其在专业领域容易产生“幻觉”；
3. 学习资源通常生成一次就结束，不能根据学习过程继续调整。

XH-202630 因此要求参赛者建立一个完整系统：先分析学习者，再由多个职责明确的 Agent 协同生成和审核专业学习资源，最后根据学习反馈动态更新决策。系统还要提供至少三种形态的个性化资源、可视化学情和路径，并用测试证明内容准确、难度匹配和知识覆盖。

### 1.2 我们准备做什么

我们要做的是一个“可插拔的 AI 个性化学习工作台”：

- 教师或内容维护者把某个领域的知识、来源、学习目标、练习、代码任务和评分标准制作成统一的 **Profile 资料包**；
- 学习者选择目标后，系统通过背景问卷和固定诊断题建立初始学情画像；
- 程序根据知识先修关系生成一条合法、可解释的个性化学习路径；
- 多个 Agent 分别负责诊断解释、资源生成、内容质疑、争议辩护、最终裁决和学习决策；
- 学习者在阅读讲义、查看代码或完成任务的任何阶段都可以针对当前内容提问；
- 系统根据客观题、代码测试、提示使用情况和实操结果更新画像与后续路径。

六周内，我们用“Pandas 数据清洗”把这条链路完整跑通。以后如果要扩展 Python、机器学习、工业软件或其他技能领域，主要工作是安装新的 Profile 资料包，而不是重新开发一套产品。

### 1.3 一句话方案

> 用 Profile 固定可信知识和评测边界，用代码保证路径与判分确定性，用多 Agent 负责个性化生成和交叉审核，用持续学习证据驱动下一步决策。

## 2. 用户与使用场景

### 2.1 主要用户

| 用户 | 当前困难 | 系统怎样帮助 |
|---|---|---|
| 有编程基础的计算机专业学生 | 基础内容重复，课程不能快速进入应用和边界问题 | 诊断后跳过已证明掌握的先修内容，提高难度并减少脚手架 |
| 非计算机背景的自学者 | 不理解 DataFrame、类型和代码错误，容易在第一步卡住 | 补充必要先修，提供分步讲义、更多示例和渐进提示 |
| 实践导向学习者 | 不愿先读大量理论，希望通过任务学习 | 先展示真实任务与验收标准，再按错误回补相关知识 |
| 教师／资料包维护者 | 课程内容难复用，AI 生成质量难控制 | 通过统一 Profile 管理知识、来源、任务、Rubric、测试和降级资源 |
| 比赛评审／研究者 | 难判断“个性化”和“多 Agent”是否真实有效 | 查看完整输入、中间决策、Agent 审核轨迹、最终资源和量化评测 |

### 2.2 典型学习场景

学习者希望掌握“使用 Pandas 清洗一份脏订单数据”。系统不是直接给一篇统一教程，而是先判断：

- 是否具备基本 Python 和 Pandas 经验；
- 是否理解 DataFrame 的行列、类型和缺失值；
- 更适合概念先行、平衡式学习还是任务先行；
- 当前有多少可用学习时间。

然后系统可能生成三种不同安排：

- 计算机专业学生：快速检查基础，直接进入重复键、类型边界与结果验证；
- 非计算机自学者：先建立 DataFrame 心智模型，再逐步完成缺失值和类型清洗；
- 实践导向学习者：先看脏数据和最终要求，在解决问题时穿插必要概念。

三人的最终专业标准相同，但路径顺序、难度、讲解方式、示例、脚手架和反馈不同。这才是本项目所说的“个性化”。

## 3. 学习者实际经历的完整流程

~~~text
选择资料包与学习目标
        ↓
背景问卷 + 固定诊断题
        ↓
生成初始学情画像，诊断 Agent 解释依据
        ↓
路径引擎按知识先修图生成合法路径
        ↓
(学习决策 Agent 解释“为什么这样安排”)
        ↓
陪伴教师 Agent 生成当前节点的个性化学习卡片
        ↓
<< (遇到有争议的决策) 猎手 Agent 质疑 → 必要时辩护 Agent 回应 → 裁判 Agent 定稿 >>
        ↓
学习者阅读，并可对任意文字、代码、报错或路径节点提问（AI解答）
        ↓
无分自测 → 客观题（选择+判断） → 代码补全 → 独立实操
        ↓
程序判分并收集证据
        ↓
更新知识点掌握度、置信度、错误类型和路径
        ↓
进入下一节点，直到完成目标
~~~

普通学习后客观题的执行边界是：AI负责在资料包知识点、来源和题型约束内生成单选/判断候选题；程序校验结构并直接比较 `correct_answer`，不再让模型决定这两类题是否答对。动态题生成或审核失败时，按知识点使用1至2道固定兜底题；固定题不足4道时标记“验证证据不足”，不能虚报题组通过。初始诊断和正式效果评测题始终使用彼此隔离的固定资产。

内容质量 CIDPP 只评价讲义、题面、反馈等教学内容的清晰性、完整性、深度、实用性和针对性：构建期由单一评价 Agent 评价，最多自动优化一次；只有高风险事实争议才进入 Hunter/Judge，不能覆盖程序硬校验。CIDPP 不评价学习者掌握度，也不能直接改写路径。学习者评价采用程序计算的 `mastery/confidence` 事实底座，并在其上正式保存 Pandas 五维能力画像；画像评分失败保留旧快照，不影响主链。

这个流程对应赛题要求的“学情画像构建 → 多智能体协同调度 → 个性化生成 → 交互反馈 → 动态决策更新”闭环。

## 4. 功能全景

### 4.1 学习前：选择目标与建立画像

1. **资料包选择**：查看可用领域、版本、支持的学习活动和质量状态。
2. **学习目标选择**：例如“独立完成一份 Pandas 数据清洗任务”。
3. **背景问卷**：收集经验、目标、可用时间和学习偏好，不采集不必要的真实身份信息。
4. **固定诊断题**：使用资料包内经过审核的题目检查真实基础，避免完全依赖自我评价。
5. **诊断报告**：显示已证明掌握、需要补足和尚未验证的知识，以及每项结论的证据和置信度。

### 4.2 学习中：路径、资源和随时提问

1. **个性化路径图**：展示当前节点、先修关系、预计时间、完成状态和安排理由。
2. **详细学习卡片**：包含学习目标、安排原因、先修知识、核心概念、分步解释、正确代码及输出、反例、常见误区、与最终任务的关系、无分自测、来源和相关知识。
3. **上下文问答**：可选择一段文字、一段代码、一个报错或路径节点直接提问，不必等到做错题。
4. **渐进提示**：作答前只给概念方向、定位步骤和局部提示；提示使用会被记录，但“提问”本身不会导致掌握度下降。
5. **协作审核可视化**：显示资源生成、猎手核查、条件辩护和裁判定稿的状态与改动摘要。

### 4.3 学习活动：至少三类资源并形成递进

本项目实际提供四类资源形态，超过赛题“至少三种”的要求：

| 资源 | 作用 | 个性化方式 | 如何判断效果 |
|---|---|---|---|
| 学习卡片／讲义 | 建立概念和方法 | 表达、示例、详略、脚手架 | 无分自测和后续任务证据 |
| 客观题 | 检查理解和概念辨析 | 难度、题面情境、提示和反馈 | AI生成后由代码比较`correct_answer`；固定诊断和故障兜底分开管理 |
| 短答题 | 开放表达与解释 | 题面和反馈 | 旧v1继续使用受约束语义判题；Pandas v2六周主链暂不加入动态简答 |
| 带脚手架的代码补全 | 从理解过渡到编写 | 空位数量、注释、API 候选、提示级别 | 公开测试和隐藏测试 |
| 独立实操任务 | 验证迁移和综合应用 | 情境表达、顺序和错误反馈 | 固定 Rubric 与隐藏测试 |

### 4.4 学习后：证据、反馈与动态调整

系统记录知识点、作答结果、代码测试、错误类型、使用过的提示和独立完成程度。完成活动后：

- 更新知识点掌握度和置信度；
- 区分“答错”“尚未验证”和“主动提问”；
- 连续正确时逐步减少脚手架并提高挑战；
- 出现先修性错误时插入短回补节点；
- 展示本次路径为什么改变；
- 保存会话并支持下次恢复。

### 4.5 教师和资料包维护功能

- 从 Markdown/TXT 源材料构建资料包草稿；
- 校验资料包目录、引用、来源和安全路径；
- 审核草稿并由维护者确认启用；
- 安全修订 active 资料包，旧版本自动归档；
- 为新领域复用同一格式接入目标、活动、测评、Rubric 和数据集。

### 5.1 为什么同时保留 Web 和 Pi TUI

当前基础框架已经是一个可运行的 Pi 扩展，拥有 /study、/study-recover、/study-profile、/study-build、/study-revise 等命令，以及 Profile 生命周期、会话记录、Graph 隔离执行和大量回归测试。

代码补全和独立实操需要编辑器、运行输出、数据预览和测试结果，因此：

- React Web 是完整学习工作台；
- Pi TUI 继续负责现有文字学习和资料包管理；
- TUI 遇到代码活动时保存同一会话，并引导打开本地 Web；
- 两个入口共用应用层、Profile、学情和学习记录，不建立两套产品逻辑。

当前框架事实与改造边界见 [09-当前基础框架技术架构与扩展边界](./09-当前基础框架技术架构与扩展边界.md) 和 [10-新版基础框架适配性诊断](./10-新版基础框架适配性诊断.md)。

### 5.2 代码和 Agent 如何分工

**代码负责不可妥协的规则：**

- Profile 版本、安全路径、文件和引用校验；
- 知识先修约束、合法路径和节点状态；
- 客观题答案、代码公开／隐藏测试和 Rubric 汇总；
- 掌握度证据权重、答案可见阶段、超时和降级；
- 是否允许跳过节点、是否完成任务。

**Agent 负责需要语言理解和个性表达的工作：**

- 解释学情和路径理由；
- 按画像生成讲义、示例、题面、提示和反馈；
- 回答学习过程中的上下文问题；
- 发现内容风险、提出辩护并作最终裁决；
- 对开放式回答做受约束的语义判断。

固定知识、来源、先修关系、Rubric、参考结果、隐藏测试和数据集不能被 Agent 任意改写。

## 6. 可插拔资料包：产品能够扩展的关键

### 6.1 Profile 是什么

Profile 是一个可以被安装、校验、启用、修订和归档的领域学习资料包。应用本身不知道“缺失值”或“重复数据”是什么意思，它只认识统一的 Profile 契约；Pandas 专业内容全部由 pandas-cleaning Profile 提供。

### 6.2 Profile v2 包含什么

~~~text
pandas-cleaning/
├─ profile.json          唯一身份、版本、能力与路径声明
├─ subject.md            领域说明
├─ knowledge_index.json  知识点与先修关系
├─ cards/ chapters/      固定学习材料和章节
├─ source_map.json       专业事实来源
├─ goals/                可选学习目标
├─ activities/           讲义、题目、代码任务模板与 fallback
├─ assessments/          诊断题、计分题、公开/隐藏测试
├─ rubrics/              实操评分标准
└─ datasets/             人工放置并校验的只读数据
~~~

现有资料包视为 Profile v1，仍可使用原来的复习功能；v2 在原有格式上增加完整学习能力。profile.json 仍是唯一清单文件，不再新增另一套 manifest。

### 6.3 如何保证资料包可靠

- 所有知识点、目标、活动、来源、Rubric 和数据集引用必须完整；
- 知识先修图必须无环；
- 数据集由维护者人工放入并校验哈希，Agent 不生成或修改 CSV 数据资产；
- 固定答案和隐藏测试不进入教学 Agent 上下文；
- 动态资源失败时，只能降级到同一知识点、同一阶段的审核过 fallback；
- 只有通过结构与质量检查的 draft 才能由维护者启用为 active。

完整规范见 [02-可插拔资料包规范](./02-可插拔资料包规范.md)，首个示范包见 [07-Pandas示范资料包设计](./07-Pandas示范资料包设计.md)。

## 7. 多智能体如何协同

### 7.1 角色体系

本项目采用“五个常驻角色，加一个条件触发角色”：

| 角色 | 核心职责 |
|---|---|
| 诊断 Agent | 把程序计算出的学情解释成学习者能理解的报告 |
| 陪伴教师／资源生成 Agent | 生成学习卡片、个性化题面、提示、反馈并回答普通问题 |
| 猎手 Agent | 主动寻找事实错误、代码错误、来源不足、难度越级和答案泄漏 |
| 裁判 Agent | 根据原稿、猎手意见和必要的辩护形成最终版本 |
| 学习决策 Agent | 解释代码已经生成的合法路径和路径变化 |
| 辩护 Agent（条件触发） | 只有猎手提出实质争议时，才为原稿逐项申辩或承认问题 |

现有 study_grade_answer 是判题 Graph 能力，不单独算作业务 Agent。

### 7.2 一份学习资源怎样产生

~~~text
陪伴教师生成原稿
      ↓
猎手检查事实、来源、代码、难度和泄漏
      ↓
没有实质争议 ───────────────┐
有实质争议 → 辩护 Agent 回应 │
                             ↓
                        裁判 Agent 定稿
                             ↓
                    通过 schema 和代码预检
                             ↓
                       展示给学习者
~~~

如果生成或审核失败，系统展示资料包内的固定审核版本，而不是把未审核原稿直接交给用户。

### 7.3 与 Pi 和 Loop Graph SDK 的关系

当前 SDK 支持串行 Graph、条件路由、schema 校验、delegate、隔离 AgentSession 和 trace，但不提供并行 fork/join 和内建 Agent 群聊。因此本项目采用真实可实现的“串行隔离子图协作”：

- 每个角色使用独立 AgentSession；
- 角色之间只交换结构化产物，不共享隐式聊天历史；
- 应用层决定是否触发辩护、如何重试、如何保存 trace 和 fallback；
- UI 展示执行顺序、结论和改动摘要。

这既满足赛题的多智能体协作和辩论要求，也不夸大现有 SDK 能力。详细设计见 [03-多智能体与Pi集成设计](./03-多智能体与Pi集成设计.md)。

### 7.4 如何比较测试 "单 Agent" 和 "多 Agent"

单 Agent 基线是陪伴教师第一次通过格式校验的原稿；多 Agent 结果是**同一份原稿**经过猎手、条件辩护和裁判后的终稿。两者不重新生成，因此比较的是审核机制本身，而不是两次随机生成的差异。

```text
同一个画像、同一个目标、同一个资料包
          ↓
生成 Agent 的原始稿 ─────────→ 作为“无审核基线”
          ↓
猎手找错 → 必要时辩护 → 裁判裁决 → 修订
          ↓
最终稿 ─────────────────────→ 作为“多 Agent 审核结果”
```

## 8. 学情诊断和个性化路径

### 8.1 学情不是一个模糊标签

系统区分三层信息：

1. **初始诊断**：问卷和固定诊断题形成进入目标前的快照；
2. **会话证据**：每次客观题、代码测试、实操、提示和讨论的可观察事实；
3. **长期画像**：跨会话聚合的知识点掌握度、置信度、错误类型和建议。

每个知识点状态分为“已掌握、学习中、需要支持、尚未验证”。“用户提问”只是软证据，不会直接降低掌握度；没有做过的内容是“尚未验证”，也不能被误写成“薄弱点”。

### 8.2 路径怎样产生

路径不是让 Agent 自由编排。程序执行以下步骤：

1. 从学习目标找出全部核心知识及先修闭包；
2. 根据诊断证据判断哪些节点可以跳过；
3. 对剩余节点做拓扑排序，保证不会先学后置内容；
4. 结合知识重要度、错误证据、学习偏好和时间预算选择顺序；
5. 为每个节点决定目标难度和脚手架等级；
6. 产生明确的原因代码，再由学习决策 Agent 解释给用户。

学习过程中，只有可靠证据越过阈值、发现新的先修错误或用户改变目标／时间时，才重算未完成路径，避免每做一道题路径就来回抖动。

### 8.3 怎样实现真正的差异

个性化至少作用于：

- 学什么以及先后顺序；
- 初始难度和后续升降；
- 讲解详略、类比和代码示例；
- 代码补全中保留多少空位、提供何种提示；
- 错误后插入哪个回补节点；
- 反馈重点和下一步建议。

目标、专业事实和最终通过标准不会因用户不同而降低。详细算法和字段见 [11-学情诊断与个性化路径设计](./11-学情诊断与个性化路径设计.md)。

## 9. 学习卡片、代码任务和判分

### 9.1 一张完整学习卡片

学习卡片不是几段临时生成的文字，而是当前路径节点的完整教学单元，必须包含：

1. 学习目标；
2. 为什么此时安排该节点；
3. 前置知识；
4. 核心概念；
5. 分步解释；
6. 正确代码与预期输出；
7. 反例和常见误区；
8. 与最终实操任务的关系；
9. 无分自测；
10. 来源；
11. 相关知识；
12. 按当前画像调整的脚手架。

### 9.2 代码任务怎样生成

Agent 不从零发明代码题。Profile 先固定任务目标、起始代码模板、允许编辑区域、参考契约、公开／隐藏测试和 Rubric。Agent 只能根据脚手架等级调整：

- 空位数量；
- 注释和步骤；
- 是否给出 API 候选；
- 题面情境和渐进提示。

这样既能个性化，又不会让每个人面对完全不同、无法比较的评分标准。

### 9.3 怎样运行和判分

React Web 使用 Monaco 作为编辑器，使用锁定版本的 Pyodide 和 Pandas 在 Web Worker 中运行代码：

- “运行”只执行代码和公开测试，帮助学习者调试；
- “提交”在干净环境中执行公开测试、隐藏测试和 Rubric；
- 无限循环或超时会终止 worker；
- 测试系统自身异常记为 evaluator error，不算学习者答错；
- 隐藏测试只返回失败维度，不暴露断言和参考实现。

完整执行器设计见 [12-学习活动代码任务与执行器设计](./12-学习活动代码任务与执行器设计.md)。

### 9.4 防止答案泄漏

| 阶段 | 可以做什么 | 不能做什么 |
|---|---|---|
| 学习卡片阶段 | 正常解释概念、代码和扩展问题 | 暴露后续隐藏测试 |
| 计分活动提交前 | 给概念和渐进提示 | 给当前答案、参考程序、隐藏断言 |
| 独立实操进行中 | 解释报错、API 和定位方法 | 直接提供完整最终程序 |
| 活动完成后 | 完整复盘答案和替代方案 | 暴露隐藏测试源码 |

权限由服务端裁剪数据，不能只靠提示词约束 Agent。

## 10. 产品界面与交互

### 10.1 React 学习工作台

~~~text
顶栏：资料包｜目标｜会话状态｜Agent 审核状态
左栏：学情画像摘要＋个性化学习路径
主区：学习卡片／题目／Monaco 编辑器／运行结果
右栏：随选随问＋来源＋审核详情
底栏：保存｜提示｜运行／提交｜下一节点
~~~

核心页面包括：

1. 资料包和学习目标；
2. 背景问卷与诊断；
3. 学情报告和路径图；
4. 详细学习卡片；
5. 客观题和代码活动工作台；
6. 学习总结与路径变化。

### 10.2 学习过程中的提问

用户可选择卡片段落后“问这里”，也可选择代码或运行错误提问。问题会携带当前内容锚点和学习阶段：

- 普通、来源内问题由陪伴教师快速回答；
- 涉及来源外新结论、版本差异或专业争议时进入猎手—辩护—裁判审核；
- 切换节点后，旧问题仍绑定原内容，不污染新节点。

### 10.3 状态和失败体验

界面必须明确显示：资料加载、内容生成、猎手核查、辩护是否触发、裁判定稿、Pyodide 加载、代码运行、暂无结果、可恢复错误和 fallback。刷新或关闭页面后，未提交代码和当前会话可以恢复。

完整界面设计见 [04-产品界面与交互设计](./04-产品界面与交互设计.md)。

## 11. Pandas 首个示范资料包

六周 MVP 只完整实现一个垂直领域：Pandas 数据清洗。学习目标是让学习者独立完成一份小型脏订单数据的清洗和验证，覆盖：

1. 读取 CSV；
2. 检查 DataFrame；
3. 处理缺失值；
4. 处理重复数据；
5. 清理类型和格式；
6. 验证清洗结果。

示范数据使用 30–50 行匿名合成订单，不允许用户任意上传 CSV。最终产出变量 clean_df，通过固定 Rubric、公开测试和隐藏测试判分。

选择这个范围有三个原因：

- 它是明确的垂直技能训练任务，符合赛题定位；
- 既包含专业知识，也能产生讲义、题目、代码补全和实操等多种资源；
- 六周内可以完成可信内容、执行环境和系统化评测，而不是做一个范围很大但无法验证的演示。

## 12. 效果评测与赛题证明

### 12.1 赛题硬指标

| 指标 | 目标 | 我们如何测 |
|---|---:|---|
| 专业知识幻觉率 | 小于 5% | 把终稿拆成原子事实，对照允许来源逐项复核 |
| 学习者画像—资源难度适配准确率 | 不低于 85% | 用预先标注 gold 的离线画像案例检查 |
| 核心知识点覆盖率 | 不低于 90% | 检查核心点是否同时有讲解、评测和应用证据 |

### 12.2 评测资产

- **60 个离线画像案例**：20 名计算机专业学生、20 名非计算机自学者、20 名实践导向学习者；
- **3 个完整差异化案例**：完整保存输入画像、中间决策、Agent 轨迹和最终资源；
- **20 组单／多 Agent 配对盲评**：同一原稿与审核终稿随机匿名比较；
- **6–10 名真实学习者探索性测试**：前测、系统学习、后测和迁移题；
- 工程测试：Profile 兼容、路径合法、Graph 合同、Pyodide、答案安全、失败恢复和现有功能回归。

真实用户样本较小，只报告可用性和学习趋势，不夸大为大样本因果结论。完整口径见 [05-测试集与效果评测方案](./05-测试集与效果评测方案.md)，赛题逐项映射见 [08-XH-202630赛题要求对照表](./08-XH-202630赛题要求对照表.md)。

## 13. 六周总体安排

六周的总体趋势是：**先冻结底层契约和可信内容，再打通诊断、路径和代码执行，然后完成多 Agent 与界面，最后集中验证和提交。**

| 周次 | 建设重点 | 本周必须看到的结果 |
|---|---|---|
| 第 1 周：打地基 | 冻结 Profile v2、LearningActivity 和共享应用接口；保留旧框架兼容 | v1 继续可用，v2 非法包能被拒绝，Pandas 包骨架建立 |
| 第 2 周：完成个性化决策 | 固定诊断题、知识状态、路径引擎和 60 案例 gold | 三类画像能生成不同但合法的路径 |
| 第 3 周：完成学习活动 | 学习卡片模型、Monaco、Pyodide、代码补全和独立实操 | 至少一套代码补全和一套实操可稳定判分 |
| 第 4 周：完成智能协作 | 陪伴教师、猎手、条件辩护、裁判、上下文问答和 fallback | 同一原稿可追踪到审核终稿，失败时可安全降级 |
| 第 5 周：完成产品闭环 | React 全流程、TUI→Web 交接、Pandas 六模块和三类完整案例 | 不修改后台文件即可完整演示诊断到画像更新 |
| 第 6 周：评测与提交 | 全量回归、60 案例、20 对盲评、用户试测、报告和视频 | 三项指标达标，形成可复查提交包 |

### 13.1 六周明确不做

登录、云数据库、公开部署、任意 CSV 上传、移动端专项应用、第二个完整领域包、真正并行 Agent 网络、模型微调和复杂教师后台均不进入 MVP。

如果进度紧张，优先削减动画、复杂图表和非核心题目变体；不得削减 Profile 校验、合法路径、代码确定性判分、防答案泄漏、fallback 和评测证据。详细安排见 [06-六周MVP范围与验收](./06-六周MVP范围与验收.md)。

### 14.1 当前已经具备

- 完整的 TypeScript 分层项目；
- Pi 的五个学习与 Profile 管理命令；

- 三种学习模式、五种难度和四种题型；
- 学习会话、Attempt、SessionEvidence 和长期画像持久化；
- 九个学习／资料包 Graph、隔离 AgentSession、schema 重试和 trace；
- 21 个测试文件及 typecheck、smoke、release 检查。

### 14.2 六周必须新增

- Profile v2 及 Pandas 完整资料包；
- 初始诊断画像、知识点状态和个性化路径引擎；
- LearningActivity、活动证据、代码补全和实操；

## 15. 成功标准与最终交付

六周结束时，一个不了解项目的评审应能亲眼看到：

1. 同一 Pandas 目标下，三类学习者得到不同且有依据的画像和路径；
2. 学习资源由多个职责明确的 Agent 生成、质疑、必要时辩护并裁决；
3. 学习者能在学习过程中随时针对文字和代码提问；
4. 讲义、题目、代码补全和独立实操形成递进，而不是彼此孤立；
5. 代码任务能真实运行、确定性判分，并安全保护隐藏测试和答案；
6. 完成活动后，画像和后续路径发生可解释的更新；
7. Pandas 内容来自可替换的 Profile v2，而不是写死在产品代码中；
8. 幻觉率、难度适配和知识覆盖达到赛题目标，并有原始测试记录可复查；
9. 原有 Pi 学习和资料包管理能力仍能正常工作。

最终交付包括：可运行源码、Pandas Profile v2、React 学习工作台、Pi TUI 兼容入口、自动测试、60 个离线案例、3 个完整差异化案例、20 组盲评、真实学习者试测报告、设计文档、部署与使用说明、PPT 和 10 分钟以内的完整闭环演示视频。

---

如果只记住本项目的一件事，应当是：

> 我们不是让多个 Agent 随机生成更多内容，而是用一个可插拔的可信资料包规定“学什么、怎样验证”，用程序保证路径和判分正确，再让多个 Agent 围绕学习者状态生成、质疑和改进“怎样教”，并根据真实学习证据持续决定“下一步学什么”。
