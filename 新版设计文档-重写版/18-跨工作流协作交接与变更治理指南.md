# 18 跨工作流协作交接与变更治理指南

> **D14同步**：负责人统一拍板架构、字段、接口、错误码、评分、目录和安全边界；组员只实现、制作资产、写 fixture、测试和记录实测。任何缺口必须问题单上报，不得在本工作流临时修补公共合同。交接以 [第一周总任务书](./第一周任务/19-第一周总任务布置与权限边界.md)、[公共合同总册](./第一周任务/21-第一周公共合同总册.md) 和各岗位任务书为准。

> 文档状态：六周团队协作执行规范。  
> 适用团队：4至5名成员，可分工开发，也允许一人兼任两个工作流。  
> 文档目的：把06号中的“生产方—使用方—独立验收方”规则转化为每天可以执行的交接、变更、复核和Git协作流程。  
> 适用范围：领域与架构、资料包与内容、执行器与后端、Agent与模型、网页与测试五条工作流。

## 1. 这里的“跨工作流”是什么意思

“跨工作流”不是指成员必须位于不同地理位置，而是指不同成员或同一成员以不同职责开发不同模块。即使团队成员坐在一起，只要一个功能跨越Profile、领域规则、执行器、Agent、网页或测试，就必须执行交接。

本项目中的一项完整功能通常经过：

```text
资料包与内容定义专业事实和业务契约
→ 领域与架构把契约实现成确定性规则和应用用例
→ 执行器与后端产生确定性代码结果
→ Agent与模型生成或解释非权威内容
→ 网页展示安全视图
→ 测试独立验证并保存证据
```

各工作流不能把自己的中间产物直接当成正式业务事实。例如：Agent候选不是正式题目，执行器进程结束不是Evidence已经提交，React显示的路径不是权威路径，内容作者写出的参考答案也需要校验和复核。

## 2. 三方关系的含义

每个跨工作流交付使用“生产方—使用方—独立验收方”三方关系。

### 2.1 生产方

生产方定义并实现某项公共能力，对合同正确性、最小测试和变更记录负责。例如：

- 领域与架构生产Profile schema和安全DTO；
- 资料包与内容生产Pandas资料、诊断题和固定任务；
- 执行器与后端生产 `ActivityResult`和环境锁；
- Agent与模型生产候选内容及审核轨迹；
- 测试身份生产正式指标和报告。

生产方不等于整条功能的唯一负责人，也不能自行宣布交付通过。

### 2.2 使用方

使用方通过公开合同调用生产方交付的能力。使用方必须处理正常、空、错误、冲突和降级结果，但不能修改生产方拥有的规则。

例如网页可以根据 `ActivityResult`显示“评测器异常”，但不能把服务端返回的学习者错误自行改成评测器错误；执行器可以读取Rubric并计算结果，但不能自行改变Rubric权重。

### 2.3 独立验收方

独立验收方不参与该交付物的主要制作，负责确认：

- 合同字段和行为与文档一致；
- 使用方可以按公开接口完成最小接入；
- 正常、边界、错误和恢复场景可复现；
- 没有泄漏答案、隐藏测试、密钥或绝对路径；
- 没有破坏Profile v1或其他已冻结能力；
- 版本、测试和已知限制记录完整。

“独立”指验收职责独立，不强制必须有第三名专职成员。小团队可以由另一工作流成员交叉复核，但不能由同一人制作、使用并独自宣布核心交付通过。

## 3. 标准交接五步法

### 3.1 第一步：生产方冻结交付合同

交付前必须说明：

1. 交付物名称和版本；
2. 生产方、使用方和验收方；
3. 输入字段及信任边界；
4. 输出字段及业务含义；
5. 正常、空、错误、冲突和恢复行为；
6. 安全限制和禁止暴露内容；
7. 最小调用示例；
8. 对应测试和运行命令；
9. 已知限制和降级方式；
10. 兼容或不兼容变更的处理方式。

只提交一个TypeScript接口或一份说明文档不算完成交接。接口必须同时具备行为说明、测试样例和错误语义。

### 3.2 第二步：提供可运行样例

生产方至少提供：

- 一个正常样例；
- 一个边界样例；
- 一个错误样例；
- 一个固定fixture；
- 一条最小验证命令；
- 一个安全或降级样例。

例如固定代码任务必须同时提供题面、起始代码、可编辑区域、公开/隐藏测试、参考实现、已知错误实现、Rubric和环境引用；不能只交题面。

### 3.3 第三步：使用方完成接入验证

使用方检查：

- 能否只依赖公开合同完成接入；
- 是否错误读取生产方内部文件或私有字段；
- 是否覆盖正常、空、错误、冲突和恢复状态；
- 是否能识别schema、revision、协议或会话版本不一致；
- 是否按合同执行固定降级；
- 是否错误复制了一套业务规则。

接入失败时，使用方提交问题，不在自己的层临时改变生产方语义。

### 3.4 第四步：验收方独立检查

验收方使用冻结合同和fixture检查，而不是只观看生产方演示。最低验收包括：

- 自动测试通过；
- 使用方完成最小接入；
- 错误和降级可复现；
- 敏感资产扫描通过；
- 旧功能没有回归；
- 版本号、摘要和变更记录正确；
- 已知限制没有被写成已支持能力。

### 3.5 第五步：保存交接记录

交接记录使用Markdown或结构化JSON保存。最低模板如下：

```text
交付编号：handoff-<日期>-<序号>
交付物：ActivityResult
合同版本：evaluation-protocol-v1
生产方：执行器与后端
使用方：领域与架构、网页与测试
验收方：领域与架构中的非作者成员、测试身份
状态：proposed / integrating / accepted / rejected / superseded
Profile revision：如适用则填写
相关测试：测试文件或命令
正常样例：固定fixture编号
错误样例：固定fixture编号
安全检查：通过/失败及证据
已知限制：例如memory_limit暂不支持可靠归因
回退方式：Node.js本地适配器
冻结时间：ISO 8601时间
后续变更：兼容规则或升级要求
```

`accepted`只表示本版本通过交接；后续产生新版本时，旧记录标为 `superseded`，不能删除。

## 4. 跨工作流问题报告

发现跨组问题时，发现者必须提交以下四项：

1. **复现输入**：Profile revision、sessionVersion、活动、代码、模型或fixture编号；
2. **期望合同**：引用的字段、状态、错误码或规则；
3. **实际结果**：日志摘要、返回DTO、截图或测试失败；
4. **影响范围**：哪些用例、页面、指标、资产或旧版本受影响。

推荐问题模板：

```text
问题编号：crossflow-<日期>-<序号>
发现工作流：网页与测试
应处理工作流：领域与架构
复现输入：profileRevision=3，sessionVersion=5，fixture=path-case-07
期望合同：缺失值处理必须位于结果验证之前
实际结果：服务端路径中结果验证先于缺失值处理
影响范围：路径页、getNextStep、恢复、60案例路径合法率
临时安全措施：阻止确认该路径；不在React中重新排序
```

问题由拥有该规则决策权的工作流修改。发现方可以阻止错误结果继续传播或启用既定fallback，但不能在自己的层改变业务事实。

## 5. 问题归属速查

| 问题 | 主要修改方 | 其他工作流可采取的临时措施 | 禁止做法 |
|---|---|---|---|
| Profile字段、引用或revision校验错误 | 领域与架构 | 拒绝坏包，继续使用旧active | 内容或网页自行绕过校验 |
| Pandas事实、来源、诊断答案或题面错误 | 资料包与内容 | 暂停相应资产，使用旧revision | 架构组凭个人理解改专业事实 |
| 掌握度、Evidence或路径顺序错误 | 领域与架构 | 阻止确认非法路径，保留复现数据 | React/TUI自行重排或重算 |
| 代码判分、超时、协议或错误归因错误 | 执行器与后端 | 保存代码和环境摘要，返回评测器异常 | 网页猜测结果或内容组改分 |
| Rubric或TaskContract含义不清 | 内容与架构共同确认 | 执行器拒绝准备任务 | 执行器自行选择权重或业务键 |
| AI越界、来源不足或泄漏答案 | Agent与模型 | 拒绝候选，使用固定fallback | 内容组事后扩展资料包迁就输出 |
| 页面状态、交互或文案错误 | 网页 | 保留服务端事实，修正展示 | 为修页面改变服务端错误语义 |
| 指标脚本、分母或报告错误 | 网页与测试中的测试身份 | 保留旧运行，创建修复任务 | 覆盖旧报告或删除失败记录 |

## 6. 三级变更规则

### 6.1 一级：局部兼容变更

适用于：

- 新增可选字段；
- 修改中文提示；
- 修复不改变外部行为的内部实现；
- 增加不改变业务语义的测试；
- 增加安全日志摘要。

流程：

```text
生产方修改
→ 补充测试和变更记录
→ 通知使用方
→ 使用方运行回归
→ 验收方确认
```

通常不升级主合同版本，但必须记录修改时间和影响范围。

### 6.2 二级：跨工作流兼容变更

适用于：

- 新增错误码；
- 新增活动或会话状态；
- 增加安全DTO字段；
- 增加Rubric维度；
- 改变缓存失效条件；
- 增加新的降级状态。

流程：

```text
提出变更说明和兼容策略
→ 生产方与全部使用方评审
→ 先更新合同、fixture和测试
→ 生产方实现兼容输出
→ 使用方完成适配
→ 独立验收
→ 在同一版本窗口合并
```

不能先修改一端，再要求其他工作流以后适配。旧使用方尚未完成适配时，生产方必须保留兼容输出或推迟合并。

### 6.3 三级：不兼容或正式资产变更

适用于：

- 删除、重命名或改变必填字段；
- 改变路径状态、Evidence或错误语义；
- 改变 `ActivityResult`合法组合；
- 修改正式资料、诊断、数据、代码任务、Rubric或测试；
- 修改正式人工标准、评测集或计算口径。

必须版本化：

| 变更对象 | 必须采取的动作 |
|---|---|
| Profile结构 | 升级 `schemaVersion`，提供兼容或迁移测试 |
| 正式资料内容 | 创建新的Profile `revision`，旧会话继续使用旧revision |
| HTTP公共接口 | 升级接口合同版本或提供兼容层 |
| 代码评测协议 | 升级 `evaluatorVersion`并补双版本契约测试 |
| Agent提示或角色合同 | 升级 `promptVersion`并使相关缓存失效 |
| 环境版本或资源阈值 | 创建新 `environmentHash`并重跑适配测试 |
| 正式效果评测 | 创建新 `evaluationId`，保留旧运行和作废原因 |

三级变更必须先评估对Profile、会话恢复、缓存、前端、评测和历史数据的影响，不得直接覆盖旧资产。

## 7. 典型交接与变更案例

### 7.1 网页发现路径错误

网页收到服务端路径 `读取→验证→缺失处理`，但知识图要求缺失处理先于验证。

正确处理：

```text
网页保留原始响应并阻止确认非法路径
→ 提交复现输入、期望先修和实际路径
→ 领域与架构修复PathEngine
→ 补领域测试和60案例回归
→ 网页重新接入服务端结果
```

网页不得自行重排，否则会造成Web、TUI、Evidence和恢复路径不一致。

### 7.2 执行器发现Rubric歧义

题面要求“删除重复订单”，但未说明使用全行还是 `order_id` 判重。

正确处理：

```text
执行器拒绝冻结测试
→ 内容组确认业务规则和来源
→ 架构组确认TaskContract字段
→ 内容组创建新Profile revision
→ 执行器按新revision生成/验证测试
→ 测试组验证旧会话仍使用旧revision
```

执行器不得自行选择业务键或调整权重。

### 7.3 Agent生成资料包外知识

Agent使用了尚未教学且不在API白名单中的高级Pandas能力。

正确处理：

```text
确定性白名单拒绝候选
→ 记录越界知识点和API
→ Agent组修正输入裁剪或提示
→ 重新生成并执行审核
→ 仍失败则使用固定兜底
```

不能为了迁就模型输出，事后临时扩展资料包知识范围。

### 7.4 评测器错误被误记成学习者错误

环境缺少锁定依赖，却返回 `learner_error/runtime_error`。

正确处理：

```text
网页只展示收到的安全结果并保存复现编号
→ 执行器组修正阶段归因
→ 新增dependency_missing测试
→ 应用层验证不创建负Evidence
→ 网页验证“不是你的代码错误”状态
```

网页不得自行把任意运行错误改为系统错误；错误归因必须在执行器/协议层修复。

### 7.5 正式评测后发现人工标准错误

不得直接编辑旧 `final-60.jsonl`、覆盖报告或删除不利案例。

正确处理：

```text
保留旧evaluationId和全部结果
→ 记录标准错误和影响案例
→ 双人重新确认标准并生成新摘要
→ 创建新evaluationId
→ 全量重新运行
→ 报告旧运行作废原因和新旧差异
```

## 8. Git协作规则

### 8.1 分支范围

使用小而明确的短分支，例如：

```text
feature/profile-v2-schema
feature/pandas-diagnostic
feature/path-engine
feature/node-evaluator
feature/react-activity-page
test/evaluation-metrics
fix/evaluator-error-attribution
```

避免在同一分支同时重写Profile、执行器、Agent和网页。跨组功能拆成“生产方合同与实现”“使用方接入”“独立测试”三个可审查提交或合并请求。

### 8.2 合并请求必填内容

```text
当前职责：领域/内容/执行器/Agent/网页/测试
问题或目标：
修改的合同或资产：
生产方和使用方：
影响的schema/revision/protocol/prompt/environment/evaluation版本：
正常测试：
错误与恢复测试：
安全检查：
是否包含受限资产：
回退方式：
需要哪些负责人确认：
```

### 8.3 跨组合并顺序

公共合同变更优先采用：

```text
生产方先加入兼容接口、fixture和测试
→ 使用方适配新接口
→ 端到端和独立验收通过
→ 若确有必要，再删除旧兼容入口
```

任何时候主分支都应保持可运行。不得把半套schema、只有前端没有后端的按钮或无法降级的模型调用合入主分支。

## 9. 兼任时如何保持独立性

4至5人团队可以兼任，但职责边界仍然有效：

1. 同一人写PathEngine和React路径页时，前者是生产方、后者是使用方；另一成员负责独立路径验收；
2. 同一人负责内容和执行器时，参考实现和测试必须由另一成员使用已知错误实现进行反向验证；
3. 同一人负责Agent和架构时，Agent输出仍必须走公开schema和权限门禁，不能直接调用内部仓储；
4. 同一人负责网页和测试时，正式评测期间使用冻结脚本和只读资产，不能边测边修改页面或标准；
5. 核心内容、隐藏测试、正式人工标准和发布门禁不能由同一人制作、使用并独自批准。

最低独立性原则是：

> 核心交付至少有一名非作者成员根据固定fixture和合同进行复核。

## 10. 每日和每周协作节奏

### 10.1 每日短会

每个工作流只汇报：

1. 已完成的可验证产物；
2. 当前阻塞及归属工作流；
3. 公共接口、Profile资产或版本是否变化；
4. 当天准备通过的门禁和需要的验收人。

不使用“完成了大部分”“基本能跑”等不可验证描述。必须指向测试、fixture、交接记录或运行编号。

### 10.2 每周冻结

每周末保存：

- 可运行源码版本；
- 测试结果；
- Profile revision；
- SDK契约版本；
- 环境摘要；
- 已接受和被拒绝的交接记录；
- 未决问题、负责人和最晚处理时间；
- 已执行的降级决定。

第3周结束后，公共类型、评分和接口原则上冻结；第5周结束后禁止新增核心功能；第6周所有正式变更必须创建新的revision、协议版本或evaluationId。

## 11. 团队执行检查表

每次跨组交付逐项确认：

- [ ] 已指定生产方、使用方和独立验收方；
- [ ] 输入、输出、错误、恢复和安全边界已写清；
- [ ] 正常、边界、错误和降级fixture齐全；
- [ ] 使用方只调用公共合同，没有读取内部文件；
- [ ] 自动测试和最小接入通过；
- [ ] 受限资产未进入前端、日志、Agent或普通缓存；
- [ ] schema、revision、协议、提示、环境和评测版本处理正确；
- [ ] 已知限制如实记录；
- [ ] 交接记录保存且状态明确；
- [ ] 后续变更不会覆盖旧资产或正式运行。

团队只需要记住四条原则：

1. 规则错了，回到拥有该规则的工作流修改；
