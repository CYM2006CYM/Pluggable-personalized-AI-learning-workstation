# 03 多智能体与 Pi 集成设计

> **D01/D06 已冻结**：`ModelExecutionPort`只执行一次模型调用，`ReviewOrchestrator`负责 Generator→Hunter→条件Defender→Judge 的严格串行、有限重试、checkpoint和fallback。Agent不能修改答案、分数、Evidence、KnowledgeState、路径或先修关系；Pandas五维评分与 CIDPP 内容评价分离。字段级合同见 [第一周公共合同总册](./第一周任务/21-第一周公共合同总册.md)。

> 版本：MVP-1.0　状态：`[MVP]`。本文件描述 Agent 在现有 Pi/Loop Graph 中的边界，不把规划能力写成当前已实现能力。

## 1. 设计原则：串行协作，不是并行 Agent 网络

系统采用“确定性业务规则 + **串行、隔离、可恢复的 Agent 协作**”。Agent 负责理解、生成、质疑、解释；TypeScript 代码负责身份、schema、来源白名单、先修关系、路径合法性、测试判分、学习状态写入和恢复。六周不实现并行 Agent 网络、消息总线、fork/join、分布式编排或多个 Agent 同时修改一份产物。

这里的“多智能体”表示多个职责明确的角色按固定顺序处理同一个结构化产物，而不是多个模型进程并行群聊：

```text
应用层 Orchestrator 读取 checkpoint
  → 调用 Generator Graph，等待完成并校验 DraftArtifact
  → 调用 Hunter Graph，等待完成并校验 ReviewIssue[]
  → 代码判断是否存在实质争议
  → 有争议才调用 Defender Graph，等待完成
  → 调用 Judge Graph，等待完成并校验 JudgeDecision
  → 代码执行最终硬门禁
  → 持久化 final/fallback，再允许下游活动读取
```

任意时刻一个 `generationRunId` 最多只有一个角色处于 `running`。下一角色只能读取上一角色已经落盘且通过 schema 校验的输出；不得读取上一 Agent 的隐式对话历史，不得提前启动 Judge，不得由前端绕过 Orchestrator 直接调用角色 Graph。

### 1.1 当前基础和新增边界

`[已有]` `createStudyWalkingSkeletonGraphs()` 当前返回 9 个业务 Graph：`generateQuestion`、`gradeAnswer`、`discussQuestion`、`summarizeSession`、`updateLearningProfile`、`buildProfileFragment`、`planProfileRevision`、`reviseProfileDraft`、`reviewProfileDraft`。Controller 通过注入的 `executeGraph(graph, params)` 逐次调用它们。`gradeAnswer` 是语义判题 Graph 能力，不是第七个业务角色；客观题和代码测试仍应优先由确定性代码判定。

`[已有]` `createIsolatedGraphExecutor()` 为每次 Graph 运行创建并最终释放独立的 in-memory AgentSession；执行时使用 `background` 参数、`delegate` 边界、Pi 的取消 signal、空默认工具集，并由 extension 统一配置 `rootMaxSteps=10`、`agentRunTimeoutMs=300000` 和 JSONL trace。该隔离方式正适合串行角色：角色之间只交换 DTO，不共享可变会话。

`[已有]` Pi 命令入口用 `ctx.isIdle()` 阻止宿主 Agent 忙碌时启动另一个学习命令，但这不等于完整的 Web 并发锁，也不等于当前SDK已经支持并行多 Agent。Loop Graph SDK属于本项目可共同修改的组件，本文描述的是当前能力基线，不固定某个提交哈希，也不把当前缺口视为SDK永久边界；新增SDK能力必须经过接口契约、能力探测和应用回归验证后才能写为可用。

`[MVP]` 新增 Diagnosis、Generator、Hunter、Defender、Judge、Learning Decision 的小 Graph，以及应用层 `ReviewOrchestrator`、角色产物仓储和 checkpoint。不得把六个角色塞进一个无法单步恢复的超长 Graph，也不修改现有 9 个 Graph 的语义来冒充新角色。

## 2. 角色与职责

| 角色 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|
| Diagnosis Agent | 固定诊断结果、Evidence、来源摘要 | 诊断解释、候选薄弱点 | 改写原始分数或绕过阈值 |
| Companion Teacher/Generator | Profile 资产、LearnerState、任务模板 | 讲义/题目/提示草稿 | 引入来源白名单外事实、生成隐藏答案 |
| Hunter | 草稿、claim、来源白名单 | `ReviewIssue[]`、引用缺失 | 直接发布或修改草稿 |
| Defender | 草稿、Hunter issues、争议 claim | 逐项支持/反驳意见 | 在没有证据时“替模型补事实” |
| Judge | 草稿、Hunter/Defender 证据、硬规则结果 | `JudgeDecision`：accept/revise/reject | 修改 Rubric、先修、测试或安全规则 |
| Learning Decision Agent | 确定性候选路径、Evidence、约束 | 路径理由/用户可读解释 | 直接决定非法节点、越过先修 |

Defender 仅在 Hunter 发现高风险问题（无来源、矛盾、代码答案泄漏、难度越界）时触发；普通低风险草稿不调用，以控制延迟和 token 成本。

六个角色是职责合同，不要求六个常驻进程。一次用户动作通常只触发其中一部分，例如诊断结果只调用 Diagnosis；路径生成先运行确定性 `PathEngine`，再调用 Learning Decision；普通已审核卡片可以直接从缓存读取。CIDPP内容质量评价由单一评价智能体完成，输出清晰性、完整性、深度、实用性和针对性建议；只有程序硬校验发现高风险事实、来源或泄漏问题时，才运行 Generator→Hunter→条件 Defender→Judge 的事实审核链。

## 3. 统一调用合同

```ts
interface GenerationRequest {
  generationRunId: string;
  subjectId: string;
  profileRevision: number;
  learnerStateRef: string;
  activityId?: string;
  allowedSourceAnchorIds: string[];
  outputSchema: string;
  leakagePolicy: "public-only" | "no-answer";
}

interface Artifact {
  artifactId: string;
  generationRunId: string;
  kind: "card" | "quiz" | "code_prompt" | "feedback";
  payload: unknown;
  claims: Array<{ claimId: string; text: string; sourceAnchorIds: string[] }>;
  status: "draft" | "accepted" | "fallback";
}

interface ReviewIssue {
  issueId: string;
  severity: "low" | "medium" | "high";
  code: "unsupported_claim" | "contradiction" | "difficulty_mismatch" | "leakage" | "schema";
  claimId?: string;
  evidence: string;
}

interface JudgeDecision { status: "accept" | "revise" | "reject"; reasons: string[]; artifactId: string; }
```

每个 Agent 输入必须是 JSON DTO；输出先经过 schema 解析和字段白名单校验，再进入下一步。原始 prompt、模型版本、耗时、token、输入哈希、输出哈希、decision 和错误摘要写入 trace；API key、完整用户答案和隐藏测试不得写入 trace。

所有角色产物还必须携带 `generationRunId`、`profileRevision`、`stage`、`createdAt`、`inputArtifactHash` 和 `promptVersion`。Orchestrator 在进入下一阶段前核对 run/revision/hash；不一致按 `stale_artifact` 失败，禁止把另一轮或另一 Profile 版本的审核意见拼接进当前链。

## 4. 运行链路

### 4.1 诊断链（`[MVP]`）

```text
固定诊断题 → Deterministic Scorer → EvidenceAggregator
         → Diagnosis Agent（解释） → LearnerDiagnostic snapshot
```

固定题的得分、阈值和置信度由代码计算；Agent 只生成可读解释和复习建议。证据不足时状态为 `unverified`，不得伪装成已掌握/薄弱。

### 4.2 资源生成审核链（`[MVP]`）

```text
Path/Activity 选择
 → Generator
 → schema + source + leakage deterministic checks
 → Hunter
 → (高风险才) Defender
 → Judge
 → accepted artifact 或审核后 fallback
```

Judge 接收硬规则结果；硬规则失败时不能通过。reject 使用同 Profile revision 的预生成资产；若无 fallback，返回可恢复错误而非空白页面。

严格串行的状态转换为：

```text
created
 → generating → generated
 → hunting → hunted
 → defending → defended        # 仅 substantiveDispute=true
 → judging → judged
 → approved | fallback | failed
```

只有应用层状态机可以改变 `stage`。每次从 `*ing` 进入完成态时，按“写临时文件→校验→原子替换→更新 checkpoint”的顺序提交。进程在 `hunting` 中断时，恢复流程检查是否已有完整 Hunter 产物：有则从 `hunted` 继续，没有则仅重跑 Hunter，不重跑 Generator，从而保证 baseline 不被覆盖。

### 4.3 学习决策链（`[MVP]`）

`PathEngine` 先由代码根据先修图、目标、知识状态和活动可用性生成合法路径；Learning Decision Agent 只解释“为什么下一步是该节点”，不能增加、删除或重排节点。用户可以在允许范围内选择章节/目标，选择结果仍通过同一个 PathEngine 重新生成路径。

## 5. 与当前 pi-study-helper 分层的适配方案

现有分层无需推翻。多 Agent 能力按下表增量接入：

| 当前层 | 可复用基础 | `[MVP]` 新增 | 明确禁止 |
|---|---|---|---|
| `extension` | 仓储/Graph/trace 的依赖组装，Pi 命令 idle/model 检查 | 组装 `ReviewOrchestrator` 和模型执行端口 | 在命令 handler 内手写角色链 |
| `application` | Controller 注入 `graphs + executeGraph` 的模式 | `ReviewOrchestrator`、短用例、重试/checkpoint/fallback | 让 Agent 自己决定调用下一个 Agent |
| `domain` | Profile、Attempt、SessionEvidence、policy | Artifact、ReviewIssue、JudgeDecision、风险路由纯函数 | 在 domain 中调用模型或 UI |
| `graphs` | code node 准备上下文、agent node + schema/validate | 每角色一个短 Graph 或职责单一的小 Graph | 一个超长六角色 Graph、并行 fork/join |
| `repositories` | Profile family、private memory、原子文件写 | review run/artifact/checkpoint repository | Agent 直接读写文件 |
| `tui/web` | TUI gateway 的显示/输入边界 | 只显示阶段、裁决摘要、fallback 和恢复操作 | 前端直连 SDK、暴露 prompt/隐藏测试 |

### 5.1 推荐代码组织

```text
src/application/review-orchestrator.ts       # 串行状态机和条件路由
src/domain/review-artifacts.ts               # DTO、状态和风险纯函数
src/graphs/study-review-graphs.ts             # 六角色小 Graph 工厂
src/repositories/review-run-repository.ts     # artifact/checkpoint 原子持久化
src/infrastructure/model-execution-port.ts    # Pi/Web 共享的执行边界
```

文件名是实现建议，职责边界为强约束。现有 `StudyWalkingSkeletonGraphs` 保持 v1 行为；新审核 Graph 建议由独立 `createStudyReviewGraphs()` 工厂创建，再由 extension/application 组合，降低对现有测试的影响。

### 5.2 Pi/TUI 调用链

当前 `extension/index.ts` 的 `executorFor(ctx)` 延迟创建与 `ExtensionCommandContext` 绑定的隔离执行器，Controller 再顺序执行 Graph。MVP 可沿用相同依赖注入形状：

```text
Pi command handler
 → controller/use case
 → ReviewOrchestrator.run(request)
 → await executeGraph(generatorGraph, input)
 → validate + persist
 → await executeGraph(hunterGraph, input)
 → ...
 → UI 展示最终状态
```

不能用 `Promise.all()` 同时运行 Hunter、Defender、Judge；Defender 的输入依赖 Hunter，Judge 的输入依赖 Hunter/Defender，任何并行都会破坏证据顺序和恢复语义。

### 5.3 Web 调用链与执行端口修正

当前隔离执行器直接依赖 Pi 的 `ExtensionCommandContext`、`ctx.modelRegistry`、`ctx.signal` 和 `ctx.cwd`，因此 React/本地 Web 不能直接实例化或调用它。MVP 先抽象：

```ts
interface ModelExecutionPort {
  execute(graphId: string, input: unknown, options: {
    runId: string; timeoutMs: number; signal?: AbortSignal;
  }): Promise<GraphStepResult>;
}
```

Pi adapter 用现有 `createIsolatedGraphExecutor(ctx)` 实现该端口；本地 Web 后端通过 OpenAI 兼容模型配置实现同一端口。React 只调用 application facade，不能持有 API Key、Graph 对象或 AgentSession。两种 adapter 必须运行同一 Orchestrator、DTO 校验、风险路由和仓储，避免形成 Web/TUI 两套多 Agent 实现。

### 5.4 与现有学习链的接点

- 出题：保留 `[已有]` `generateQuestion`；v2 动态题先生成 baseline，再按风险进入审核链，通过后才给学习者。
- 判题：保留 `[已有]` `gradeAnswer` 处理语义短答；选择/判断/代码题走确定性判定，不能包装成 Judge Agent。
- 讨论：保留 `[已有]` `discussQuestion` 作为普通来源内问答基础；涉及新事实、来源冲突或答案泄漏风险时转入审核链。
- 总结与画像：保留 `[已有]` `summarizeSession`、`updateLearningProfile`；新 Agent 产物先转换成 Evidence/Artifact 摘要，不能直接改长期画像。
- Profile 构建/修订：已有 build/revision/review Graph 可继续使用；GoA/OAEO 仅作为离线候选，不与实时审核链混合。

应用 Controller 负责解析命令/HTTP 请求、权限和 session CAS，调用 Graph facade；Graph 只负责一次有界模型推理；Repository 负责持久化。Agent 不直接访问文件系统、`ProfileFamilyRepository`、`PrivateMemoryRepository` 或 UI。

现有命令（`[已有]`）：`/study`、`/study-recover`、`/study-profile`、`/study-build`、`/study-revise`。`[六周新增][已冻结]` Web/TUI统一调用 `LearningRuntimeFacade` 的17个短用例，旧 `submitDiagnostic` 等名称仅作兼容别名，具体列表见 [第一周公共合同总册](./第一周任务/21-第一周公共合同总册.md)。

## 6. 串行调度、隔离、重试与恢复

每次 Graph 调用建立独立 AgentSession，设置最大步数、超时、模型和 Profile revision；禁止 AgentSession 共享可变上下文。Orchestrator 必须使用逐步 `await`，每步完成、校验、落盘后才调下一步。失败按类型处理：无效 JSON 只重试当前角色一次并缩小 schema；超时使用静态 fallback；模型拒答记录 `model_error`；来源冲突进入 Defender；Judge 失败保留 baseline 和审核意见并提示稍后重试。每个阶段写 checkpoint（runId、stage、artifactId、attempt、profileRevision），进程重启后从最后成功阶段恢复，不能重复写入 ActivityEvidence。

重试不得回退并重新生成 baseline，除非 Generator 自身未成功。否则单/多 Agent 对照会失去同源性。Judge 要求 revise 时，由代码生成新的 revision artifact，再根据预设策略执行一次有上限的复核；不得形成无界 Agent 自循环。

同一 `generationRunId` 使用应用层互斥锁；不同 run 可以排队处理，但 MVP 不承诺同一 extension 内并发 root execution。Web/TUI 并发写通过 `sessionVersion` CAS；冲突返回 `409/session_conflict`，重新加载后由应用层处理，不由 Agent 猜测或自动合并语义产物。

## 7. 普通与高风险路由

| 路由 | 条件 | Agent 链 |
|---|---|---|
| 直接读取 | 已审核卡片或同 revision 缓存 | 不调用 Agent，读取 accepted artifact |
| 普通问答 | 来源内低风险解释、无代码答案 | Companion Teacher → deterministic checks |
| 普通客观题 | 单选/判断动态题 | Generator → deterministic checks → CIDPP evaluator；高风险才进入 Hunter/Judge |
| 标准 | 新题/反馈或来源 claim | Generator → deterministic checks → CIDPP evaluator → Hunter/Judge（按风险） |
| 高风险 | Hunter 高危、代码任务、敏感内容 | Generator → Hunter → Defender → Judge → 人工/静态 fallback |

GoA/OAEO 只列为资料初始化、批量生成或离线修订的`[候选]`，不进入实时问答/逐题反馈主链。

## 8. 单 Agent 与多 Agent 对照

评测必须在相同 Profile revision、模型、提示版本、预算和输入下比较：单 Agent 直接生成 vs 多 Agent 审核链。记录质量、来源正确率、幻觉率、延迟、token 成本、fallback 比例；不以“角色数量”作为效果证明。MVP 采用 20 组盲评，小样本仅报告趋势。

## 9. 安全与数据边界

来源白名单、答案隔离、输出长度、内容 schema 和命令权限由代码强制。模型输入不包含 API key、其他用户数据或隐藏测试。多 Agent trace 用 hash/摘要替代敏感正文；展示页面只显示阶段、耗时和审核结论，不暴露系统 prompt。

## 10. 测试与验收

- 串行顺序测试：同一 run 的 `startedAt/endedAt` 不重叠，Judge 不得早于 Hunter/Defender 完成；代码中无角色级 `Promise.all`。
- 适配回归：现有 9 个 Graph、五个命令和 v1 Profile 行为保持不变；新增审核 Graph 使用独立工厂和端口。
- 隔离测试：每个角色获得新的 AgentSession，只收到显式 DTO；上一角色隐式消息不出现在下一角色上下文。
- Graph 输入缺失、无效 JSON、超时、错误来源、Judge reject、fallback、checkpoint 恢复。
- Hunter 发现 claim 问题时 Defender 必须触发；低风险路径不应无条件触发 Defender。
- Agent 不能修改 `knowledge_index`、路径先修和隐藏测试。
- 相同输入和缓存键可复现；模型不可用时 Pandas 预生成资产仍能完成端到端演示。
- trace 可关联 `generationRunId → artifact → issues → decision → activityAttempt`。
- 恢复测试：在 Generator、Hunter、Defender、Judge 后分别中断，只重跑未完成阶段；baseline hash 始终不变。
- Web/TUI 一致性测试：两端通过同一 Orchestrator 和仓储得到相同 decision；CAS 冲突不产生两个 final artifact。

验收时必须能用一条 `generationRunId` 展示完整时间线，明确说明“角色按顺序运行、每步使用独立会话、上一阶段完成后下一阶段才开始”。界面可以用步骤条表现，但不得用同时闪烁的多个 Agent 节点暗示并行运行。
