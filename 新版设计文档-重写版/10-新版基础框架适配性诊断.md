# 10 新版基础框架适配性诊断

> **冻结勘正**：Pandas五维画像已由 D01 正式纳入产品上层；它不能替代 `KnowledgeState` 或驱动路径。完整AI代码题仍为候选，P0代码任务固定资料包任务，P0编辑器为 `textarea`。

> 目的：判断现有 `pi-study-helper` 能否承载 Pandas 个性化学习 MVP，并给出按依赖排序的最小改造。结论不是推倒重写，而是保留 walking skeleton，通过 adapter/facade 增量演进。

## 1. 诊断结论

| 能力 | 当前状态 | 结论 |
|---|---|---|
| Pi 命令、TUI、Graph 执行 | `[已有]` | 可直接复用，先做 v1 回归 |
| Profile v1 canonical 包 | `[已有]` | 可承载 Pandas 基础内容；补 v2 扩展字段 |
| draft/active/archived、原子写、恢复 | `[已有]` | 作为资料安全基线；增加 revision/CAS 测试 |
| 章节/卡片学习 | `[已有]` | 作为统一入口之一，不另建系统 |
| 诊断与 KnowledgeState | `[缺口]` | `[MVP]` 新增确定性评分和证据模型 |
| 个性化路径 | `[缺口]` | `[MVP]` 新增 PathEngine；Agent 只解释 |
| 代码活动/评测 | `[缺口]` | `[MVP]` 以 `CodeEvaluationPort` 接入双后端 |
| 多 Agent 审核 | `[部分已有]` | 有隔离 executor；`[MVP]` 增加角色 Graph 和路由 |
| Web | `[缺口]` | `[演示]` React/Vite + facade，轻量本地服务 |
| 教学内容质量 CIDPP | `[六周新增]` | 构建期单一评价 Agent 检查内容质量；不评价学习者、不覆盖事实硬门禁 |
| 学习者五维画像 | `[六周新增][已冻结]` | 诊断后初始快照、学习单元后异步更新；雷达图可降级隐藏但画像数据保留 |

## 2. 现有结构与适配关系

### 2.1 命令到应用层

当前 `extension/index.ts` 直接创建 Repository、Graph 和 Controller，并将 `ctx.ui` 传入 Controller。新增 Web 不应复制这一装配逻辑；应抽取 `createLearningRuntimeDependencies`，Pi 和 HTTP 入口共享，但保留 Pi 的 model/idle 检查在 Pi adapter 中。

### 2.2 Profile 到学习运行时

`ProfileFamilyRepository` 当前能列 active、读取 canonical 文件、构建/修订；`study-profile.ts` 能从 knowledge index 解析 chapter/section/card 并加载材料。此能力适合做 `ProfileReader` 端口。v2 知识点、source anchor、activity 和 dataset 通过新增 adapter 读取，不修改 v1 文件含义。

### 2.3 Session 到 Evidence

当前 `StudySession` 只记录题数、正误、scope 和 mode；`Attempt` 保存题面、答案、判题结果、知识点和建议；`LearningProfile` 是聚合摘要。MVP 需要新增 `Evidence`/`KnowledgeState`，并把旧 Attempt 映射为 evidence source，不能直接把旧 `accuracy` 当作完整 mastery。

### 2.4 Graph 到多 Agent

现有 Graph 是学习主循环，执行器已提供隔离、步数、超时和 trace。新增 Graph 应通过同一 executor，按 run 固定 Profile revision 和输出 schema；不得在 Agent prompt 中放入隐藏测试、API key 或其他用户数据。

## 3. 必要改造方案（按顺序）

### 阶段 A：契约冻结（第 1 周，`[MVP]`）

新增 contracts：ProfileManifestV2、SourceAnchor、KnowledgePointRecord、LearnerDiagnostic、Evidence、KnowledgeState、LearningPath、LearningActivity、ActivityAttempt、CodeEvaluationPort、Facade DTO 和错误码。完成 v1→v2 adapter；运行现有 tests/typecheck 作为门禁。

### 阶段 B：领域与仓储（第 2–3 周）

增加 `PathEngine`（纯函数、可测、无模型依赖）、EvidenceAggregator、活动/尝试仓储和 checkpoint 仓储。Profile revision、sessionVersion、attemptId 全部持久化。路径构建结果保存 snapshot，已完成节点不可被动态重排。

### 阶段 C：Graph/Agent（第 4 周）

增加 Diagnosis、Generator、Hunter、条件 Defender、Judge 和 Decision explanation Graph。每个阶段 DTO schema 验证，失败回退预生成资源；trace 记录角色、耗时、token、缓存和决策。

### 阶段 D：执行器/Web（第 5 周）

实现 Pyodide Worker 预览和 Node 子进程正式评测，通过 `CodeEvaluationPort` 隔离差异。React/Vite 页面只消费 facade；代码活动在 TUI 通过 localhost 深链或待处理提示转入 Web。

### 阶段 E：冻结与评测（第 6 周）

锁定模型/提示/Profile/任务测试版本，运行 60 个 gold、单/多 Agent 盲评、真实学习者探索性测试和恢复/安全矩阵。

## 4. 关键适配接口

```ts
interface ProfileReader {
  loadActive(subjectId: string): Promise<ProfileSnapshot>;
  listKnowledgePoints(subjectId: string, revision: number): Promise<KnowledgePointRecord[]>;
  getActivity(subjectId: string, revision: number, activityId: string): Promise<LearningActivity>;
}

interface PathEngine {
  buildPath(input: { profile: ProfileSnapshot; state: KnowledgeState; goal: Goal }): LearningPath;
}

interface CodeEvaluationPort {
  prepare(activity: LearningActivity): Promise<PreparedActivity>;
  run(input: RunInput): Promise<ActivityResult>;
}
```

所有接口返回领域错误码（`profile_invalid`、`path_infeasible`、`model_timeout`、`evaluator_error`、`session_conflict` 等），UI 再映射为用户提示。

## 5. 适配风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| v1 类型字段不足 | 改动扩散 | adapter + 可选 v2 字段，旧 fixture 保持可读 |
| Controller 绑定 Pi UI | Web 重复逻辑 | 先抽 facade，再写 Web |
| Graph 输出不稳定 | 运行时崩溃 | JSON schema、一次修复重试、静态 fallback |
| 动态路径跳过先修 | 错误个性化 | PathEngine 纯代码合法性检查 |
| 代码执行不安全 | 演示风险 | Pyodide Worker + Node 限时/限输出；声明受信环境 |
| 任务全靠模型生成 | 质量/成本失控 | 预生成 gold、模板、缓存和审核门禁 |
| 外部评测器集成过重 | 延期 | 只经 Port 调研；Node 方案可独立演示 |
| 多 Agent 延迟过高 | UX 失败 | Defender 条件触发、缓存、fallback |

## 6. 明确不适配/延期

不把当前框架改造成 FastAPI/Milvus/Neo4j/消息队列架构；不引入并行 Agent 图；不将 `LearningProfile` 直接改成未经验证的 CIDPP 五维评分；不以雷达图驱动路径；不做生产级沙箱、多用户和云部署。

## 7. 验收用例

1. 旧 `/study demo-review`、`/study-recover`、`/study-profile` 在 v2 adapter 下回归通过。
2. Pandas Profile 可加载知识点、来源和活动，非法引用在启用前被拒绝。
3. 同一诊断输入产生确定性 state/path；Agent 只能提供解释。
4. 生成链在模型超时、Hunter 高危、Judge reject 时均能 fallback 并恢复。
5. `CodeEvaluationPort` 能区分 learner error/evaluator error，重复提交幂等。
6. Web 与 TUI 调用相同 facade，sessionVersion 冲突可提示并恢复。
7. 干净环境按 README 启动后完成 Pandas 端到端演示。

## 8. 开发决策记录

- `[已有]` 能力优先复用，新增功能不修改旧 Profile 语义。
- `[MVP]` 以确定性 domain 为核心，把模型限制在解释和内容草稿。
- `[候选]` Pyodide/Monaco 细节、外部代码评测器、SSE 仅在原型验证成功后采用。
- `[六周新增]` 内容质量 CIDPP 可进入资料包构建和动态题的一次轻量优化；失败时使用未优化但已通过程序硬校验的版本。
- `[候选/调研]` 学习者五维评分、雷达图和 GoA/OAEO 只在原型或离线实验中验证，不阻塞实时主链。
