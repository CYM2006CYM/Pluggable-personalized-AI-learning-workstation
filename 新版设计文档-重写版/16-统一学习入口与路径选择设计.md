# 16 统一学习入口与路径选择设计

> **D05同步**：章节学习、系统推荐、目标学习、快速实战和查漏补缺共用唯一 `LearningRuntimeFacade`；固定17个短用例见 [第一周公共合同总册](./第一周任务/21-第一周公共合同总册.md)。资源化 HTTP 路由每个只调用一个短用例，旧聚合名称仅兼容别名。

> 状态：`[MVP]`。章节学习、系统推荐、目标学习、快速实战和查漏补缺是同一个学习引擎的模式，不是五套产品。

## 1. 统一原则

所有模式共用 `profileId + revision`、`StudySession`、`Evidence`、`KnowledgeState`、`PathEngine`、`LearningActivity` 和 Web/TUI facade。区别只在进入时提供的约束：章节模式指定起点，推荐模式使用诊断，目标模式指定目标，快速实战限制时长，查漏补缺选择低 mastery 节点。

## 2. 开始页选择

开始页先选资料包，再显示五种模式卡片：

| 模式 | 必要输入 | 系统行为 | 典型用户 |
|---|---|---|---|
| 章节学习 | 章节/知识点 | 以先修检查后的节点为起点 | 想按教材学习 |
| 系统推荐 | 诊断或已有画像 | 按 mastery、目标和期限生成路径 | 不知道从哪开始 |
| 目标学习 | 目标知识点/截止时间 | 反向找先修并安排活动 | 为任务补技能 |
| 快速实战 | 时长/难度 | 选一个可独立完成的活动 | 想马上练习 |
| 查漏补缺 | 已有 evidence | 选择低 mastery 且高重要性节点 | 复习薄弱点 |

点击模式后统一调用 `startSession`，生成 session draft；必要诊断未完成时进入诊断，而不是另建“路径大厅”。用户可以查看系统推荐并从合法候选中改选；最终仍由 `PathEngine` 生成 path snapshot。

## 3. 路径选择与约束

路径节点来自 Profile 的知识点 DAG。引擎执行：资料包 allow-list→目标过滤→先修拓扑排序→mastery/难度匹配→活动覆盖→时长预算→可行性校验。任何用户选择都转成约束，不直接写入顺序。无可行路径时返回缺失先修和建议诊断活动。

路径卡片显示节点、预计时长、难度、活动类型、选择理由和来源；“系统推荐”与“我的调整”并列展示差异。确认后保存 `LearningPath{pathId,version,nodes,generatedBy:"deterministic_engine"}`，后续只重排未完成节点。

## 4. 运行中切换模式

用户可在当前 session 选择“继续当前路径”“加入目标节点”“切换为快速实战”。切换不清空历史 evidence；应用层生成新的 path version，并将旧版本标记 superseded。已完成或锁定节点不可被重排，当前活动提交期间禁止切换，避免状态竞争。

## 5. 统一交互与 TUI 兼容

Web 的路径列表和 TUI 的章节菜单都调用 `getNextStep(sessionId)`；两者展示可不同，但节点 ID、活动 ID、证据和版本必须一致。TUI 无法承载代码编辑器时返回 localhost 深链；用户在 Web 提交后，TUI 下次读取同一 checkpoint 并继续。

## 6. 失败、恢复与降级

诊断/路径 Agent 超时不阻塞：使用确定性基础路径和固定理由。Profile revision 失效时要求重新选择 active profile；CAS 冲突时重新加载。时间不足时将复杂图形改为步骤列表，将动态重排改为“完成节点后重新计算”，不得维护第二套路由逻辑。

## 7. 测试与验收

- 五种模式均能创建 session 并落到同一学习工作台。
- 随机知识点 DAG 下所有输出路径满足先修且无重复节点。
- 系统推荐与用户调整使用同一 path engine；切换模式不丢 evidence。
- Web/TUI 对同一 session 的 next step、activity 和 summary 一致。
- 刷新、恢复、profile 版本冲突和 Agent fallback 有自动化测试。
- 演示脚本覆盖“诊断→推荐路径→用户改选→完成活动→查漏补缺”的完整闭环。

## 8. 六周安排

第 2 周整理知识点 DAG 和模式约束；第 3 周实现统一 facade、PathEngine 和路径确认页；第 5 周接入活动执行器；第 6 周做五模式回归和 showcase。MVP 不增加独立推荐服务、图数据库或并行 Agent。
