## 5. 单仓库、无分支协作规则

本周所有人使用同一个GitHub仓库和同一个默认分支，不创建个人分支，不使用PR。为避免覆盖，所有上传必须串行。

### 5.1 术语

- `【拉取】`：从GitHub默认分支取得前一位成员已经上传的最新提交。
- `【上传】`：完成本地提交、取得上传锁、再次确认远端无新提交后，把提交推送到GitHub默认分支。
- `【只读审计】`：拉取最新仓库后运行检查并记录问题，不修改被审计者拥有的文件。
- `【上传锁】`：负责人维护的单一口头或群消息锁。同一时刻只能一人持有；上传完成并通知提交编号后才释放。
- `【无需上传】`：当天该岗位不负责主上传，不上传空提交、临时文件或未脱敏材料。

### 5.2 每次上传的强制顺序

```text
① 在群内申请上传锁
→ ② 确认没有其他人正在上传
→ ③ 保存并提交本人本地修改
→ ④ 拉取GitHub默认分支的最新提交
→ ⑤ 若有冲突，只由对应文件所有者处理
→ ⑥ 运行本次修改要求的最小测试
→ ⑦ 上传到GitHub
→ ⑧ 在群内公布提交编号、修改文件和测试结果
→ ⑨ 释放上传锁
```

如果第④步拉取到了新提交，必须在上传前重新运行最小测试。未经上传锁不得上传；上传失败不得让下一位开始。

### 5.3 每日主上传原则

每天只设置一个主上传岗位；具体负责人任务、时间区间、拉取/上传动作和条件修复已经集中列在第7节。非主上传岗位当天可以阅读、准备和只读审计，但不上传个人记录；记录交给主上传人统一提交。只有第6天审计发现真实阻塞时，相关文件所有者才按负责人指定顺序条件上传修复。

| C在评测文件中复制ActivityResult | Git可能不冲突但语义会分叉 | C拉取并导入A维护的类型 |
| D为了模型输出增加自己的Evidence或路径字段 | Agent会形成第二事实源 | D记录端口缺口，不扩展权威事实 |
| E为了让测试通过修改实现或降低标准 | 审计失去独立性 | E上传失败报告，由所有者按序修复 |
| 任一成员修改19—26号文档 | 任务和合同会出现多个版本 | 提交问题，由负责人统一修改并上传 |

### 9.2 冲突责任

| 冲突类型 | 处理者 | 规则 |
|---|---|---|
| 拉取时出现同一文件文本冲突 | 文件所有者 | 停止上传；由所有者处理并重新测试，其他人等待 |
| 公共字段、接口或错误码冲突 | 负责人决定，A实施 | 负责人更新20/21号，A先上传公共类型，下游依次拉取 |
| Profile资产与Schema冲突 | 负责人决定，A/B按归属修复 | B不扩Schema，A不改专业资产 |
| 评测语义与任务契约冲突 | 负责人裁决，B/C各改自己的文件 | 不在执行器中增加单题旁路逻辑 |
| 测试期望与合同冲突 | 负责人裁决，所有者或E修复 | E不得自行降低标准；合同未变时由实现者修复 |
| 依赖、锁文件或公共配置冲突 | 负责人亲自处理 | 其他成员等待负责人上传后重新拉取 |

## 10. 上传提交格式

每次上传后，成员必须在群内发送：

```text
成员/岗位：
提交编号：
本次修改文件：
依据的21号合同章节：
运行的测试：
测试结果：
是否存在待负责人决定的问题：
下一位允许拉取：是/否
```

提交只处理一个岗位的一类变化，不得夹带其他岗位文件、19—26号负责人文档、依赖锁文件、缓存、密钥、宿主路径或未脱敏数据。

## 11. 第一周统一验收

第一周末必须同时满足：

- 默认分支上没有未解决冲突或未说明的回滚；
- A/B/C/D/E均已上传交接材料，且每次上传可追溯；
- A公共骨架早于B/C/D正式实现；
- B早于C最终任务语义核对，C早于D最终结果核对；
- E的最终审计基于A/B/C/D全部上传后的最新仓库；
- 审计失败由文件所有者修复，E完成复验；
- v1回归、Profile坏包、评测错误、Agent权限和安全泄漏检查有真实记录；
- 未决设计问题已经登记，未被组员自行拍板；
- 负责人完成最终拉取、测试和Go/No-Go。

未通过时只允许继续修复阻塞项，不得带着未决公共合同进入第二周。
