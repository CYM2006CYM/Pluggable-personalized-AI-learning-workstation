测试依据是 21 号公共合同；测试只能验证合同，不能在测试中重新定义产品规则。

## 阅读边界

### 必须阅读

- 19号：只阅读测试职责、交付格式、交叉验收和 Go/No-Go 规则。
- 20号：只阅读D01、D07、D09、D12和D14；这些决定画像、评测、编辑器、隔离和权限边界。
- 21号：完整阅读合同A、合同B和合同E；合同C只阅读 Agent 轨迹裁剪和泄漏边界。
- 04号：阅读页面状态、P0 `textarea`、恢复和错误展示。
- 05号：阅读契约测试、效果评测、安全测试和人工标准隔离。
- 06号：只阅读测试周次、门禁、降级顺序和交付标准。
- 09号：阅读当前测试入口、源码分层和已有命令事实。
- 18号：阅读交接、变更、回归和独立复核规则。

### 不作为本周设计依据

- 02、07号具体内容和数据；E验证格式和隔离，不裁决专业事实。
- 03号 Agent提示词和角色实现；E只验证安全输入、结构和fallback。
- 11号掌握度公式和路径算法；E验证结果，不改算法阈值。
- 12、15号执行器内部实现；E只依据21号结果合同编写测试。

当测试发现设计未定义行为时，不用测试期望“反向设计”产品规则；先标记 `spec_gap`，由负责人决定合同修改或延期。

## 执行边界

### 本周允许执行

- 建立测试命名、fixture命名、临时目录、运行记录和人工评测记录规范。
- 编写安全 DTO、错误码、版本冲突、重复请求、Evidence幂等和恢复测试。
- 编写 Profile 坏包、来源字段、资产隔离、隐藏资产泄漏和 v1 回归扫描。
- 为 C、D 和 A 提供独立验证，不修改被测实现的公共语义。
- 记录真实环境基线和命令失败原因；使用录制模型响应替代实时模型。

### 本周禁止执行

- 不得在测试中新增字段、错误码、阈值、Rubric、路径规则或环境上限。
- 不得为了通过测试修改正式人工标准、扩大测试容忍度或删掉失败案例。
- 不得将隐藏答案、参考实现、密钥、绝对路径、私有代码或隐藏测试放进公开 fixture。
- 不得实现完整 React 页面、高级编辑器或第二套业务规则来绕过合同测试。
- 不得把环境安装失败报告为业务失败，也不得把未运行的测试标记为通过。

### 缺口处理

测试无法判断预期结果时，记录输入、当前输出、缺失条款和建议负责人；在负责人批准前测试状态为 `blocked/spec_gap`，不能自行选择一个结果作为正式标准。

## 任务

1. 恢复依赖并记录 typecheck、test、smoke 和 verify 基线；环境失败必须单独记录。
2. 按 21 号合同建立安全 DTO、版本冲突、重复请求和错误状态测试。
3. 建立测试、fixture、临时目录、运行记录和人工标准命名规范。
4. 建立隐藏答案、参考实现、密钥、宿主绝对路径和私有原始代码的扫描清单。
5. 准备开始、诊断、路径、活动和错误页面的安全数据目录；第一周不实现完整 React 页面。
6. 为 A/C/D 合同编写独立验证记录和周末 Go/No-Go 检查表。

## 交付物

- 基线报告；
- 测试和 fixture 规范；
- 安全 DTO 清单；
- 泄漏扫描清单；
- Go/No-Go 清单；
- `handoff-w1-e-*` 交接记录。

## 不负责

不直接读写仓储，不复制领域规则，不修改评分、错误码、Rubric、路径或环境上限；不以页面完成代替契约测试。

## 验收

A 检查 DTO 不复制领域规则，C 检查代码结果展示语义，D 检查 Agent 轨迹裁剪，B 检查评测模板不让内容作者自我裁决。
