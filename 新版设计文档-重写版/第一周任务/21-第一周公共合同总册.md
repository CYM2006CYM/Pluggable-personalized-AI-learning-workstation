# 合同 A：《公共类型与错误码最终合同》

## A.1 双层事实模型

```text
Evidence（不可变正式证据）
  → mastery/confidence/KnowledgeState（程序计算的知识点事实）
  → capabilityProfileRevision（AI评分的 Pandas 五维画像快照）
```

- `mastery`：知识点掌握度，范围 `0—1`，由确定性程序根据正式 Evidence 计算。
- `confidence`：证据充分度，范围 `0—1`，表示当前掌握判断的可信程度，不等于掌握度。
- `KnowledgeState`：知识点的掌握度、置信度、状态和证据版本组成的确定性事实；它是先修判断和路径规划的唯一权威。
- `capabilityProfileRevision`：Pandas 五维画像的独立版本；只用于能力展示、解释和建议，不得覆盖知识点事实。
- `evidenceVersion`：正式 Evidence 提交事务的单调版本。
- `profileRevision`：资料包修订版本；画像和会话必须记录它。
- `sessionVersion`：会话写入版本，用于并发写保护。
- `modelId`：产生 AI 画像或 Agent 结果的模型标识，不包含密钥。
- `promptVersion`：产生模型结果的提示词版本。

## A.3 五维画像规则

五维固定为：

| ID | 中文含义 | 允许的AI输出 |
|---|---|---|
| `syntax_api` | 基础语法与 API | 分数、置信度、理由、Evidence引用 |
| `data_abstraction` | 数据理解与抽象 | 分数、置信度、理由、Evidence引用 |
| `cleaning_reasoning` | 清洗逻辑与推理 | 分数、置信度、理由、Evidence引用 |
| `validation_debugging` | 结果验证与调试 | 分数、置信度、理由、Evidence引用 |
| `engineering_independence` | 工程规范与独立完成 | 分数、置信度、理由、Evidence引用 |

规则：

1. 诊断完成后生成初始画像快照。
2. 每个学习单元完成后异步批量更新，不阻塞学习页面。
3. AI直接输出可观察维度的 `0—100` 分、`0—1` 置信度、中文理由和 Evidence 引用。
4. 没有直接证据的维度为 `unverified`，不显示0分，也不参与个性化建议。
5. 程序校验分数范围、置信度范围、引用存在性、版本匹配和幂等，不把 AI 分数重新聚合为另一套分数。
6. AI评分失败、超时、引用无效或结构无效时，保留旧画像并标记 `not_updated`，不阻断学习。
7. 同一学习者、`profileRevision`、`evidenceVersion`、`modelId` 和 `promptVersion` 组合必须幂等返回同一画像。
8. 五维画像不得修改 `mastery`、`confidence`、`KnowledgeState`、先修关系、路径顺序、代码分数或 Rubric 结果。
9. CIDPP 五维（清晰性、完整性、深度、实用性、针对性）只评价教学内容，不进入画像。

| 错误码 | 中文含义 | 正式Evidence | 可重试 | 前端动作 |

| `profile_revision_conflict` | 会话与请求修订不一致 | 否 | 否 | 提示恢复旧会话 |

| `idempotency_conflict` | 同一幂等键对应不同请求体 | 否 | 否 | 要求新requestId |

| `prerequisite_violation` | 尝试进入未解锁节点 | 否 | 完成先修后 | 展示缺失先修 |
| `activity_not_found` | 活动或任务不存在 | 否 | 否 | 返回路径错误 |

| `test_failed` | 测试断言失败 | 是学习者错误 | 是 | 显示安全测试反馈 |
| `timeout` | 用户代码超时 | 是学习者错误 | 是 | 显示超时反馈 |
| `output_limit` | 输出超过限制 | 是学习者错误 | 是 | 显示输出限制反馈 |
| `disallowed_import` | 使用禁止导入 | 是学习者错误 | 是 | 显示允许库提示 |
| `submission_contract_error` | 提交结构不符合任务契约 | 是学习者错误 | 是 | 修正提交格式 |
| `evaluator_error` | 评测器或资产错误 | 否 | 是 | 保留草稿，不改变掌握度 |
| `evaluator_start_failed` | 评测器进程无法启动 | 否 | 是 | 转固定降级/重试 |
| `evaluator_timeout` | 评测器自身超时 | 否 | 是 | 记录故障 |
| `dependency_missing` | 锁定环境缺少依赖 | 否 | 否 | 标记环境故障 |

# 合同 B：《统一学习入口最终接口》

## B.1 唯一 Facade

唯一应用入口为 `LearningRuntimeFacade`。Web、HTTP、Pi/TUI 只能调用 Facade，不能直接读写仓储、调用模型端口或访问测试资产。旧文档中的 `StudyAppFacade`、`submitDiagnostic`、`runActivity` 只作为兼容别名，不得创建第二套业务实现。

```ts
type LearningEntryMode = "chapter" | "recommended";

interface CreateRequestMeta {
  requestId: string;               // 创建会话的幂等编号；此时尚无sessionId/sessionVersion
}

interface WriteRequestMeta {
  requestId: string;               // 客户端生成的幂等请求编号
  sessionId: string;               // 会话编号
  sessionVersion: number;          // 客户端已知会话版本
  profileRevision: number;         // 会话绑定的资料包修订
}

interface ReadRequestMeta {
  sessionId: string;               // 要读取的会话
  sessionVersion: number;          // 客户端当前视图版本；只用于检测陈旧视图
  profileRevision: number;         // 会话创建时绑定的修订
}
```

`startSession`只使用`CreateRequestMeta`；服务端读取该`subjectId`当前已启用的active revision，生成`sessionId`，以`sessionVersion=1`创建会话并把实际`profileRevision`返回客户端。客户端不能指定旧revision创建新会话。创建后的写请求统一扩展`WriteRequestMeta`；纯读取扩展`ReadRequestMeta`，不强行创建`requestId`。服务器必须先校验请求、会话身份、版本及活动/路径归属，再执行写入；过期请求不得覆盖新版本。

`goal`、`task_first`和`gap_review`只保留在未来设计记录中，不进入第一周公共枚举、六周HTTP请求或Pandas界面；未来启用必须修订公共合同。

## B.6 草稿、提交和恢复顺序

```text
openActivity 创建服务端 attemptId
→ saveActivityDraft 保存用户草稿
→ prepareActivityRun 签发公开包或生成私有准备对象
→ preview 可重复，不生成正式Attempt/Evidence
→ submitActivity 使用 requestId + attemptId 幂等提交
→ LearningSessionUnitOfWork 映射 ActivityEvidence
→ LearningSessionRepository 正式提交 Evidence、KnowledgeState、路径和 latest.json
```

确定的通过、部分通过、学习者错误只形成一次正式 Attempt/Evidence。修改代码后再次正式提交必须新建 attemptId；同一 attemptId 不得重复计分。`evaluator_error` 保存失败运行和草稿，但不生成负 Evidence。

事务恢复顺序：预分配版本 → 写候选 → 重算知识状态 → 必要时生成路径后缀 → 更新会话候选 → 原子替换 `latest.json` → 写提交标记。中断后从最后完整 Evidence 重算派生数据，隔离不完整候选。

# 合同 C：《模型与串行 Agent 最终合同》

## C.1 ModelExecutionPort

该端口只负责执行一次指定 Graph/Agent 调用，不负责角色顺序、重试、checkpoint、缓存、fallback或业务裁决。

```ts
interface ModelExecutionPort {
  execute(

}

  safeContext: Record<string, unknown>; // 经过裁剪的安全上下文
```

## C.2 ReviewOrchestrator

```text
Generator（生成候选）
→ 结构、Profile白名单和来源硬校验
→ Hunter（独立寻找错误、遗漏边界和泄漏）
→ 风险判断
   ├─ 有实质争议 → Defender（逐项回应）
   └─ 无实质争议 → 跳过 Defender
→ Judge（最终裁决）
→ 最终确定性校验
→ accepted / fallback / failed
```

`ReviewOrchestrator`必须：

1. 严格按阶段串行，上一阶段成功保存 checkpoint 后才进入下一阶段。
2. 为每个阶段检查 `runId`、`profileRevision`、输入摘要哈希和 `promptVersion`。
3. 对无效 JSON、超时、拒绝、来源冲突和版本冲突使用固定错误码。
4. 允许有限重试和有限修订；超出预算直接 fallback，不形成无限循环。
5. 保存可审计的裁剪轨迹、角色状态、耗时、模型和提示版本。
6. 生成结果缓存时绑定资料包修订、输入摘要、模型和提示版本。
7. 失败时使用已通过硬校验的固定内容或固定任务，不阻塞学习路径。

## C.3 角色权限矩阵

| 角色 | 允许输入 | 允许输出 | 禁止输入/输出 |
|---|---|---|---|
| `Generator` | 安全知识点摘要、目标难度、允许来源、固定模板 | 题面/讲解/反馈候选、引用候选 | 不得接收隐藏测试、密钥、完整原始答案 |
| `Hunter` | 候选产物、公开约束、来源摘要 | 缺陷、边界、泄漏和难度问题 | 不得直接改候选或修改分数 |
| `Defender` | Hunter问题、候选引用、硬校验结果 | 逐项回应、支持或反驳 | 不能在无来源时补造事实 |
| `Judge` | 候选、硬门禁、Hunter/Defender产物 | `accepted/revise/rejected` | 不得修改Rubric、测试、路径或答案 |
| `CIDPP` | 教学内容和来源摘要 | 清晰性、完整性、深度、实用性、针对性评分和建议 | 不评价学习者，不进入掌握度 |
| `CapabilityScorer` | 活动声明的可观察维度、正式Evidence安全摘要 | 五维分数、置信度、理由、引用 | 不得修改KnowledgeState、路径或代码分数 |

所有 Agent 均不得修改：客观答案、代码分数、Rubric 阻断结果、正式 Evidence、KnowledgeState、路径、先修关系、隐藏测试和环境锁。

## C.4 安全上下文

允许传入：知识点编号、活动类型、目标难度、脚手架等级、错误类别摘要、来源编号、公开任务契约、已裁剪的 Evidence 摘要和版本信息。

禁止传入：用户身份敏感信息、私人原始答案、完整用户代码（除非活动解释明确需要且经过裁剪）、诊断正确答案、隐藏测试、参考实现、完整 Rubric 阻断规则、API密钥、宿主绝对路径、任意文件内容和未登记来源。

## C.5 录制响应和失败 fixture

测试必须提供：

- 正常 JSON 响应：结构和来源合法；
- 无效结构响应：缺字段、错误类型或越权字段；
- 超时响应：端口返回可重试错误；
- Provider失败响应：固定 fallback；
- 版本冲突响应：不得拼接旧轮次结果；
- Hunter高风险响应：触发 Defender；
- Hunter无争议响应：跳过 Defender；
- Judge拒绝响应：使用固定内容，不发布候选。

自动测试默认使用录制响应或固定适配器，不依赖实时模型。

## C.6 候选 AI 代码题边界

六周 Pandas 代码补全和独立实操固定使用 `profile_fixed`。完整 AI 代码题生成、生成缓存、测试生成和参考实现生成不创建 P0 正式接口；它只能作为候选研究，未来必须通过安全、可复现、可信 Oracle、实际执行、串行审核、缓存一致性和固定兜底门禁。

# 合同 D：《Pandas 数据与来源最终合同》

## D.1 来源登记和裁决

官方 Pandas 文档、教材和公开教程均可进入来源列表和正式讲义，但技术事实有优先级：

1. API、参数、返回行为和版本敏感事实发生冲突时，官方 Pandas 文档裁决。
2. 教材和公开教程可以提供必要摘录、改写示例和初学者解释，不得覆盖官方事实。
3. 每条正式断言必须登记 `sourceId`、来源类型、名称、版本或访问日期、定位、许可证、摘录范围、摘要哈希和适用知识点。
4. 未登记许可证或定位的材料不得进入 active Profile。
5. 第2周前半由内容组提出候选来源，第2周末由负责人批准后才进入 active Profile。
6. 来源更新创建新的 Profile revision；不得覆盖正在使用的旧 revision。

`profile.json`是唯一清单，不得创建第二个manifest。Profile v2删除v1中重复表达目录生命周期的`slot`字段；生命周期只由仓储所在的`active/draft/archived`区域与清单`status`共同校验。

字段和条件资产规则：

1. 根对象必填`subjectId/name/schemaVersion/status/version/revision/revisionOf/capabilities/paths`；`slot`和其他未知核心字段直接拒绝。
2. `paths.subject/chapters/knowledge/goals/sources/quality`是所有v2的基础必填路径；每个路径必须是Profile根内的安全相对路径，拒绝绝对路径、`..`、空段和符号链接穿越。
3. `goals`资产至少包含一个`LearningGoalDefinition`；`goalId`、知识点和活动引用在同一revision内唯一、存在且闭合。章节和推荐入口都必须绑定合法`goalId`。
4. `capabilities.diagnostic=true`时，`paths.diagnostic`和`paths.assessments`必填，诊断题面、私有答案及知识点映射必须存在；为false时不得假装诊断已通过，`startSession`返回`diagnosticRequired=false`。
5. `modalities`含`quiz`时`paths.activities`和`paths.assessments`必填；含`code`或`practice`时`paths.activities/rubrics/datasets/referenceSolutions/environments`全部必填，并且`runtimes`不得为空。
6. `reading`能力至少要求基础`subject/chapters/knowledge/sources`；`cards`是可选增强，不作为v2启用硬门禁。
7. `taskGeneration`只用于构建期资产或候选AI代码题，不是六周运行主链必填路径；存在时仍必须通过路径和引用安全校验。
8. 扩展字段只允许以`x-`开头；扩展值必须是JSON可序列化值，不能覆盖核心字段语义。
9. 所有ID使用稳定英文标识，不能在修订中重用已删除ID；重复ID、类型错误、无效路径、无效引用或声明能力但缺少资产均返回`invalid_profile`。
10. `revision`单调递增；进行中的会话继续使用创建时绑定的旧revision。`active`只读运行资产，`draft`由构建和修订流程写入，`archived`只读。
11. Profile v1按原`status + slot`合同只读兼容；删除`slot`只适用于v2，不能借机修改v1解析器或旧fixture。旧`LearningProfile`或总正确率不得直接转换成知识点级`mastery`。

## D.3 Pandas 七列数据合同

固定字段顺序：

```text
order_id, customer_id, amount, city, order_date, status, note
```

### `order_id`

- 去首尾空格。
- 缺失值删除整行。
- 作为重复记录判定主键。

### `customer_id`

- 去首尾空格。
- 缺失保留缺失，不因缺失删除整行。

### `amount`

- 去除 `¥`、`￥`、逗号和空格。
- 合法值转换为数值。
- 空字符串、非法文本和无穷值转换为缺失并保留行。
- 不擅自四舍五入，不改变原始业务精度规则。

### `city`

- 去首尾空格。
- 使用负责人批准的小型映射表：

```text
上海市 → 上海
沪     → 上海
北京市 → 北京
京     → 北京
广州市 → 广州
穗     → 广州
```

- 未知非空城市保留清洗后的原值。
- 不把未批准的新别名自动加入映射表。

### `order_date`

- 只接受 `YYYY-MM-DD`。
- 其他格式解析失败并转缺失。
- 输出为无时区日期时间；具体 dtype 名称待环境原型后写入环境锁。

### `status`

- 去空格并转小写。
- 只允许 `completed`、`pending`、`cancelled`。
- 其他值转缺失并保留行。

### `note`

- 去首尾空格。
- 缺失转换为空字符串。

## D.4 重复记录和结果合同

重复 `order_id` 选择规则固定为：

```text
可解析日期优先
→ 日期最新优先
→ 日期相同、缺失或都不可解析时保留原始第一条
```

保持被选中记录的原始相对顺序，不额外排序。`clean_df` 只保留固定七列和固定顺序；金额为数值，日期为无时区日期时间，字符串字段使用 Pandas 字符串类型。最终唯一行数、具体 dtype 名称和环境摘要由第2—5周原型确认后写入环境锁。

数据规模：一份约30行公开 CSV；2—3份私有变体，每份约20—30行。内容组制作候选 CSV、参考清洗脚本和测试，负责人批准最终 CSV、映射、异常分布和唯一 `clean_df`。

## D.5 资产目录和权限

```text
profiles/<profile-id>/
├─ profile.json                         # 唯一清单；应用层读取，构建器写draft
├─ sources/                             # 来源元数据；内容组写draft，运行时只读
├─ chapters/                            # 章节、讲义和知识点；内容组写draft，学习运行时读取
├─ activities/                          # 活动题面和公开包；应用层读取
│  └─ <activity-id>/tests/private/      # 学习任务私有测试；Node读取，前端/Agent不可读
├─ datasets/                            # 公开数据和私有变体；执行器按任务读取

└─ quality/                             # 质量门禁和CIDPP摘要；构建阶段写入

evaluation/runtime/hidden/              # 正式效果评测隐藏集；第6周评测器读取
```

隐藏测试、参考实现、完整 Rubric、诊断答案、API密钥、宿主绝对路径和学习者原始答案不得进入前端、普通 Agent 上下文、普通日志或公开 fixture。

# 合同 E：《Python 评测协议最终合同》

## E.1 评测阶段

正式评测阶段固定为：

```text
prepare
→ user_code
→ public_tests
→ hidden_tests
→ summarize
```

阶段不可跳过、不可交换。每次正式运行使用服务端生成的 `runId` 和 `attemptId`，创建唯一临时目录；完成、失败或取消后清理目录。

## E.2 进程、文件和协议通道

1. Node 父评测器持有任务私有测试、隐藏断言、Rubric 阻断规则和环境锁。
2. 用户 `stdout`、`stderr` 与结构化协议通道完全分离。
3. 结构化结果写入 `evaluator-result.json`；不得把用户 stdout 解析为协议。
4. 公开阶段和隐藏阶段使用两个干净 Python 进程，不能复用用户进程状态。
5. 隐藏测试和参考实现不得写入用户工作目录、浏览器 Worker 消息、Agent上下文或普通日志。
6. 使用明确 Python 可执行文件、固定工作目录、最小环境变量和 `shell:false`。
7. 禁止运行时安装依赖；具体允许库和版本由环境锁规定。
8. 超时必须终止进程树；输出必须截断并转义；最终清理临时目录。

## E.3 结果 Envelope

```ts
type EvaluatorStage =
  | "prepare"
  | "user_code"
  | "public_tests"
  | "hidden_tests"
  | "summarize";

interface EvaluatorResponseEnvelope {
  protocolVersion: string;          // 结果协议版本
  runId: string;                    // 一次运行编号
  attemptId: string;                // 正式提交尝试编号
  stage: EvaluatorStage;            // 当前阶段
  status: "ok" | "failed" | "evaluator_error";
  errorKind?: "learner" | "evaluator";
  errorCode?: string;               // 使用合同A错误码
  score?: number;                   // 0—1；确定性计算
  dimensionResults?: Record<string, number>; // Rubric维度分数
  outputSummary: string;            // 裁剪后的安全输出
  environmentHash: string;          // 环境锁摘要，不含宿主路径
  assetBundleHash: string;          // 测试包摘要
  durationMs?: number;              // 服务端测量
}
```

结果文件只允许写入结构化 Envelope；评测器崩溃、结果损坏或协议字段缺失必须返回 `result_protocol_invalid` 或 `runner_crash`，不能被用户 stdout 伪造为成功。

## E.4 错误归因和优先级

错误优先级固定为：

```text
准备/环境错误
→ 协议错误
→ 用户代码错误
→ 测试断言失败
```

| 类别 | 示例 | 是否计学习者负证据 |
|---|---|---:|
| 准备/环境错误 | `environment_mismatch`、`dependency_missing`、`test_asset_invalid` | 否 |
| 协议错误 | `result_protocol_invalid`、`runner_crash` | 否 |
| 用户代码错误 | `syntax_error`、`runtime_error`、`timeout`、`output_limit`、`disallowed_import` | 是，作为学习者错误 |
| 测试断言失败 | `test_failed` | 是，作为学习者错误 |

`evaluator_error` 保存失败运行和草稿，不生成负 Evidence，不降低 `mastery`，不改变路径。

## E.5 DataFrame 比较规则

比较用户结果与参考结果时：

1. 先重置索引，再按业务行顺序比较。
2. 列名、列顺序和 dtype 分别检查；任一不满足即失败。
3. 同一位置的缺失值视为相等。
4. 缺失值出现在不同位置时失败。
5. 数值比较使用固定容差：`atol = 1e-6`、`rtol = 1e-5`。
6. 不允许为了通过测试自动排序、自动改列名或自动改变业务行顺序。
7. 具体 dtype 名称必须与环境锁一致，环境不匹配时返回 `environment_mismatch`，不归因于用户代码。

## E.6 Rubric 和通过条件

```text
score = Σ(dimensionScore × dimensionWeight)
```

正式活动完成必须同时满足：

- 总分达到该任务 Rubric 的通过阈值；
- 所有阻断测试通过；
- 输出契约满足；
- 环境和协议阶段全部成功。

Rubric 权重、阻断测试和通过阈值由 Profile 固定资产提供，执行器不能重定义。Agent只能把确定性结果转换为中文反馈，不能修改分数或阻断结论。

## E.7 CodeEvaluationPort 和预览边界

```ts
interface CodeEvaluationPort {
  prepare(input: PrepareEvaluationInput): Promise<PreparedEvaluation>;
  run(
    input: RunEvaluationInput,
    signal: AbortSignal,
  ): Promise<ActivityResult>;
}
```

`prepare` 校验活动、模式、资料包修订、任务版本和环境锁，并创建适配器私有的 `PreparedEvaluation`；该对象不得序列化给前端。`run` 接收幂等编号、用户代码和已准备任务，返回合同A的 `ActivityResult`。

浏览器 Pyodide Worker 只加载公开执行包、运行用户代码和公开测试、返回裁剪输出，并在取消/超时后销毁 Worker。它不接收隐藏测试、参考实现、完整 Rubric，不生成正式 Attempt/Evidence，不修改掌握度。

Node 正式适配器读取服务端私有资产，分别执行公开和隐藏阶段，生成最终 Envelope，保存 Attempt/Evidence 候选并交给会话事务。外部评测器未来只能替换适配器，不得改变活动、结果和 Evidence 合同。

## E.8 幂等和恢复测试要求

必须测试：

- 同一 `requestId + attemptId` 重复提交只产生一次正式 Attempt/Evidence；
- 修改代码后新 attemptId 才能再次正式提交；
- 每个事务步骤中断后恢复不重复执行或计分；
- 评测器错误只保存运行记录和草稿；
- Windows 进程树终止和临时目录清理有真实记录；
- Node 连续运行同一任务得到一致结论；
- Pyodide 只能预览，Node 是正式权威结果；
- 隐藏测试、参考实现、完整 Rubric、密钥和宿主路径不泄漏。

## E.9 版本和环境锁

`environment-lock.json` 至少记录：Python、Pandas、Pyodide、Node.js 版本；允许库和版本；执行器版本；操作系统和平台；时间、输出、代码和数据大小上限；内存限制是否可靠；数据集、测试包和环境摘要；原型证据编号。所有具体数值只能来自第3—5周原型和压力测试。
