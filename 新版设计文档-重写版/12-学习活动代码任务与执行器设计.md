# 12 学习活动、代码任务与执行器设计

> **D07/D10/D11/D12同步**：六周 Pandas 代码补全和独立实操均为 `profile_fixed`；完整AI代码题仅候选。正式评测遵循 `prepare→user_code→public_tests→hidden_tests→summarize`，Node权威、Pyodide仅公开预览。活动仓储不提交正式 Evidence，唯一提交者是 `LearningSessionRepository`；学习测试和正式评测隐藏集按固定路径物理隔离。

> 文档状态：六周最小可行产品实现基线。  
> 适用对象：负责 Profile 第2版、活动领域模型、任务生成、代码执行、网页、终端、学情证据和测试的开发人员或 AI。  
> 核心决策：Pandas六周代码补全与独立实操均使用资料包预置任务；母题/思维原子约束下的完整AI代码变式只作为候选扩展，不进入六周正式主链。  
> 文档边界：本文规定活动、任务来源、执行、判分、提交和恢复；Profile 资产总规范见02号文档，串行智能体见03号文档，页面见04号文档，效果评测见05号文档，学情证据和版本提交协议见11号文档。

> **六周实施隔离（必须遵守）**：本文后续关于 `ai_generated`、`TaskGenerationService`、AI生成测试、参考实现候选、变异测试和任务缓存的章节属于候选扩展或构建期研究设计。Pandas六周正式排期只实现 `profile_fixed` 固定代码补全、固定最终实操、公开/隐藏测试、确定性 Rubric、Node/Pyodide 原型、提交恢复和固定兜底；不得把候选章节中的接口列为P0，也不得因尚未实现完整 AI 代码题生成而阻塞六周交付。

## 1. 阅读规则、证据纪律与名词

### 1.1 状态标签

- **[已有]**：当前 `pi-study-helper` 源码和测试已经具备。
- **[六周新增]**：六周最小可行产品必须实现。
- **[备选方案A]**：Profile 资料包预置完整任务和测试资产。
- **[备选方案B]**：AI 在 Profile 约束内生成题目、测试和参考实现，再执行和审核。
- **[六周固定]**：Pandas代码补全和独立实操均使用资料包固定任务；完整AI代码题生成降为候选扩展。
- **[待实测冻结]**：必须通过原型或压力测试确定，不能在本文凭空填写。
- **[候选]**：有价值但只有原型通过才接入，例如外部代码评测器和 Monaco 高级能力。
- **[延期]**：六周不实现，例如生产级恶意代码沙箱、云端多租户判题和任意在线装包。

正文使用中文。真实目录、文件名、接口、类型、方法、字段和错误码保留英文，首次出现时给出中文解释。所有示例目录和建议源码均使用英文名称；右侧中文注释必须说明职责、读写者、私有性、恢复能力或当前状态。

### 1.2 重要名词

| 名词 | 中文含义 | 不能误解为 |
|---|---|---|
| `LearningActivity` | 路径中的一项正式学习活动 | Profile 能力模态或学习卡片 |
| `MotherTask` | 维护者录入或选定的母题；用于抽取稳定解题结构 | 要原样塞进学习路径的全部题库 |
| `ThinkingAtom` | 从母题抽取并经维护者确认的最小出题约束 | 自由联想的思维链或模型隐藏推理 |
| `TaskVariant` | 在母题和思维原子约束内形成的审核变式 | AI从零发明的新领域任务 |
| `TaskBundle` | 一道代码题可执行、可判分的完整资产集合 | 只有题面的一段自然语言 |
| `ActivityDraft` | 学习者尚未正式提交的代码草稿 | 正式作答证据 |
| `ActivityRunTrace` | 一次公开“运行”的非计分记录 | 正式提交结果 |
| `ActivityAttempt` | 一次已完成或已确定失败的正式提交 | 旧版 `Attempt` 的同义类型 |
| `ActivityEvidence` | 从正式提交确定性投影出的学情证据 | 模型对代码能力的主观评价 |
| `CodeEvaluationPort` | 屏蔽 Pyodide、Node.js 和外部评测器差异的端口 | 一个具体进程或生产级沙箱 |
| `Rubric` | 确定性评分维度、权重、阻断规则和阈值 | Agent 可以临时改写的提示词 |

## 2. 当前基础框架事实与适配边界

### 2.1 [已有]能力

当前源码已经具备：

- `ReviewQuestion`：表示一道当前复习题，携带题型和题干；选项、参考答案、解析、来源、知识点、难度等多数字段为可选，当前没有独立题库仓储；
- `Attempt`：保存用户提交、订正历史、最终结果、知识点和讨论摘要；
- `StudySession`：保存当前 Pi 终端学习会话及题目统计；
- `SessionEvidence`：由会话和旧作答临时推导观察事实、最终答对证据和未验证主题；
- `ProfileFamilyRepository` 和 `PrivateMemoryRepository`：提供 Profile 家族与本地私有学习记录；
- `study_grade_answer` 智能体工作流图：对当前所有非放弃的旧版答案提交统一进行语义判定；明确放弃由控制器直接保存为 `gave_up`；
- 安全相对路径/根目录约束、单文件临时写后重命名、Pi 终端交互、单次工作流图隔离会话和工作流轨迹。

### 2.2 当前明确没有实现

当前仓库没有：

- `LearningActivity`、`ActivityAttempt` 或 `ActivityEvidence`；
- React 网页、Monaco、Pyodide Web Worker 或浏览器 Python 运行环境；
- Node.js 启动的 Python 正式评测子进程；
- `CodeEvaluationPort`、公开/隐藏测试执行和 Rubric 判分；
- 环境锁、活动草稿恢复、提交幂等、代码任务缓存；
- 容器或操作系统级强隔离沙箱。

所以本文出现的上述类型、接口、目录和流程均为[六周新增]。当前 `createIsolatedGraphExecutor()` 隔离的是智能体 `AgentSession`，不是 Python 代码沙箱，严禁复用其名称宣称已经能安全执行学习者代码。

### 2.3 适配矩阵

| 当前能力 | 新能力 | 复用方式 | 禁止误用 |
|---|---|---|---|
| `ReviewQuestion` | `McqActivity`、`ExplainActivity` | 使用适配器复用安全题面展示和来源字段 | 代码补全/实操不能加入旧 `QuestionType` |
| `Attempt` | `ActivityAttempt` | 终端只读取新版作答安全摘要 | 不强塞部分得分、错误分类和 Rubric 到旧字段 |
| `SessionEvidence` | `ActivityEvidence` | 复用“代码产生事实、Agent不改事实”原则 | 旧投影不能表示测试、提示等级和环境版本 |
| `StudySession` | 新版活动会话 | 通过11号统一 facade 和新仓储关联 | 旧会话没有版本比较、活动和代码草稿 |
| `study_grade_answer` | 文本语义反馈 | 仅用于 `explain` 等开放文本反馈 | 不能覆盖固定答案、测试、超时或阻断失败 |
| Profile 第1版 | Profile 第2版活动资产 | 第1版继续原流程；第2版显式声明能力 | 不能假设第1版有测试、数据、Rubric和环境 |
| 智能体隔离执行器 | 串行任务生成审核 | 复用每次Graph创建/释放独立AgentSession、默认无工具、扩展层步数/超时和结构校验 | 当前没有Generator/Hunter/Defender/Judge角色或跨阶段checkpoint；也不能执行Python |
| `_user/summaries/` | 新版活动私有目录 | 分开保存并由统一应用入口读取 | 不破坏旧批次格式来假装完成迁移 |

当前所有非放弃的旧版答案提交实际都调用 `study_grade_answer`，不能写成客观题当前已经由代码确定性判分。[六周新增] `mcq` 才由服务端按固定答案精确判定。

02号到本文领域模型的适配固定为：02号 `activities[].type` 映射本文 `kind`；02号 `knowledgePointIds` 拆成一个 `primaryKnowledgePointId` 和零到多个 `supportingKnowledgePointIds`；02号 `capabilities.runtimes` 与本文 `environmentRef/runtimePolicyId` 取交集。只有02号唯一 manifest 正式加入活动、任务生成、测试、Rubric、数据、参考实现和环境引用后，这些资产才允许落盘和启用；本文目录不能作为绕开02号校验的第二清单。

### 2.4 与其他文档的唯一边界

- Profile 第2版的真实资产目录和清单字段以02号文档为唯一规范；本文只定义执行器需要的语义。
- 五类正式活动与09、11号文档一致。
- AI 审核严格采用03号的 `Generator → Hunter → 条件 Defender → Judge` 串行链。
- HTTP 路径采用04号文档的复数形式。
- 学情 Evidence、`sessionVersion/evidenceVersion/pathVersion` 与提交标记遵循11号文档。
- 外部评测器仅是15号文档的调研对象，不能写为当前选定实现。

## 3. 五类 LearningActivity

### 3.1 活动枚举

```ts
type ActivityKind =
  | "mcq"               // 客观题；服务端固定答案判定
  | "code_completion"   // 代码补全；完成已有程序的局部缺口
  | "coding_practical"  // 独立实操；按业务契约完成综合任务
  | "explain"           // 解释或迁移说明；以文本反馈为主
  | "debug";            // 调试题；修复给定缺陷并通过测试
```

`reading/quiz/code/practice` 是02号文档中的资料包能力模态，不是 `ActivityKind`。学习卡片和无分自测是教学资源或非正式练习，也不新增活动枚举。

### 3.2 公共类型

```ts
type TaskSource =
  | "profile_fixed" // 资料包预置完整任务
  | "ai_generated"; // 基于母题/思维原子生成并通过审核的变式任务

type ScaffoldLevel =
  | "none"           // 路径层不预设脚手架；活动中可独立作答
  | "hint"           // 路径层允许概念或局部步骤提示
  | "worked_example";// 路径层允许完整的同类范例

type AssistanceLevel =
  | "independent"    // 未查看提示或范例
  | "hinted"         // 查看局部提示
  | "worked_example" // 查看完整同类范例
  | "answer_exposed";// 当前题答案已展示；不生成掌握度证据

interface ActivityBase {
  activityId: string;                  // 同一profileRevision内唯一活动编号
  profileRevision: number;             // 活动所属Profile修订号
  kind: ActivityKind;                  // 正式活动类型
  allowedSources: TaskSource[];        // 模板允许来源；实际来源由TaskAssignment锁定
  primaryKnowledgePointId: string;     // 唯一主要计分知识点
  supportingKnowledgePointIds: string[]; // 辅助知识点；本次不重复计分
  goalIds: string[];                   // 可使用本活动的学习目标
  title: string;                       // 安全标题；不得包含答案
  prompt: string;                      // 学习者可见题面
  difficulty: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 沿用当前五档难度
  estimatedMinutes: number;            // 正整数分钟数；由资产或审核结果给出
  sourceAnchorIds: string[];           // 题面专业事实的来源锚点
  templateVersion: string;             // 固定模板或生成约束版本
  fallbackId?: string;                 // AI模板的固定兜底活动；固定任务可省略
  leakagePolicyId: string;             // 服务端答案泄漏阶段策略编号
  runtimePolicyId?: string;            // 代码活动运行策略；文本活动可省略
  allowedScaffolds: ScaffoldLevel[];   // Profile允许的路径脚手架等级
}
```

六周Pandas版本的 `code_completion` 和 `coding_practical` 均将 `allowedSources` 固定为 `["profile_fixed"]`。完整 `ai_generated` 代码任务只保留为候选扩展，必须另行通过生成、测试、参考实现、审核、缓存和安全门禁后才能启用。`mcq`、`explain` 和 `debug` 也不能借用代码任务来源配置直接开启动态来源。

### 3.3 五类子类型

```ts
interface McqActivity extends ActivityBase {
  kind: "mcq";                         // 客观题
  subtype: "single_choice" | "judgment"; // 六周只支持单选或判断，不开放多选
  options: string[];                    // 安全选项；不含答案标记
  evaluatorRef: string;                 // 服务端固定答案和解析引用
}

interface ExplainActivity extends ActivityBase {
  kind: "explain";                     // 开放解释或迁移说明
  responseContract: string;             // 要求覆盖的内容和格式
  deterministicRubricRef?: string;      // 有可确定规则时才允许正式计分
}

interface CodeActivityBase extends ActivityBase {
  kind: "code_completion" | "coding_practical" | "debug"; // 三类Python代码活动
  starterCode: string;                  // 初始代码；不得含完整答案
  editableRegions: EditableRegion[];    // 允许学习者修改的区域
  entryPoint: string;                   // 学习者可读入口摘要；权威协议在TaskContract
  outputContract: string;               // 学习者可读输出摘要；权威协议在TaskContract
  datasetRefs: string[];                // Profile内数据集引用
  publicTestRefs: string[];             // 学习者可运行的公开测试引用
  hiddenTestRefs: string[];             // 仅正式评测器读取的隐藏测试引用
  rubricRef: string;                    // 服务端Rubric引用
  referenceSolutionRef: string;         // 仅资产校验和评测维护者读取
  knownWrongSolutionRefs: string[];     // 必须被测试拒绝的典型错误实现
  environmentRef: string;               // 锁定运行环境引用
  allowedLibraries: string[];           // 允许导入的库及模块名
}

interface CodeCompletionActivity extends CodeActivityBase {
  kind: "code_completion";              // 局部代码补全
}

interface CodingPracticalActivity extends CodeActivityBase {
  kind: "coding_practical";             // 独立综合实操
  businessAcceptanceCriteria: string[];  // 学习者可见业务验收条件
}

interface DebugActivity extends CodeActivityBase {
  kind: "debug";                        // 修复预置缺陷
  defectCategory: string;                // 安全缺陷类别，不直接给修复答案
}

type LearningActivity =
  | McqActivity
  | ExplainActivity
  | CodeCompletionActivity
  | CodingPracticalActivity
  | DebugActivity;
```

### 3.4 可编辑区域

```ts
interface EditableRegion {
  regionId: string;           // 区域唯一编号
  startMarker: string;        // 起始标记；必须在starterCode中唯一出现
  endMarker: string;          // 结束标记；必须在起始标记之后唯一出现
  required: boolean;          // 是否必须修改或填写
  maxCharacters?: number;     // 可选代码长度限制；来自运行策略
}
```

所有区域必须不重叠且能够重建完整代码。服务端提交时重新检查只读区哈希；浏览器界面限制编辑只是体验能力，不是安全边界。

### 3.5 一次提交只产生一条主要知识点证据

综合任务可能覆盖多个知识点。六周版本规定：

1. 每个活动只有一个 `primaryKnowledgePointId`；
2. 一次正式 `ActivityAttempt` 最多为该主要知识点产生一条掌握度 Evidence；
3. 辅助知识点只保存 Rubric 维度、错误标签和覆盖信息，不自动增加掌握度；
4. 若以后需要一题对多个知识点计分，必须单独设计权重和幂等，本版本不得临时复制证据。

这样可以避免一次综合实操在学情聚合器中被重复计算多次。

## 4. 活动来源策略与第3周决策门

### 4.1 两种方案不是两套执行器

- **[备选方案A] `profile_fixed`**：题面、数据、测试、Rubric 和参考实现随 Profile 交付。
- **[备选方案B] `ai_generated`**：AI 根据学情在 Profile 约束内生成整套候选资产，经过执行和串行审核后才能使用。

两种来源最终都转换成同一个 `TaskBundle`，共享相同执行器、提交、结果、证据和安全协议。不能为 AI 题另建一个缺少隐藏测试或恢复能力的快捷接口。

### 4.2 冻结策略

```ts
interface ActivitySourcePolicy {
  codeCompletion: "profile_fixed" | "ai_generated"; // 代码补全来源；Pandas六周固定为profile_fixed
  codingPractical: "profile_fixed" | "ai_generated";// 独立实操来源；Pandas六周固定为profile_fixed
  fallback: "profile_fixed";            // AI失败时固定回退来源
  decisionStatus: "fixed" | "pending" | "frozen";// Pandas六周为fixed；未来候选扩展再冻结
  frozenAt?: string;                    // 冻结时的ISO 8601时间
  decisionEvidenceRef: string;          // 原型与进度检查记录
}
```

未来扩展时，代码补全和独立实操仍可分别决策，允许：

```text
code_completion = ai_generated       # 代码补全AI原型通过
coding_practical = profile_fixed     # 实操保留固定资产以控制风险
```

Pandas六周正式评测期间，同一活动类型不能随机切换来源；服务端按 `profile_fixed` 运行。未来候选扩展若配置仍为 `pending`，也必须按 `profile_fixed` 运行并记录配置告警。

### 4.3 决策门

未来某类活动只有全部满足以下条件，才可在六周后冻结为 `ai_generated`：

1. 至少一项端到端 AI 原型通过全部硬门禁；
2. 题面、数据、测试、参考实现和 Rubric 能在锁定环境重复运行；
3. Hunter 和 Judge 能发现预置的测试缺陷或答案泄漏样例；
4. 预生成、审核缓存、恢复和固定兜底已经连通；
5. AI 失败不会阻断学习路径；
6. 固定任务私有测试、05号保留评测资产和已物化私有测试不出现在前端、普通日志、Agent 输入和轨迹；Generator候选私有规格按7.7节立即隔离；
7. 第1至3周其他 P0 工作没有因原型延期；
8. 团队仍能在第4至6周完成网页、双后端、正式评测和修复。

任一条件不满足，该活动类型继续使用 `profile_fixed`。这是一条未来扩展的可靠性门禁，不影响Pandas六周固定主链。

## 5. TaskBundle 完整任务资产

### 5.1 类型

```ts
interface TaskContract {
  entryPoint: EntryPointSpec;     // 结构化入口；不能靠自然语言猜调用方式
  inputDescription: string;       // 输入结构、字段和前置条件
  output: OutputSpec;             // 结构化结果提取和比较协议
  outputDescription: string;      // 学习者可见输出语义
  invariants: string[];           // 必须始终成立的可测试不变量
  prohibitedBehaviors: string[];  // 例如写文件、联网或修改输入
}

type EntryPointSpec =
  | { kind: "function"; name: string; argumentFixtureIds: string[] }
  | { kind: "variable"; name: string }
  | { kind: "script"; resultVariable: string };

type OutputSpec =
  | { kind: "json_value"; comparison: "equals" | "approx"; tolerance?: number }
  | { kind: "dataframe"; comparisonRef: string; includeIndex: boolean }
  | { kind: "exception"; allowedTypeNames: string[] };

interface DataFrameComparisonSpec {
  comparisonId: string;                  // Profile白名单比较规则编号
  columnOrder: "exact" | "ignore";       // 是否要求列顺序一致
  requiredColumns: string[];             // 必须存在的列
  dtypeRules: Record<string, string[]>;   // 每列允许的规范dtype名称
  indexPolicy: "reset" | "preserve" | "ignore"; // 索引重置、保留或忽略
```

### 5.2 通用不变量

- 所有 ID 在同一 `profileRevision` 内稳定且唯一；
- 所有文件引用必须在 Profile 或受限评测资产根内；
- 公共 DTO 不序列化 `hiddenTests`、`referenceSolutionRef` 或完整阻断规则；
- Rubric 权重和必须为1，使用未四舍五入值计算；
- `passThreshold`、测试和环境不能由前端或 Agent 在运行时覆盖；
- `assetBundleHash` 变化必须使缓存和旧准备任务失效；
- 同一任务包在同一环境和输入上应得到确定性结果。

`LearningPath.activityIds` 引用稳定活动模板。`openActivity` 首次打开时创建不可变 `TaskAssignment`，把模板锁定到一个具体 `bundleId/assetBundleHash`；刷新、TUI交接和正式提交必须读取同一 assignment，不能重新匹配画像后换题。只有用户明确放弃未提交任务并创建新路径/assignment，旧 assignment 才可标为 `superseded`。

## 6. 备选方案A：Profile 预置任务

### 6.1 资料包必须携带什么

每项代码补全或实操至少提供：

- 题面、学习目标和来源锚点；
- 起始代码和可编辑区域；
- 输入输出契约；
- 合成数据集或确定性数据生成器；
- 公开和隐藏测试；
- 确定性 Rubric；
- 参考实现；
- 至少一个已知错误实现；
- 运行环境要求；
- 固定安全反馈映射；
- 同知识点最小兜底活动。

### 6.2 Profile 启用前门禁

1. 所有引用存在、摘要匹配且路径安全；
2. 起始代码可以解析或按预期暴露待补全区域；
3. 参考实现通过全部公开和隐藏测试；
4. 起始代码不能被误判完成；
5. 每个已知错误实现至少被一个对应测试拒绝；
6. 公开测试能够解释基本契约但不泄漏隐藏边界；
7. 隐藏测试没有超出题面承诺；
8. 数据、测试、参考实现使用同一个环境锁；
9. Rubric 权重、阻断测试和通过阈值完整；
10. 任务难度、脚手架和预计时间与知识点约束相符；
11. 无模型和无网络时固定任务仍然能够正式评测。

固定任务是两种主方案共同的故障兜底。即使代码补全选择 AI 主来源，每个核心知识点也必须至少保留一项审核通过的固定活动。

## 7. 备选方案B：母题与思维原子约束下的 AI 变式任务

### 7.1 设计目的

该方案不是让 AI 在学习页面临时从零创造题目。它把高成本工作前移到 Profile 构建/修订阶段：维护者只需准备少量有代表性的母题，系统从母题中抽取并确认可复用的“思维原子”，再批量生成、执行、审核一组变式任务。学习运行阶段主要做“画像匹配和选择”，必要时才在原子允许范围内生成一个低成本新变式。

这里的“思维原子”是**可外显、可审核的任务设计元数据**，不是要求模型输出隐藏思维链。它描述知识覆盖、难点、解题操作、错误模式和变式边界，不能保存或索取模型内部推理过程。

本机制来自项目方对“母题—思维原子—画像匹配—低成本变式”的后续明确决策，是 `[备选方案B]` 的实现约束，不是当前源码已有能力。它取代14号任务提示词中“AI只改题面或脚手架”的早期概述；14号中的示意缓存数量、修订次数等也不构成运行规范，实际预算必须经第3周原型冻结。

### 7.1.1 “有约束”具体约束在哪里

约束不能只依靠提示词，而要同时存在于四个层级：

| 层级 | 由谁执行 | 约束内容 | 越界时怎么处理 |
|---|---|---|---|
| 选择约束 | 确定性 TypeScript 代码 | 选择母题、必选原子、目标难度、脚手架、可用槽位和生成预算 | 无合法组合时直接使用已接受变式或固定兜底 |
| 输出约束 | JSON Schema 与字段白名单 | AI只能输出规定字段、枚举和引用，不能输出宿主命令、任意路径或任意测试脚本 | 结构校验失败即拒绝，最多允许一次受限修订 |
| 执行约束 | 测试构建器与受限执行器 | 从结构化规格生成测试，运行参考实现、起始代码和错误实现，核对题面契约 | 任一硬门禁失败即拒绝，Agent不能覆盖 |
| 审核约束 | Generator→Hunter→条件Defender→Judge | 查知识越界、难度错配、契约矛盾、测试遗漏和答案泄漏 | Judge只能依据已落盘证据输出 `accept/revise/reject` |

因此，AI不是“根据一段用户画像自由出题”，而是接收已经由程序算好的 `TaskVariantPlan`，在一个封闭的任务空间中完成候选资产。用户画像只能改变允许个性化的部分，不能改变专业正确性和最终通过条件。

### 7.2 MotherTask 与 ThinkingAtom

```ts
interface MotherTask {
  motherTaskId: string;              // 母题稳定编号
  activityKind: "code_completion" | "coding_practical"; // 母题只覆盖两种来源待决代码活动
  primaryKnowledgePointId: string;   // 唯一主要知识点
  supportingKnowledgePointIds: string[]; // 允许组合的辅助知识点
  prompt: string;                    // 经人工确认的母题题面
  promptTemplateId: string;          // 维护者批准的题面模板；AI只能填写声明槽位
  contract: TaskContract;            // 输入、输出和不变量
  starterCode: string;               // 母题起始代码
  starterCodeHash: string;           // 防止生成阶段整体替换起始代码
  editableRegions: EditableRegion[]; // AI参考补丁只能填入这些区域
  allowedApiIds: string[];           // 可在补丁中使用的Profile API白名单
  datasetSchemaRef: string;          // 数据字段规范，不等于具体用户数据
  oracleRef: string;                 // 可信判定规则/维护者参考实现；用于生成预期结果
  rubricRef: string;                 // 固定Rubric骨架
  referenceSolutionRef: string;      // 仅构建验证使用
  sourceAnchorIds: string[];         // 专业事实来源
  authorStatus: "maintainer_accepted"; // 只有维护者确认接受后才能抽取原子
}

type AtomKind =
  | "knowledge_coverage"  // 必须覆盖的知识点和API
  | "reasoning_operation" // 需要完成的可观察解题操作
  | "difficulty_driver"   // 造成难度变化的因素
  | "error_pattern"       // 希望检查的典型错误
  | "scaffold_rule"       // 不同学习者可获得的脚手架
  | "variation_slot"      // 允许改变的情境、数据或代码槽位
  | "invariant";          // 所有变式都必须保持的契约

interface ThinkingAtom {
  atomId: string;                    // 同一Profile修订内唯一编号
  motherTaskId: string;              // 来源母题
  kind: AtomKind;                    // 原子用途
  label: string;                     // 维护者可读说明
  knowledgePointIds: string[];       // 必须来自Profile白名单
  observableRequirement: string;     // 可由测试/Rubric验证的要求
  variationSlot?: VariationSlot;     // 仅variation_slot原子允许；结构化限定可变值
  forbiddenChanges: string[];        // 生成时不得改变的内容
  requiresAtomIds: string[];         // 必须同时出现的原子；程序校验闭包
  conflictsWithAtomIds: string[];    // 不能同时出现的原子；程序拒绝冲突
  minDifficulty?: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 可选最低适用难度
  maxDifficulty?: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 可选最高适用难度
  sourceAnchorIds: string[];         // 原子专业依据
  extractionVersion: string;         // 抽取提示和规则版本
  reviewStatus: "pending" | "accepted" | "rejected"; // 维护者对原子的审核状态，不是Judge决策枚举
}

interface VariationSlot {
  slotId: string;                     // 同一母题内唯一槽位编号
  valueType: "string" | "number" | "boolean"; // 唯一允许值类型
  required: boolean;                  // 生成计划是否必须赋值
  allowedValues?: Array<string | number | boolean>; // 小型离散白名单；元素类型必须一致
  generatorRecipeId?: string;         // 大型取值由Profile确定性生成器提供
  affects: "prompt" | "dataset" | "starter_code"; // 允许变化的位置
}
```

例如 Pandas 缺失值代码补全母题可以产生：

- 知识覆盖原子：必须使用 `isna` 识别缺失；
- 解题操作原子：先识别目标列，再选择填补或删除策略；
- 难度原子：单列固定策略较简单，多列且按业务规则区分策略较难；
- 错误原子：把空字符串误当成已被 `isna` 识别的缺失；
- 变式槽位：列名、缺失比例、允许的填充值和业务情境；
- 不变量原子：输出仍为 DataFrame，关键列类型和行数规则满足题面。

这些内容必须能映射到题面、测试或 Rubric；无法观察和验证的“思维描述”不得作为发布门禁。

### 7.3 原子抽取与人工确认

```text
维护者录入少量MotherTask                    # 不要求准备大量成题
  → AI抽取ThinkingAtom候选                  # 只输出结构化任务设计元数据
  → Profile ID、来源和难度范围校验          # 确定性校验
  → 维护者批准/修改/拒绝每个原子            # 人工确认语义
  → 保存accepted atoms和原子集合版本         # 后续生成只读
```

AI 抽取结果不能自行生效。至少一名内容维护者确认：原子确实来自母题、知识点与来源正确、难度因素合理、可由测试观察。Profile 修订时母题、来源或 Rubric 变化，受影响原子和变式缓存全部失效。

### 7.4 TaskVariantPlan 组合计划

```ts
interface LearnerTaskBand {
  knowledgeStateBand: "unverified" | "support_needed" | "learning" | "ready" | "mastered"; // 11号知识状态档位
  targetDifficulty: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 路径引擎给出的目标难度
  scaffoldLevel: ScaffoldLevel;       // PathEngine已经决定的脚手架
  recentErrorCategoryIds: string[];   // Profile错误类别注册表中的编号；去重、稳定排序并限制数量
  availableMinutes: number;           // 正整数时间预算
}

interface VariantPlanInput {
  profileRevision: number;             // 会话锁定资料包修订号
  primaryKnowledgePointId: string;     // 路径节点唯一主要知识点
  activityKind: "code_completion" | "coding_practical"; // 本次任务类型
  learnerBand: LearnerTaskBand;        // 脱敏低维学情，而非原始作答
  atomSetVersion: string;              // 已接受原子集合版本
  datasetSchemaHash: string;           // 数据字段规范摘要
  profileOrder: string[];              // 母题/原子的稳定声明顺序
  selectionPolicyVersion: string;      // 组合、排序和冲突规则版本
}

interface TaskVariantPlan {
  planId: string;                     // 组合计划编号
  motherTaskId: string;               // 唯一母题骨架
  selectedAtomIds: string[];          // 只能选择维护者已接受的原子
  learnerBand: LearnerTaskBand;       // 低维学情输入，不含私人答案
  variationValues: Record<string, string | number | boolean>; // 以slotId为键；类型和值均经槽位校验
  requiredTestDimensions: string[];   // 必须生成/复用的测试维度
  fixedRubricRef: string;             // 不允许AI更改的Rubric骨架
  datasetRecipeId: string;            // 程序预先选择的Profile数据生成器
  deterministicSeed: number;          // 由规范化计划输入摘要派生，AI不能改
  rowCount: number;                   // 环境锁范围内的数据行数，AI不能改
  difficulty: "S-R" | "S-U" | "M-U" | "M-A" | "C-A"; // 从路径请求复制
  scaffoldLevel: ScaffoldLevel;       // 从路径请求复制，不能由AI提升或降低
  estimatedMinutes: number;           // 时间预算内的正整数，AI只据此控制题量
  generationBudgetId: string;         // 令牌、时间和修订次数配置
}
```

组合规则由确定性代码执行：

1. 选择与路径主要知识点相同的母题；
2. 必选该母题全部 `invariant` 和 `knowledge_coverage` 原子；
3. 根据最近错误选择最多一个相符 `error_pattern`；
4. 根据目标难度选择处于允许范围的 `difficulty_driver`；
5. 根据 PathEngine 脚手架选择对应 `scaffold_rule`；
6. `variation_slot` 只能从 `allowedValues` 或 Profile 数据生成器取值；
7. 不允许组合相互冲突、跨 Profile 或超过时间预算的原子；
8. 组合无解时直接选择已批准固定任务或变式，不让 AI 猜测。

规范化后的同一 `VariantPlanInput` 必须产生同一 `TaskVariantPlan`。AI 只根据计划实现题目，不负责自行选择知识点或判断用户能力。

真正参与确定性计算的输入还必须包含：`profileRevision`、主要知识点、活动类型、目标难度、脚手架、规范化错误类别、原子集合版本、资料包声明顺序、数据 schema 摘要和预算。错误类别先按有限枚举过滤、去重并按固定顺序排列；`planId` 使用完整规范化输入的摘要生成。数据 seed 由 `profileRevision + planId` 摘要确定，不能由 Generator 临时选择。任何输入变化都会形成新计划和新缓存键。

### 7.4.1 哪些内容可以随画像变化

| 内容 | 是否允许变化 | 变化依据 | 约束来源 |
|---|---|---|---|
| 主要知识点、先修关系、最终任务目标 | 否 | 不读取画像改写 | Profile与PathEngine |
| 通过阈值、阻断项、Rubric权重 | 否 | 所有学习者一致 | 固定Rubric骨架 |
| 允许的Python/Pandas API和运行环境 | 否 | 所有学习者一致 | Profile白名单与环境锁 |
| 题目情境、列名和合成数据参数 | 是 | 从已批准槽位取值 | `variation_slot.allowedValues`与数据规则 |
| 难度因素 | 是 | `targetDifficulty` | 已接受的`difficulty_driver`范围 |
| 起始代码留空位置和提示层级 | 是 | `scaffoldLevel` | `EditableRegion`与`scaffold_rule` |
| 典型错误针对性 | 是 | 最近的安全错误类别 | 已接受的`error_pattern`，不发送原始答案 |
| 参考实现和测试的业务结论 | 否 | 必须保持同一专业标准 | `invariant`、任务契约和确定性门禁 |

### 7.4.2 一个完整的受约束生成示例

以下只是说明算法如何工作，不是直接写入 Pandas 资料包的正式题目或阈值：

```text
路径节点：缺失值识别代码补全
安全学情摘要：knowledgeStateBand=learning，recentErrorCategory=empty_string_confusion
路径决定：targetDifficulty=S-U，scaffoldLevel=hint

确定性选择：
  motherTask = missing-value-completion-01
  必选原子 = [识别缺失, 保持输出DataFrame, 保持关键列类型]
  错误原子 = [不要把空字符串等同于isna识别的缺失]
  难度原子 = [单列、固定处理规则]
  槽位值 = { targetColumn: "score", strategy: "fill" }

AI可以生成：
  中文业务情境、题面措辞、该槽位对应的起始代码缺口、结构化测试候选和可编辑区答案候选

AI不可以生成：
  新知识点、环境外API、不同通过阈值、删除阻断项、任意宿主脚本、资料包外数据或完整用户画像

程序随后验证：
  题面是否只要求既定知识点；参考实现是否通过；起始代码是否失败；
  “把空字符串直接当isna缺失”的错误实现是否被拒绝；输出类型、列和行数契约是否保持。
```

如果 `targetColumn="score"` 不在数据 schema、`strategy="fill"` 不在槽位白名单，或任务要求了尚未声明的分组统计，候选必须在调用执行器前被拒绝。不能让 Judge 以“题目看起来合理”为理由放行。

AI 生成的是结构化 `GeneratedTestSpec`，不能自由输出会在宿主机执行的测试 Python 脚本。确定性 `TestBuilder` 只接受白名单 fixture、比较器、不变量和入口，生成实际公开测试和候选私有测试。任何未知比较器、任意表达式、导入、文件路径或命令字段立即拒绝。AI 可以生成候选参考实现，但它属于不受信任代码，只能在与用户代码相同的受限执行边界内运行；该边界仍不是生产级沙箱。候选测试、参考实现和任务在全部门禁通过前没有业务效力。

`candidatePrivateTestSpecs` 只表示未来不下发给学习者的任务私有测试候选，不是05号正式效果评测的 `runtime/hidden/`，也不是人工保留测试集。05号正式隐藏集、固定任务的有效私有测试和任何保留评测答案都不得进入 Generator、Hunter、Defender 或 Judge 上下文。Generator 的候选私有规格由编排器从结构化输出中立即分离，写入受限候选资产；后续角色只接收测试维度、覆盖摘要、执行结果、失败类别和内容摘要，不接收规格正文。它们也不得进入普通 Agent trace。通过最终门禁后，`TestBuilder` 生成的资产才成为该任务包的私有测试；这并不改变正式评测资产与学习任务资产的隔离。

为进一步降低模型输出风险，AI 参考实现不能提交任意完整 Python 文件。它只能用 `referencePatches` 填写 `MotherTask.editableRegions`，不能增加 import、顶层语句、文件/进程/网络调用或环境锁外 API；程序核对 `starterCodeHash`，解析抽象语法树并执行允许节点/调用白名单，再把重建后的完整代码视为不受信任代码运行。静态白名单仍不是强安全，因此第3周安全原型未通过时，`ai_generated` 决策门必须失败并使用固定任务。

AI同时提出补丁和测试候选仍可能出现“相互自洽但专业结论错误”。所以预期结果不能由候选参考补丁自行定义：`expectedValueRef` 和 `invariantId` 必须引用 Profile 的可信注册表，由 `MotherTask.oracleRef` 指向的维护者参考实现或确定性规则生成预期值。候选参考补丁只是被测对象之一。无法由可信 Oracle、白名单不变量或维护者已确认预期值判定的新语义，MVP 一律拒绝，不能只凭 Hunter/Judge 的语言判断发布。

题面也不允许无边界自由写作。MVP 使用 `promptTemplateId + variationValues` 渲染维护者模板；如 Generator提出额外说明，每条专业或业务断言必须通过 `claimRefs` 绑定允许的裁剪来源片段，否则拒绝。学习者反馈同理只保存 `feedbackCodes`，最终中文文案由服务端白名单映射，避免生成文本泄漏答案或编造建议。

结构化测试还要满足组合规则：`dimensionId`、`fixtureRecipeId`、`targetRef`、`expectedValueRef` 和 `invariantId` 都必须在当前 Profile 对应注册表中存在；`equals/approx/dataframe_values` 必须有 `expectedValueRef`，`invariant` 必须有 `invariantId`，`raises` 必须引用契约允许的异常类型，互斥字段不能同时出现。未知引用统一返回 `candidate_test_invalid`，不得退化为字符串拼接执行。

实现时至少提供五个单一职责服务，避免把关键判断埋在提示词里：

```ts
validateMotherTask(input)                  // 校验母题、模板、可编辑区、来源、Oracle和环境引用
buildVariantPlan(input)                    // 纯函数：规范化画像安全摘要并生成唯一计划
validateGeneratedCandidate(input)          // 纯函数：校验结构、槽位、来源、补丁和测试引用
materializeAndVerifyCandidate(input)       // 物化数据/测试/代码并执行正向与反向门禁
acceptVariant(input)                       // 仅在Judge accept后最终复验、计算摘要并形成AcceptedTaskVariant
```

每一步返回有限错误码和不可变输入/输出摘要。只有 `acceptVariant` 产出的 `assetBundleHash` 可以进入 `TaskAssignment`；Generator产物、审核中的临时摘要和复验前的任务包都不能分配给学习者。

### 7.8 前置批量生成与运行时低成本变式

Profile 构建/修订阶段应完成大部分工作：

1. 为 Profile 实际启用的学情档位和实际需要的难度组合生成 `TaskVariantPlan`，不机械生成所有笛卡尔积；
2. AI 生成候选题面、测试和参考实现；
3. 运行第8节全部硬门禁和串行审核；
4. 将通过的任务保存为 `accepted TaskVariant`；
5. 形成“知识状态档位/错误类别/脚手架 → 已批准变式”的索引。

学习运行阶段按以下优先级取题：

```text
完全匹配的accepted变式
  → 按稳定选择键排序后的首个合法accepted变式
  → 在accepted母题和原子内生成低成本新变式
  → profile_fixed固定兜底
```

低成本新变式仍必须经过确定性执行、Hunter 和 Judge；未完成前不阻塞页面，直接展示已批准缓存或固定兜底。不得因“只是变式”跳过测试和审核。

稳定选择键依次为：学情档位精确匹配、难度距离、脚手架匹配、错误类别匹配、`variantPlanHash`、`bundleId`。不使用“最近生成时间”随机换题；选择策略版本写入缓存键，相同输入始终选中同一接受产物。

## 8. AI 任务的确定性验证与严格串行审核

### 8.1 调用链

```text
读取MotherTask/accepted ThinkingAtoms            # 构建阶段已审核的思维骨架
  → 代码生成确定性TaskVariantPlan                # 学情只决定原子选择和槽位
  → Generator生成GeneratedTaskCandidate          # 按计划实现候选，不可扩展边界
  → 结构、路径、来源和白名单校验                  # 确定性硬门禁
  → 在锁定环境运行参考实现                        # 必须通过全部候选测试
  → 运行起始代码、已知错误和自动变异代码          # 检查测试能否拒绝错误
  → Hunter检查题面—测试—参考实现一致性             # 独立核查
  → 高风险争议才运行Defender                      # 条件触发
  → Judge输出accept/reject/revise                   # 与03号串行审核状态一致
  → 最终重新运行并计算assetBundleHash              # 防审核后资产变化
  → 保存accepted缓存或返回profile_fixed fallback  # 学习流程不阻塞
```

每一步完成、校验并落盘后才能开始下一步；禁止角色级 `Promise.all()`。恢复时只重跑未完成阶段，已经验证的 Generator baseline 不被覆盖。

### 8.2 硬门禁

- 参考实现通过全部公开和隐藏测试；
- 起始代码不能直接通过；
- 至少一个已知错误实现被对应测试拒绝；
- 若实现了确定性变异器，至少一种与知识点有关的变异被拒绝；
- 测试不依赖上一次进程的全局状态；
- 数据生成规则使用固定 seed，输出可重复；
- 题面、输入输出契约和断言一致；
- 不导入环境锁之外的库；
- 不引用 Profile 之外的知识点、活动、数据或来源；
- `motherTaskId/variantPlanId/atomIds` 全部存在且已批准；
- 题面、测试和参考实现逐项覆盖计划中的必选原子；
- 没有修改 `forbiddenChanges` 或使用未授权变式槽值；
- 题面、公开测试和安全反馈不泄漏参考实现或隐藏断言；
- 预计时间和难度处于 Profile 允许范围；
- Rubric 维度映射完整且不能改变固定权重/阈值。

任何硬门禁失败，Judge 只能 `reject` 或请求受限 `revise`，不能 `accept`。`accept/revise/reject` 是 Judge 的本轮决策；持久化任务产物使用 `accepted` 表示已经通过完整链路，二者不能混用。

### 8.3 角色职责

| 角色 | 可以做 | 不能做 |
|---|---|---|
| Generator | 按 `TaskVariantPlan` 生成候选题面、数据规则、测试规格和参考实现 | 自行选择知识点/原子、发布、改学情、改环境、改Rubric阈值 |
| Hunter | 根据题面、安全测试维度摘要、执行结果和资产摘要找原子遗漏、契约矛盾、无效测试、难度错配和泄漏 | 获取候选私有测试正文、修改候选或直接发布 |
| Defender | 对高风险意见逐条提供已有公开资产和门禁证据 | 获取候选私有测试正文、无证据时替Generator补事实 |
| Judge | 根据硬门禁状态和裁剪后的审核产物决定 `accept/revise/reject` | 获取候选私有测试正文、覆盖执行失败或隐藏安全规则 |

最多允许一次受限修订。修订后必须从结构校验开始重新执行全部硬门禁。模型失败、超时、无效结构、超过修订预算或最终拒绝时立即返回固定兜底任务。

### 8.4 反向测试边界

“自动变异代码”不是要求六周实现通用变异测试平台。MVP 可为 Pandas 核心任务提供有限、确定性的变异器，例如删除 `drop_duplicates`、使用错误业务键、漏掉 `errors="coerce"` 或不执行最终断言。若某类任务没有可信变异器，只要求已知错误实现门禁，并在质量报告中记录限制，不能编造“变异覆盖率”。

## 9. 诊断后预生成和缓存

### 9.1 触发时机

只有未来扩展明确冻结为 `ai_generated` 才触发代码任务的运行时匹配；Pandas六周固定主链不执行此流程。大部分候选变式应在 Profile 构建/修订阶段生成并审核。运行时流程为：

1. 诊断完成并确认路径；
2. 应用层选取路径近期可能到达的代码节点；
3. 代码根据学情选择母题、accepted原子和变式计划；
4. 优先读取构建阶段已经批准的匹配变式；
5. 没有合适变式且生成预算允许时，后台生成低成本新变式并完整审核；
6. 到达节点时优先读取已批准缓存；
7. 缓存未完成、损坏或失败时直接返回固定兜底，不让页面无限等待。

这里的“后台”表示本地 Node.js 应用层中的可恢复任务，不表示消息队列、并行 Agent 网络或云任务系统。

### 9.2 缓存键

```text
cacheKey = hash(
  subjectId, profileRevision, knowledgePointId, activityKind,
  difficulty, scaffoldLevel, knowledgeStateBand, errorCategory,
  motherTaskId, thinkingAtomSetVersion, variantPlanHash,
  taskConstraintVersion, environmentHash, datasetSchemaHash,
  promptVersion, modelId
)
```

- 使用真实身份 `subjectId`，不用模糊的 `profileId`；
- 不把姓名、完整答案或其他私人原文放入键；
- 同一键必须命中同一批准任务；
- Profile、模板、约束、环境、数据 schema、提示或模型变化都会形成新键；
- 缓存失效不删除固定 fallback；
- 正式评测记录实际命中的任务来源和缓存键摘要。

### 9.3 缓存内容

缓存保存：接受后的 `TaskBundle`、母题编号、思维原子集合、变式计划、来源锚点、测试包摘要、参考实现验证摘要、`generationRunId`、各审核阶段编号、生成耗时、token、重试和环境摘要。参考实现和任务私有测试可以保存于服务端受限缓存区，但不能进入前端缓存、普通日志、Agent trace或公开 `TaskBundleView`；05号正式隐藏评测资产不得进入该缓存。

## 10. 各活动的执行与判分责任

| 活动 | 是否进入Python执行器 | 正式判定者 | 是否生成正式学情证据 |
|---|---|---|---|
| `mcq` | 否 | 服务端固定答案比较 | 确定提交后生成客观题证据 |
| `code_completion` | 是 | 测试和确定性Rubric | 生成代码提交证据 |
| `coding_practical` | 是 | 阻断测试、维度Rubric和总阈值 | 生成实操证据 |
| `explain` | 否 | 默认由Graph给语义反馈；有确定Rubric才正式计分 | 默认只写软证据 |
| `debug` | 是 | 测试验证缺陷是否修复 | 生成代码提交证据 |

模型不能覆盖固定答案、语法错误、运行错误、超时、测试结果、阻断失败、Rubric权重或通过阈值。Agent 可以解释安全错误类别和开放文本反馈，但不能修改 `score`。

## 11. 环境锁与资源限制

### 11.1 不把本机观察写成发布事实

当前开发机能够观察到 Python 3.13.14 和 Pandas 2.3.3，当前仓库未安装 Pyodide。这些只说明本机状态，不能作为产品环境决定。Pyodide、Python、Pandas、Node.js 的发布版本和所有资源阈值必须通过第3至5周原型后冻结。

### 11.2 EnvironmentLock

```ts
interface EnvironmentLock {
  environmentId: string;           // 活动引用的稳定环境编号
  schemaVersion: 1;                // 环境锁格式版本
  nodeVersion: string;             // 本地服务Node.js版本
  pythonVersion: string;           // 正式提交Python版本
  pandasVersion: string;           // 锁定Pandas版本
  pyodideVersion?: string;         // 开启浏览器预览时必须填写
  allowedLibraries: Array<{        // 预装白名单；禁止运行时任意安装
    name: string;                  // Python导入名
    version: string;               // 已验证版本
  }>;
  platform: string;                // 原型验证的平台，例如Windows架构
  evaluatorVersion: string;        // 评测协议和实现版本
  limits: {                         // 全部阈值通过原型后冻结
    wallClockMs: number;           // 墙钟超时；正整数，来自实测
    stdoutBytes: number;           // 标准输出上限；正整数
    stderrBytes: number;           // 标准错误上限；正整数
    sourceBytes: number;           // 用户代码大小上限；正整数
    datasetBytes: number;          // 单任务数据上限；正整数
    memoryBytes?: number;          // 仅后端已证明可靠限制时填写
  };
  capabilityFlags: {                // 只记录故障测试已证明的能力
    reliableMemoryLimit: boolean;  // 是否真正具备可靠内存限制
    networkIsolation: boolean;     // 是否经测试证明禁止网络
    processTreeTermination: boolean; // 是否经测试能终止整个进程树
  };
  prototypeEvidenceRef: string;    // 版本/阈值的原型证据记录
  environmentHash: string;         // 规范化锁文件摘要；计算时排除本字段
  createdAt: string;               // 冻结时间
}
```

所有数字必须为正数且来自原型、故障注入或压力测试。没有可靠内存隔离时 `memoryBytes` 省略、`reliableMemoryLimit=false`；不能只因 Node.js 有子进程接口就宣称支持强内存限制。

所有摘要使用同一规范：UTF-8、对象键字典序、数组保持业务顺序、无多余空白；`environmentHash` 计算时排除自身，`assetBundleHash` 排除自身并覆盖活动、契约、数据摘要、测试摘要、Rubric和环境引用。摘要算法和版本写入 manifest，防止不同序列化方式产生假冲突。

### 11.3 版本冻结流程

1. 第3周建立最小 Node.js/Python 子进程原型；
2. 建立 Pyodide Worker 预览原型；
3. 在相同代码、数据和10组公开输入上比较关键结果；
4. 选择两端共同支持的 Pandas 行为子集；
5. 压测时间、输出、代码和数据大小；
6. 写入 `environment-lock.json` 和原型证据；
7. 正式任务只使用锁文件声明能力；
8. 运行环境不匹配返回 `environment_mismatch`，不静默换版本。

## 12. 服务端 CodeEvaluationPort 与浏览器运行接口

### 12.1 端口定义

```ts
type EvaluationMode = "preview" | "submit";

interface PrepareEvaluationInput {
  activity: CodeCompletionActivity | CodingPracticalActivity | DebugActivity; // 待准备代码活动
  profileRevision: number;        // 会话锁定的Profile修订号
  mode: EvaluationMode;           // 公开预览或正式提交
  environment: EnvironmentLock;   // 已校验环境锁
  assetBundleHash: string;        // 防止准备后资产被替换
}

interface PreparedEvaluation {
  preparedId: string;             // 一次准备结果编号
  mode: EvaluationMode;           // 运行模式
  activityId: string;             // 已锁定活动编号
  profileRevision: number;        // 已锁定修订号
  environmentHash: string;        // 已锁定环境摘要
  assetBundleHash: string;        // 已锁定任务包摘要
  expiresAt: string;              // 防止陈旧准备结果长期复用
}

interface RunEvaluationInput {
  requestId: string;              // 幂等请求编号
  executionId: string;            // preview使用runId，submit使用attemptId
  prepared: PreparedEvaluation;   // 适配器准备结果
  code: string;                   // 学习者代码；大小受环境锁限制
}

interface CodeEvaluationPort {
  prepare(input: PrepareEvaluationInput): Promise<PreparedEvaluation>; // 服务端校验并准备公开或私有资产
  run(
    input: RunEvaluationInput,
    signal: AbortSignal,          // 用户取消和超时中断信号
  ): Promise<ActivityResult>;
}

interface BrowserCodeRunner {
  run(
    bundle: PublicExecutionBundle, // 服务端投影的公开执行包；不含私有测试和参考实现
    code: string,                  // 当前浏览器内的学习者代码
    signal: AbortSignal,           // 取消或超时时销毁本轮Worker
  ): Promise<ActivityResult>;
}
```

### 12.2 私有准备状态

真实适配器通常还需要测试引用、临时目录策略或内部句柄。这些私有数据必须保存在服务端适配器内部，以 `preparedId` 索引；公共 `PreparedEvaluation` 不包含隐藏测试路径。正式提交只在服务端根据 `preparedId` 取回私有状态。

`preparedId` 不是唯一事实源。服务重启或内存状态丢失后，服务端必须用 `activityId + profileRevision + environmentHash + assetBundleHash` 从只读 Profile/批准缓存重新构建相同准备状态；资产摘要不一致则返回 `environment_mismatch` 或重新 `prepare`，不能继续陈旧提交。

浏览器不能直接获得服务端 `PreparedEvaluation`，也不能远程调用持有隐藏资产的端口对象。预览模式由服务端执行 `prepare()` 校验后，再通过确定性投影生成13.2节的 `PublicExecutionBundle`；浏览器的 `BrowserCodeRunner` 只消费该公开包并启动 Worker。`CodeEvaluationPort.run()` 在六周版本用于服务端权威提交；若未来增加服务端预览才使用其 `preview` 模式。两端共享 `ActivityResult` 结果结构，但不共享私有准备对象，Pyodide也不冒充服务端端口实现。

### 12.3 三种适配形态

- Pyodide 预览：服务端准备并投影公开包，浏览器 `BrowserCodeRunner` 在 Worker 中运行；
- Node.js/Python 端口实现：本地服务持有私有测试并执行权威正式提交；
- 外部评测器适配器：[候选]，只有15号调研门禁通过才替换 Node.js 底层。

外部评测器只替换端口实现，不能改变活动、HTTP、作答、证据和页面契约。10、15号文档中较早的签名视为概念草案，实现以本节为唯一规范。

## 13. Pyodide 公开运行

### 13.1 职责

浏览器专用 Web Worker 用于即时“运行”：

1. 服务端校验会话、活动、`profileRevision` 和公开资产；
2. 返回 `PublicExecutionBundle`；
3. React 把公开包和当前代码交给 Worker；
4. Worker 加载锁定 Pyodide、公开数据副本和公开测试；
5. 执行代码并返回标准化安全结果；
6. 浏览器可在内存或 `sessionStorage` 保存 `ActivityRunTrace`，但服务端不保存预览结果，也不创建正式 `ActivityAttempt` 或 Evidence。

### 13.2 PublicExecutionBundle

```ts
interface PublicExecutionBundle {
  runId: string;                   // 服务端签发的公开运行编号
  sessionId: string;               // 所属会话
  activityId: string;              // 当前活动
  profileRevision: number;         // 锁定修订号
  environmentId: string;           // 公开环境编号
  starterCodeHash: string;         // 起始代码摘要
  publicDatasetFiles: Array<{      // 可下发的合成数据
    name: string;                  // Worker内相对文件名
    content: string;               // 数据内容；受大小限制
    hash: string;                  // 完整性摘要
  }>;
  publicTestSources: string[];      // 公开测试源码；不得包含隐藏断言
  expiresAt: string;               // 公开包有效期
  bundleHash: string;              // 公开包摘要
}
```

04号的 `POST /api/activities/:id/run` 在六周实现中只负责校验和签发公开运行包/`runId`；实际 Python 代码在浏览器 Worker 中执行。Worker 结果不回传服务端，因此服务端不能声称持有预览结果。页面刷新可以恢复代码草稿，但不恢复已经消失的预览输出。若以后确需持久化预览，必须新增带 `runId + bundleHash` 校验的独立回传协议；不复用正式提交接口。若后续改为服务端预览，也必须保持“不含隐藏测试、不计掌握度”的语义。

### 13.3 生命周期与失败

- 第一次打开代码页时按需加载，不让开始页承担大型运行时资源；
- 每次运行使用干净 Python 命名空间；若无法可靠清理则重建 Worker；
- 用户取消、墙钟超时或输出超限后终止 Worker，并在下一次运行前重建；
- Python stdout/stderr 钩子和 Worker 消息层按字节流式计数，达到上限立即停止转发并销毁 Worker，不能先聚合无限输出再截断；
- Pyodide 初始化或公开资产损坏返回预览不可用，不产生学习者错误；
- 代码草稿保留，学习者仍可正式提交到 Node.js；
- Worker 不接收隐藏测试、参考实现、完整 Rubric、密钥或宿主绝对路径。
- Worker 销毁后忽略携带旧 `runId` 的迟到消息；不同运行不能共享 Python 全局对象。

Web Worker 隔离 DOM，但 Pyodide 可能通过 JavaScript 桥接访问 Worker 全局能力和网络接口，不能把“放入 Worker”当成安全沙箱。公开 Worker 不持有可调用提交/隐藏接口的长期凭据；本地服务对每次请求重新校验会话和活动权限；页面采用内容安全策略和 Worker 资源白名单。禁用 `import js`、`fetch`、本地接口探测和其他 JavaScript 桥接属于[待实测冻结]能力，只有原型确实阻断后才能在环境锁声明。否则仍按受信任测试者边界运行。

公开运行不更新 `mastery`，也不触发11号路径重算。

## 14. Node.js/Python 正式提交

### 14.1 调用链

```text
POST /api/activities/:id/submit                   # 正式提交入口
  → 校验requestId/attemptId/sessionVersion        # 幂等和并发
  → 校验profileRevision/activity/assetBundleHash  # 锁定任务资产
  → 服务端读取环境、数据、测试和Rubric             # 不信任客户端资产
  → 创建唯一临时任务根                             # 仅本次执行
  → 校验代码大小、可编辑区域和导入策略              # 运行前门禁
  → 干净进程执行公开阶段                            # 用户可见反馈基础
  → 新干净进程执行隐藏阶段                          # 父评测器保留隐藏资产
  → 确定性汇总Rubric和阻断测试                      # Agent不可覆盖
  → 原子保存Attempt/Evidence/版本候选               # 遵循11号协议
  → 最后提交会话标记                                # 一次公开新版本
  → 终止进程并清理临时目录                          # 失败也清理
```

### 14.2 子进程启动规则

- 使用显式 Python 可执行文件和参数，`shell: false`；
- 工作目录固定为本次唯一临时目录；
- 环境变量采用最小白名单，不继承接口密钥和无关宿主变量；
- 禁止运行时 `pip install`、`micropip.install` 或任意在线安装；
- 用户代码、数据和执行协议按内容写入，不拼接到 shell 命令；
- 墙钟超时后尝试终止整个进程树；Windows 必须使用经原型验证的 Job Object 或等价受控机制，不能把普通 `child.kill()` 当成已终止孙进程；能力不可靠时环境锁标记为 false、检测并记录残留；
- 标准输出和错误输出按字节上限截断并转义；
- 每次正式提交使用全新数据副本和进程；
- 公开与隐藏阶段不共享用户全局状态；
- 两阶段使用相同固定随机 seed、固定时区/locale和输入资产，禁止依赖当前时间、系统随机源和未声明外部状态；
- 临时目录只保存执行所需内容，不保存应用密钥和其他会话数据。

导入白名单通过语法树检查、运行环境只预装允许库和测试用例共同约束，但它不能抵御刻意使用动态导入、标准库副作用或系统调用的恶意代码。若环境锁没有证明网络隔离，不能把“禁止网络”写成已经由本地子进程强制保证；本项目依赖受信任测试者边界。

### 14.3 隐藏测试边界

理想边界是父评测器持有隐藏断言和预期结果，只把用户程序的必要输出交给测试判定，并向学习者返回维度和错误类别。六周 Node.js/Python 方案在受信任本地环境中实现这一逻辑隔离，但不宣称能够抵御学习者主动探测同机进程或文件系统。

隐藏阶段至少做到：

- 隐藏测试不进入 React、浏览器网络响应、Agent 输入或普通日志；
- 不把隐藏源码复制进用户可编辑工作目录；
- 用户反馈不包含隐藏输入、精确预期值、断言文本和参考实现；
- 临时文件和错误栈中的宿主路径在返回前裁剪；
- 发布说明明确这是本地受信任测试边界，不是生产级恶意代码沙箱。

### 14.4 安全声明

Windows 本地子进程的工作目录、环境变量、超时和输出限制属于风险降低措施，不等于操作系统级安全隔离。网络隔离、可靠内存限制和进程树终止只有在原型与故障测试通过后才能在 `capabilityFlags` 标为 `true`。容器、低权限账户、系统调用过滤和远程判题属于[延期]或外部评测器候选。

## 15. ActivityResult、错误归因与 Rubric

### 15.1 结果类型

```ts
type ExecutionStatus =
  | "completed" // 评测流程完整完成
  | "cancelled" // 用户取消预览；正式提交通常不允许前端撤销已提交事实
  | "failed";   // 评测基础设施没有完成协议

type Verdict =
  | "passed"  // 达到阈值且阻断项通过
  | "partial" // 有有效得分但未通过
  | "failed"  // 确定性学习者结果未通过
  | "none";   // 评测器错误，不能归因学习者

type ErrorKind = "learner_error" | "evaluator_error" | null;

type EvaluationStage =
  | "prepare"        // 环境和资产准备
  | "user_code"      // 用户代码加载/执行
  | "public_tests"   // 公开测试
  | "hidden_tests"   // 隐藏测试
  | "rubric"         // Rubric汇总
  | "commit";        // 持久化提交

type LearnerErrorCode =
  | "syntax_error"              // 用户代码语法错误
  | "runtime_error"             // 用户代码运行异常
  | "test_failed"               // 一个或多个确定性测试失败
  | "timeout"                   // 健康执行器中用户代码超过任务时限
  | "output_limit"              // 用户输出超过锁定上限
  | "disallowed_import"         // 导入环境白名单外库
  | "submission_contract_error" // 缺少入口、结果变量或输出结构错误
  | "memory_limit";             // 仅可靠归因时使用

type EvaluatorErrorCode =
  | "evaluator_start_failed"   // 无法启动评测进程
  | "evaluator_timeout"        // 评测基础设施没有按协议结束
  | "dependency_missing"       // 锁定依赖不可用
  | "test_asset_invalid"       // 测试、数据或Rubric损坏
  | "result_protocol_invalid"  // 子进程返回结果不符合协议
  | "runner_crash"             // 无法可靠归因的进程崩溃
  | "environment_mismatch";    // 实际环境与锁文件不一致

interface DimensionResult {
  dimensionId: string;           // Rubric维度编号
  score: number;                 // 0..1确定性得分
  passed: boolean;               // 本维度是否达到要求
  blocking: boolean;             // 是否为阻断维度
  feedbackCodes: string[];       // 安全反馈类别，不含隐藏断言
}

interface ActivityResult {
  stage: EvaluationStage;              // 结果或错误发生阶段
  executionStatus: ExecutionStatus; // 评测流程状态
  verdict: Verdict;                  // 学习者判定
  errorKind: ErrorKind;              // 学习者错误、评测器错误或无错误
  errorCode?: LearnerErrorCode | EvaluatorErrorCode; // 具体错误；无错误时省略
  score?: number;                    // 0..1；evaluator_error时省略
  dimensionResults: DimensionResult[]; // 维度结果；异常时可为空
  safeFeedback: string[];            // 裁剪后的中文反馈
  stdoutExcerpt?: string;            // 截断并转义的标准输出
  stderrExcerpt?: string;            // 截断并裁剪路径的标准错误
  durationMs: number;                // 非负运行毫秒数
  evaluatorVersion: string;          // 实际评测器版本
  environmentHash: string;           // 实际环境摘要
  assetBundleHash: string;           // 实际任务资产摘要
}
```

合法组合必须通过结构校验：

| 场景 | `executionStatus` | `verdict` | `errorKind` | `score` |
|---|---|---|---|---|
| 正式通过 | `completed` | `passed` | `null` | 必填，达到阈值 |
| 部分正确 | `completed` | `partial` | `learner_error` | 必填，范围0..1 |
| 确定未通过 | `completed` | `failed` | `learner_error` | 必填，可为0 |
| 评测器异常 | `failed` | `none` | `evaluator_error` | 必须省略 |
| 用户取消预览 | `cancelled` | `none` | `null` | 必须省略 |

`cancelled` 只允许用于 preview。`errorKind=null` 时不得携带错误码；学习者/评测器错误码必须与对应 `errorKind` 匹配。

### 15.2 错误归因规则

- 用户代码在健康执行器中超过任务时限：`learner_error/timeout`；
- 评测器启动、通信或内部流程超时：`evaluator_error/evaluator_timeout`；
- 环境缺少锁定库：`evaluator_error/dependency_missing`；
- 测试资产本身抛错：`evaluator_error/test_asset_invalid`；
- 进程崩溃且无法证明由用户内存使用导致：`evaluator_error/runner_crash`；
- 只有可靠内存限制和归因通过原型后，才返回 `learner_error/memory_limit`；
- 临时目录清理失败记录安全告警，不应覆盖已经可靠得到的判分，但不能返回敏感路径。

归因按阶段优先：准备阶段失败先判评测器异常；用户代码语法/运行失败判学习者错误且不启动测试；用户代码成功后，测试 harness 自身协议或资产失败判评测器异常；只有 harness 健康且断言未满足才判 `test_failed`。父进程只解析 `EvaluatorResponseEnvelope`，用户 stdout 永远走独立通道，不能伪造 JSON 协议结果。

### 15.3 Rubric 公式

```text
score = Σ(dimensionScore × dimensionWeight)
```

正式 `passed` 必须同时满足：

1. 每个维度得分均来自确定性测试或静态检查；
2. 总分不低于 `passThreshold`；
3. 所有 `blocking=true` 的维度和测试通过；
4. `TaskContract` 的入口、输出和不变量成立；
5. 执行环境和资产摘要与准备阶段一致。

界面显示可四舍五入，内部判定使用未四舍五入值。Agent 可以解释代码风格或错误原因，但不能生成正式 `dimensionScore`，也不能用语言评价覆盖阻断失败。

## 16. 草稿、运行、正式提交和学情证据

### 16.1 数据类型

```ts
interface HintEvent {
  hintId: string;                  // 提示事件编号
  level: Exclude<AssistanceLevel, "independent">; // 本次实际使用的帮助等级
  occurredAt: string;              // 服务端记录时间
}

interface ActivityDraft {
  sessionId: string;               // 所属会话
  activityId: string;              // 所属活动
  profileRevision: number;         // 锁定Profile修订号
  draftVersion: number;            // 草稿比较写版本
  code: string;                    // 私有用户代码
  codeHash: string;                // 草稿摘要
  hintEvents: HintEvent[];         // 已使用提示，不允许回退最高等级
  updatedAt: string;               // 最后保存时间
}

interface ActivityRunTrace {
  runId: string;                   // 一次公开运行编号
  sessionId: string;               // 所属会话
  activityId: string;              // 所属活动
  codeHash: string;                // 运行代码摘要；普通轨迹不保存完整代码
  result: ActivityResult;          // 公开预览结果
  createdAt: string;               // 服务端记录时间
}

interface ActivityAttempt {
  attemptId: string;               // 一次正式提交编号
  requestId: string;               // 幂等请求编号
  sessionId: string;               // 所属会话
  activityId: string;              // 所属活动
  assignmentId: string;            // 锁定的具体TaskAssignment
  profileRevision: number;         // 锁定Profile修订号
  source: TaskSource;              // 实际使用的任务来源
  assetBundleHash: string;         // 任务包摘要
  submissionRef: string;           // 私有submission.py引用，不嵌入普通摘要
  codeHash: string;                // 提交代码摘要
  highestAssistance: AssistanceLevel; // 提交前实际最高帮助等级
  resultRef: string;               // 私有result.json引用；领域仓储加载后再组合ActivityResult
  attemptStatus: "completed";      // 只有确定学习者结果才形成正式Attempt
  createdAt: string;               // 服务端接收时间
  committedAt?: string;            // 会话提交标记公开后的时间
}

interface ActivityEvidence {
  evidenceId: string;              // 交给11号Evidence适配器的编号
  attemptId: string;               // 一一关联正式提交
  primaryKnowledgePointId: string; // 唯一计分知识点
  normalizedScore: number;         // 0..1标准化结果
  independence: AssistanceLevel;   // 与11号Evidence.independence完全一致
  errorTags: string[];             // 安全学习错误类别
  evaluatorVersion: string;        // 判题版本
  environmentHash: string;         // 环境摘要
  assetBundleHash: string;         // 测试资产摘要
  createdAt: string;               // 服务端生成时间
}

interface EvaluationFailureRecord {
  requestId: string;               // 失败请求幂等编号
  attemptId: string;               // 可重试的服务端提交编号
  assignmentId: string;            // 当时锁定题目
  errorCode: EvaluatorErrorCode;   // 基础设施错误
  stage: EvaluationStage;          // 失败阶段
  environmentHash: string;         // 环境摘要
  createdAt: string;               // 服务端记录时间
}
```

`ActivityRunTrace` 是浏览器会话内的可选类型。六周接口不设计预览结果回传，因此它不得写入服务端活动仓储；页面刷新后只恢复代码草稿，预览输出可以丢失。

`ActivityDraft` 是应用层内存领域对象，包含完整 `code`。文件仓储投影时，`draft.py` 保存代码，`draft.json` 只保存其余字段、`codeHash` 和 `codeRef`，避免在两个文件重复保存代码。`ActivityAttempt` 同理：`attempt.json` 保存 `resultRef`，完整安全结果只保存于 `result.json`；仓储读取后才组合成包含 `ActivityResult` 的领域视图。

### 16.2 ActivityEvidence 到11号 Evidence 的映射

| ActivityEvidence/Attempt | 11号 Evidence | 映射规则 |
|---|---|---|
| `code_completion` 或 `debug` | `kind="code_completion"` 或 `debug`、`source="code_submit"` | 使用正式测试的 `normalizedScore` |
| `coding_practical` | `kind="coding_practical"`、`source="practical_rubric"` | 使用确定性Rubric总分 |
| `primaryKnowledgePointId` | `knowledgePointId` | 只映射主要知识点 |
| `highestAssistance` | `independence` | 四个枚举值一一对应 |
| `attemptId` | `attemptId` | 作为正式证据幂等键 |
| `profileRevision` | `profileRevision` | 来自Attempt锁定修订号 |
| `assetBundleHash` | 11号 Evidence 对应的审计扩展字段 | 不能替代评测器版本 |

判定结果优先按 `verdict` 映射，而不是只看总分：`passed → correct`，`partial → partial`，`failed → incorrect`。因此即使总分为1但某个阻断测试失败，仍映射为 `incorrect`；实现不需要人为把分数封顶。`evaluator_error` 不创建 `ActivityAttempt` 或 `ActivityEvidence`，只保存 `EvaluationFailureRecord`。`answer_exposed` 可以创建11号 `impact="none"` 审计记录，但不分配正式 `evidenceVersion`。

### 16.3 打开、运行和提交

- `openActivity` 创建或恢复 `ActivityDraft`，并由服务端签发下一次可用 `attemptId`；
- 自动保存只更新草稿，不生成 Evidence；
- “运行”使用新的 `runId`，可以重复，不占正式 `attemptId`；
- 正式“提交”使用固定 `requestId + attemptId`；
- 相同二元组重复请求返回第一次已提交响应；
- 一个 `attemptId` 完成后不能覆盖；修改代码后再次正式提交必须使用新 `attemptId`；
- `evaluator_error` 保存草稿和 `EvaluationFailureRecord`，保留同一 `attemptId` 供重试，不创建正式 Attempt，也不分配11号 `evidenceVersion`；
- 确定的通过、部分正确或学习者错误保存一次 `ActivityAttempt` 和一次 `ActivityEvidence`；
- `answer_exposed` 提交可以保存 Attempt，但 Evidence 适配器将其标为不影响掌握度。

### 16.4 提示等级不可回退

`ScaffoldLevel` 是 PathEngine 在进入活动前选择的教学支架；`AssistanceLevel` 是学习者实际使用的最高帮助程度。提交证据使用整个活动期间的最高 `AssistanceLevel`：从未打开帮助是 `independent`；看过提示是 `hinted`；看过同类范例是 `worked_example`；看过当前答案是 `answer_exposed`。关闭面板不会降低等级，避免把已获得完整帮助的提交误写为独立完成。

### 16.5 提交事务

`ActivityRepository` 只负责草稿、任务分配、提交代码、评测结果和 `ActivityAttempt` 候选，不拥有跨学情版本的最终提交权。应用层用 `LearningSessionUnitOfWork` 协调它与11号 `LearningSessionRepository`：把 `ActivityEvidence` 映射成11号通用 `Evidence`，写入11号 `evidence/`，再更新知识状态、路径候选和唯一 `latest.json`。因此不存在活动仓储和会话仓储各自提交一份正式 Evidence 的双事实源。

```text
保存submission.py候选                 # 私有代码，尚未成为正式Attempt
  → 保存ActivityResult候选             # 确定性评测结果
  → 保存ActivityAttempt候选            # 幂等关联requestId和attemptId
  → 映射并写入11号Evidence候选          # ActivityEvidence只是适配输入；evidence/是唯一正式事实
  → 更新KnowledgeState候选             # 复用11号聚合器
  → 必要时生成新路径候选                # 只重算未完成后缀
  → 最后原子更新latest.json             # 一次公开session/evidence/path版本
```

所有读取者只接受不超过11号 `latest.json` 提交标记的结果。若完整 `ActivityResult` 候选已落盘，恢复器根据 `requestId/attemptId` 补提交或隔离候选。若子进程可能已完成但结果尚未可靠落盘，原结果不可恢复：系统可以在确定性环境下用同一资产和 seed 重跑，或返回 `evaluator_error` 让用户重试；无论哪种都不得重复提交 `ActivityAttempt`、Evidence 或版本。本文不承诺进程崩溃后“绝不重复执行代码”，只承诺业务事实幂等。

## 17. HTTP、统一 facade 与 TUI

### 17.1 HTTP 路径

采用04号文档的唯一规范：

- `POST /api/activities/:id/run`：校验并签发公开执行包和 `runId`，不接收执行结果；
- `POST /api/activities/:id/submit`：执行权威正式提交。

不采用旧的单数 `/api/activity/{activityId}/run` 加 `action` 设计。

### 17.2 写请求字段

| 字段 | 中文含义 | 信任规则 |
|---|---|---|
| `requestId` | 网络请求幂等编号 | 客户端创建后重试保持不变 |
| `sessionId` | 当前学习会话 | 服务端加载并校验 |
| `sessionVersion` | 预期会话比较写版本 | 不一致返回 `session_conflict` |
| `profileRevision` | 会话锁定资料包修订号 | 必须等于服务端会话，不能切换 |
| `activityId` | 当前路径活动编号 | 必须属于当前可用节点 |
| `runId` | 公开运行编号 | `/run` 请求不携带；服务端响应签发并写入公开包 |
| `attemptId` | 正式提交编号 | `openActivity` 签发；`/submit` 必须携带该编号 |
| `draftVersion` | 用户代码草稿版本 | 避免旧页面覆盖新草稿 |
| `code` | 当前用户代码 | 私有数据，受大小和区域校验 |

客户端不得提交或覆盖：`hiddenTestRefs`、完整 Rubric、参考实现、允许库白名单、环境版本、测试文件、评分阈值和任意宿主路径。

### 17.3 LearningRuntimeFacade 用例

`LearningRuntimeFacade` 增加：

- `openActivity`：读取安全活动视图并创建/恢复草稿；
- `saveActivityDraft`：比较写保存代码；
- `prepareActivityRun`：校验并签发公开运行包；
- `submitActivity`：幂等正式提交并提交证据事务；
- `getActivityAttempt`：返回安全作答摘要；
- `recoverActivity`：按检查点恢复草稿或已提交结果。

facade 依赖活动仓储端口与 `CodeEvaluationPort`，不直接拼 shell 命令或读取隐藏文件路径。

### 17.4 TUI 适配

- `mcq`、`explain` 可以在 Pi 终端中使用安全文本视图；
- 代码补全、调试和实操显示目标、预计时间和一次性 localhost 深链；
- 深链只携带短期本机会话令牌，不含绝对路径、测试、答案或接口密钥；
- Web 完成后，TUI 调用 facade 读取同一 `ActivityAttempt` 摘要和下一步；
- Web 不可用时保存 pending 活动并提示恢复，不把代码任务临时改成简答题，也不误标完成。

## 18. 答案泄漏与上下文提问

### 18.1 服务端阶段

| 阶段 | 可以提供 | 禁止提供 |
|---|---|---|
| `learning` | 完整概念、非当前题示例、反例和普通问答 | 后续隐藏测试和当前任务答案 |
| `preSubmit` | API解释、概念提示、局部步骤和错误类别 | 当前完整实现、参考答案、隐藏断言 |
| `inProgress` | 定位建议、局部伪代码、允许的提示等级 | 独立实操完整最终方案 |
| `postComplete` | 完整复盘、参考思路和替代方案 | 隐藏测试源码及内部预期数据 |

阶段由服务端根据活动、提交状态和 `leakagePolicyId` 决定。前端隐藏按钮和提示词都不是安全边界；所有 Agent 上下文必须先由服务端裁剪。

### 18.2 提问规则

允许解释：

- Python/Pandas 错误信息的含义；
- 当前知识点相关 API 概念；
- 某个公开失败维度的定位思路；
- 不包含完整实现的局部伪代码。

禁止返回：

- 当前任务完整代码；
- 隐藏测试输入、断言和精确预期值；
- 参考实现；
- 绕过可编辑区域或运行限制的方法。

提问本身只形成11号软证据，不改变掌握度。若问答意外暴露当前完整答案，服务端将最高帮助等级提升为 `answer_exposed`。

上树是本文对02号已声明资产的职责展开，不是第二套目录规范。02号已经在 `profile.json.paths` 中声明 `task-generation`、`rubrics`、`reference-solutions`、`environments`、`activities`、`assessments` 和 `datasets`；实施时必须直接读取这些路径，不能由运行时代码猜测目录或另建 manifest。当前源码尚未实现全部v2目录的加载、构建、修订和启用校验，这属于六周新增工作；在实现完成前，相关目录可处于“规范已冻结、资产/代码未完成”状态，但不得再改变路径含义。

## 21. 安全与隐私清单

1. API Key、模型认证和无关环境变量不传给 Python 子进程。
2. 浏览器、Worker、React state、DOM、网络响应和缓存不出现隐藏测试与参考实现。
3. Agent 生成上下文不包含固定隐藏任务、完整私人答案和其他用户数据。
4. 用户代码不能通过请求传入宿主路径、测试路径、环境覆盖或允许库白名单。
5. 输出在展示前截断、转义并裁剪绝对路径，防止 HTML/ANSI 注入和信息泄漏。
6. 数据集为合成/授权资产并按摘要校验；六周不接受任意 CSV 上传。
7. 每次执行使用干净数据副本，不允许正式提交污染下一次运行。
8. 测试和参考实现不能由智能体直接修改 active 资产。
9. Node.js 子进程方案只承诺本地受信任测试环境，不能在报告中写“完全安全沙箱”。
10. 运行轨迹记录执行编号、版本、耗时、错误码、来源和摘要，不记录完整用户代码、隐藏断言或密钥。

隐藏测试和参考实现在应用层不会下发给学习者界面，但产品接收者若拥有本机文件系统权限，仍可能直接检查本地资产。这是轻量本地产品的已知边界；六周目标是防止通过正常界面、接口、日志和智能体上下文意外泄漏，而不是对本机管理员实现不可提取保护。

## 22. 测试设计

### 22.1 活动和资产契约

- 五类联合类型穷尽，未知类型被拒绝；
- ID、修订、知识点、来源、环境和 fallback 引用有效；
- 可编辑区域存在、唯一、不重叠并可重建完整代码；
- Rubric 权重和为1、阈值为 `0..1`、阻断维度完整；
- 测试、数据、参考实现和环境引用无悬空；
- 参考实现通过全部测试；
- 起始代码和已知错误实现不能误通过；
- Profile 第1版无需这些资产仍能继续旧流程。

### 22.2 固定任务方案

- 每个核心代码知识点至少有一项可执行固定任务；
- 无模型、无网络时可以打开、运行和正式提交；
- 数据 seed、参考输出和测试结果可重复；
- 公开反馈足够定位类别但不泄漏隐藏边界；
- 资料包损坏时拒绝启用，不影响旧 active。

### 22.3 AI 生成方案

- MotherTask 必须是维护者批准状态，不能由未审核网络内容直接进入抽取；
- 母题模板、起始代码摘要、可编辑区域、API白名单和可信Oracle引用缺一时拒绝；
- ThinkingAtom 必须引用母题、Profile知识点和来源，无法观察的抽象描述被拒绝；
- 原子抽取后未经维护者批准不能参加组合；
- 必填槽位缺失、值类型错误、白名单越界、原子依赖不全或冲突时拒绝；
- 必选知识覆盖/不变量原子不会因画像不同被删除；
- 相同规范化 `VariantPlanInput` 产生相同 `TaskVariantPlan`、seed和稳定任务选择；
- 冲突原子、越界难度、未授权槽值和跨Profile组合被拒绝；
- 三类典型画像能够匹配不同的难度、脚手架或错误原子，但最终标准不变；
- Generator 输出缺字段、越权 API、越界难度和非法来源被拒绝；
- Generator 未覆盖计划原子或私自增加原子时被拒绝；
- 整体替换起始代码、越过可编辑区或增加未允许导入的参考补丁被拒绝；
- 未绑定来源的题面断言和未注册的反馈码被拒绝；
- AI候选测试与AI参考补丁即使相互通过，只要不符合可信Oracle或白名单不变量仍被拒绝；
- 正式隐藏评测集、固定任务私有测试和接受后的私有测试不进入角色输入或普通轨迹；
- 参考实现失败、起始代码误通过、错误实现未被拒绝时不能发布；
- Hunter 能发现预置的测试遗漏、契约矛盾和泄漏；
- Defender 只在实质争议时触发；
- Judge 不能覆盖硬门禁；
- 一次受限修订后重新运行全部门禁；
- 模型超时、无效结构、审核拒绝、缓存损坏均返回固定 fallback；
- 恢复不重跑已完成 Generator baseline；
- 同一缓存键读取同一批准任务。

还需比较“运行时从零生成”和“构建期变式缓存”的调用次数、token、等待时间和失败率。正式方案必须证明学习页面优先命中已批准变式或固定兜底，不能把多轮生成审核等待转嫁给正在做题的用户。

### 22.4 第3周决策门测试

分别模拟：

- 代码补全 AI 通过、实操 AI 失败；
- 代码补全 AI 失败、实操固定；
- 两类都通过；
- 配置未冻结；
- AI 已冻结但单题生成失败。

验证每类独立选择、`pending` 默认固定、失败立即 fallback，以及正式评测记录实际来源。

### 22.5 执行器矩阵

至少覆盖：

- 正确和部分正确；
- 语法错误、运行错误、测试失败；
- 无限循环和输出过大；
- 非白名单导入和缺少输出入口；
- 用户代码抛出的异常；
- 测试资产抛错、依赖缺失和评测器崩溃；
- 预览取消、用户代码超时和评测器超时；
- 重复 `requestId/attemptId`；
- 陈旧 `sessionVersion`；
- 进程树终止、残留临时目录和清理告警。

### 22.6 双后端一致性

- 同一锁定任务、代码和公开输入在 Pyodide 与 Node.js 各运行至少三次；
- 第5周用10组公开输入比较通过/失败结论和关键输出；
- 隐藏测试只在正式提交运行，不要求预览端具有隐藏判定；
- 差异必须归因到环境或后端能力，并写入环境锁和限制报告；
- Pyodide 不可用时 Node.js 正式提交仍可运行；
- 外部评测器失败时回退 Node.js，不改变上层 `ActivityResult`。

### 22.7 证据、并发和恢复

- 公开运行不产生 Evidence、不改变 mastery、不重算路径；
- 确定学习者结果只保存一次 Attempt 和 Evidence；
- `evaluator_error` 不分配证据版本、不降低学情；
- 刷新恢复草稿但不自动提交；
- 同一幂等请求返回原结果；
- 陈旧会话版本不覆盖新提交；
- 在结果、Attempt、Evidence、KnowledgeState、路径和提交标记阶段分别注入中断；
- 已落盘完整结果可补提交；未知结果允许确定性重跑，但不得重复提交、计分或移动已完成节点；
- Profile 新修订不重判旧 attempt；
- Web 和 TUI 读取同一安全摘要。

### 22.8 协议、平台与越权专项

- Windows Job Object/等价机制能否终止孙进程；能力为 false 时能检测和报告残留；
- Python 协议通道与 stdout/stderr 分离，用户打印伪 JSON 不能注入 `EvaluatorResponseEnvelope`；
- UTF-8 分片、二进制输出、stdout/stderr 同时洪泛时按字节流式限制；
- 路径穿越、符号链接/结点、非法文件名和摘要篡改被拒绝；
- 动态导入、`os/subprocess/socket`、JavaScript桥接和本地接口探测按环境能力得到如实结果，不能用测试伪造网络隔离；
- 服务分别在 prepare、公开阶段、隐藏阶段和 commit 前重启，能够重建准备状态或安全失败；
- 两个相同 `requestId/attemptId` 并发竞争只有一个业务提交成功；
- 用户异常、测试异常和协议异常按 `stage` 正确归因；
- Pyodide 取消/销毁后迟到消息不更新新运行，运行间无全局状态污染；
- AI候选参考实现、测试规格和恶意槽值不能触及宿主文件、进程、网络和未知比较器。

### 22.9 泄漏扫描

自动扫描：React state、DOM、网络响应、浏览器缓存、Worker 消息、Agent 输入、普通日志、工作流轨迹、Attempt、错误反馈和前端构建产物。

不得出现：隐藏断言、参考实现、完整 Rubric 阻断逻辑、API Key、宿主绝对路径、未裁剪隐藏输入和精确预期输出。

### 22.10 Pandas 和通用函数题验收

Pandas 固定或 AI 任务必须按题面契约验证列名、数据类型、行数、业务键唯一性、缺失值、格式和结果不变量。通用 Python 函数题验证函数签名、返回值、边界输入和异常行为。两者使用同一端口与结果协议，不建立 Pandas 专属执行器。

13. 当前 `pi-study-helper` 源码、Profile 第1版和五个旧命令没有被本文误写为已实现新能力。
