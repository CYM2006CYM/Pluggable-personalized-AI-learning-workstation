## 1. 阅读规则与状态标签

本文先写源码事实，再写扩展设计。任何没有源码或测试支撑的能力都不能标为“已有”。

- **[已有]**：当前源码、固定样例和测试已经证明。
- **[六周新增]**：六周最小可行产品必须补充。
- **[演示]**：比赛展示必须能够观察到，通常依赖六周新增能力。
- **[可降级]**：进度不足时仍能保持闭环的替代实现。
- **[候选]**：可以采用，但必须先通过原型。
- **[调研]**：需要比较、验证或形成报告，不能写成已确定方案。
- **[延期]**：六周不实现。

正文优先使用中文。真实目录、文件名、命令、类型、方法、字段、错误码以及 Node.js、TypeScript、React、Pyodide 等产品名称保留英文，以便和代码精确对应；首次出现时说明中文含义。

## 2. 当前运行基线

### 2.1 技术和依赖

| 项目 | 当前事实 | 作用和边界 |
|---|---|---|
| Node.js | `>=22` | JavaScript 服务端运行环境；执行 TypeScript 扩展和本地脚本，不是 Python 代码沙箱 |
| TypeScript | ESM 模块 | 带静态类型的 JavaScript 语言；当前全部业务代码语言，编译只检查类型，不输出构建文件 |
| Pi Coding Agent | 开发依赖 `0.80.3`、运行时同伴依赖 | Pi 智能体宿主；提供扩展入口、模型、终端界面和模型注册表 |
| Loop Graph SDK | 项目内共同开发组件；当前应用清单仍通过 Git 依赖接入 | 工作流图开发工具包；SDK源码和应用均可按项目需要修改，不能把某个提交哈希当成永久架构边界 |
| Vitest | 测试框架 | TypeScript 测试工具；当前 `tests/` 有 21 个测试文件 |
| TypeBox | 同伴依赖 | TypeScript 结构化类型工具；不能代替全部领域校验 |

`package.json` 中的主要验证命令：

```text
npm run typecheck          # 检查源代码和测试代码类型
npm test                   # 运行 Vitest 测试
npm run check:docs         # 检查项目内文档链接
npm run smoke:extension    # 验证 Pi 扩展能否加载
npm run verify             # 类型、测试、文档、扩展和发布检查组合
```

这些命令是后续改造的回归门禁。若本机尚未安装依赖，应先执行 `npm ci`，不能把“命令不存在”误判为业务测试失败。

### 2.2 SDK 的版本与协同开发规则

Loop Graph SDK 是本项目的一部分，不按不可修改的第三方黑盒管理。当前 `package.json` 使用带 Git 引用的依赖，只能说明当前工作区的接入方式，不能推出“六周内必须固定在某个提交”。SDK 可以为串行协作、结构化产物、检查点或统一模型端口补充必要能力，但修改必须同时考虑现有应用调用方。

开发和验收采用以下规则：

1. SDK 与 `pi-study-helper` 可以在同一迭代中修改；不得用固定提交哈希阻止合理的接口调整。
2. SDK 源码纳入共享工作区后，开发期优先使用本地工作区依赖，使应用测试直接覆盖当前SDK源码；在源码尚未纳入前，保留现有Git依赖仅作为临时接入方式。
3. 不把Git提交哈希写入领域数据、HTTP协议、Profile、学习会话、缓存键或效果指标；这些业务数据不应因SDK内部提交变化而整体失效。
4. 需要追踪的是SDK的接口契约版本、运行产物结构版本和能力验证结果，而不是将某次源码提交永久冻结。
5. 修改导出类型、工作流执行语义、轨迹结构、取消/超时行为或会话生命周期时，必须同步修改适配器并运行类型检查、单元测试、扩展冒烟测试和SDK能力探测。
6. 仅修改SDK内部实现且公共契约和行为测试不变时，不要求修改业务版本；接口不兼容时必须升级契约版本并提供同一迭代内的应用适配。
7. 阶段演示和正式评测仍要保存当时实际使用的SDK契约版本及验证报告，保证结果可解释；这属于可追踪记录，不代表后续禁止继续修改SDK。

因此，本文后续所写的“当前SDK能力”都是当前代码和探测结果的事实快照，不是SDK未来能力的上限。若项目直接扩展SDK实现新能力，应先用测试证明，再把对应条目从“应用层补充”修订为“SDK已有”。

### 2.3 当前扩展启动装配

入口文件是 `src/extension/index.ts`。实际装配顺序如下：

```text
Pi 加载扩展入口
 → resolveStudyDataRoot() 解析本地数据根目录
 → 创建 traces/ 并建立 JSONL 运行轨迹记录器
 → 创建 ProfileFamilyRepository
 → 创建 PrivateMemoryRepository
 → 创建 ProfileBuildJobRepository
 → createStudyWalkingSkeletonGraphs() 创建九个现有工作流图
 → 注册五个 Pi 命令
 → Pi session_start 时初始化 demo-review 并扫描待恢复批次
```

隔离执行器不是扩展启动时立即创建。`executorFor(ctx)` 在某个命令第一次需要模型工作流时才调用 `createIsolatedGraphExecutor(ctx)`，并在该命令控制器生命周期内复用执行函数。它仍会为每一次工作流图运行创建新的隔离会话。

### 2.4 数据根目录解析

`src/config/data-paths.ts` 使用以下优先级：

```text
构造仓储时显式传入 dataRoot
 → 环境变量 PI_STUDY_DATA
 → 用户主目录/.pi/agent/study-helper-data
```

返回值始终解析为绝对路径，业务代码不依赖当前命令行工作目录。测试通过显式 `dataRoot` 使用临时目录，避免污染真实学习数据。

### 3.1 分层依赖规则

```text
extension（扩展入口） ──→ application（应用用例）
       │                         │
       ├─→ tui（终端交互）       ├─→ graphs（模型工作流）
       │                         └─→ repositories（持久化）
       └────────────────────────────→ infrastructure（通用设施）

application / graphs / repositories / tui ──→ domain（领域类型与规则）
```

这张图表示目标依赖方向。当前 `domain/study-profile.ts` 为加载学习材料而直接依赖 `ProfileFamilyRepository`，这是已知的局部反向依赖；六周抽取 `ProfileReader` 时应消除，不能继续扩大。必须遵守：

1. 领域层不得调用 Pi 界面、React、文件选择器或模型。
2. 终端和网页不得直接修改资料包文件或长期画像。
3. 智能体工作流不得直接决定路径合法性、客观题正确性和代码测试结果。
4. 仓储只负责数据一致性，不负责生成教学文案。
5. 扩展入口只装配对象，不应复制应用流程。

## 4. 七层架构的当前职责和扩展点

| 层次 | [已有] 主要职责 | 当前限制 | [六周新增] 接入方式 |
|---|---|---|---|
| 扩展入口层 | 注册命令、创建仓储/工作流/控制器、处理宿主事件 | 依赖 `ExtensionCommandContext` | 组装统一应用入口和 Pi 模型执行适配器 |
| 应用层 | 组织会话、画像、构建、修订和重试 | 学习控制器约700行且混合终端交互 | 保留兼容外观，抽取诊断、路径、活动、提问、完成和恢复短用例 |
| 领域层 | 类型、策略、资料解析、结构校验、证据纯计算 | 只有第1版复习模型 | 增加知识状态、路径、活动和错误分类纯规则 |
| 智能体工作流层 | 有界模型推理、输出结构校验、隔离运行 | 没有六类新增角色和跨进程恢复 | 增加职责单一的小工作流图；资源审核四阶段由应用层严格串行编排，诊断解释与学习决策分别单步调用 |
| 仓储层 | 本地资料包、私有记忆、构建任务和原子文件写 | 无路径/活动/审核检查点仓储 | 增加会话版本、路径快照、活动提交和审核运行存储 |
| 基础设施层 | 安全路径和原子写 | 不是代码沙箱 | 增加模型执行接口和代码评测适配器，不削弱路径安全 |
| 终端交互层 | 材料、题目、选择和文本输入 | 不适合代码编辑器和复杂路径可视化 | 调用同一应用入口；代码任务通过 localhost 深链进入网页 |

## 5. 五个现有命令的真实调用链

### 5.1 `/study [subjectId]`：学习会话

**启动检查**：命令要求 Pi 当前空闲且已选择模型。未给 `subjectId` 时通过终端列出 active 资料包。

**当前流程**：

```text
extension handler
 → StudySessionController.run()
 → 选择资料包、范围、学习模式、难度和题型
 → PrivateMemoryRepository.createPendingBatch()
 → study_generate_question
 → TUI 显示题目并收集答案
 → 确定性选择题/判断题判定，或 study_grade_answer 语义判题
 → 可选 study_discuss_question
 → 保存 Attempt 和 running session
 → study_summarize_session
 → 完成批次并保存 summary.md
```

**写入数据**：`session.json`、`attempts/<question-id>.json`、`summary.md`。

**失败行为**：已创建的 pending batch 和完成的 Attempt 保留；控制器尽量把运行中会话标为 interrupted 或留待 `/study-recover`。当前恢复不是从任意模型节点续跑，而是读取已落盘会话与作答后补总结或中断。

**适配方式**：现有三种模式继续工作。第2版路径和活动通过统一应用入口新增，不删除此控制器；客观题可用适配器映射到现有 `ReviewQuestion/Attempt`，代码活动转入网页。

### 5.2 `/study-recover`：恢复未完成会话

**启动检查**：只要求 Pi 空闲；若需要补生成总结，还需要运行时模型能力。

**当前流程**：列出各 active 学科的 pending batch，筛选可恢复会话，让用户选择“补总结并完成”或“标记中断”。补总结时依据已保存会话和 Attempt 重新构造 `SessionEvidence`。

**当前边界**：只能恢复学习记录批次，不会恢复未完成的 Profile 修订智能体、网页草稿、路径重算、代码子进程或多智能体审核阶段。

**适配方式**：[六周新增] 统一恢复页在现有 pending batch 基础上增加路径、活动和审核运行检查点；旧批次仍按当前规则读取。

### 5.3 `/study-profile [subjectId]`：长期画像

**启动检查**：要求 Pi 空闲且有模型。只处理具有待消费批次的 active 学科。

**当前流程**：

```text
选择学科和 pending 批次
 → 领域代码聚合确定性总数与会话证据
 → study_update_learning_profile 生成画像候选
 → 用户确认
 → 原子保存 learning_profile.json
 → 已消费批次从 pending 移入 archived
```

**失败行为**：用户未确认或模型失败时不归档批次；旧画像保留。

**适配方式**：当前 `LearningProfile` 作为兼容摘要视图保留；新增 `KnowledgeState` 作为路径输入，但不能用旧总正确率直接替代知识点掌握度。

### 5.4 `/study-build [sourceDir]`：资料包构建

**启动检查**：要求 Pi 空闲且有模型；源目录只递归接受 Markdown/txt 等受支持文本。

**当前流程**：盘点文件并计算内容摘要→建立构建任务→分批调用 `study_build_profile_fragment`→保存每批 fragment→领域代码组装 canonical draft→结构检查→用户选择启用、保留或丢弃。

**恢复能力**：`profile_build_jobs/<job-id>/` 保存任务和已完成分片；状态为 `extracting`、`draft_ready` 或 `failed` 的任务可继续。模型不会重新处理已经保存的有效分片。

**适配方式**：第2版不能只让仓储新建额外目录。必须同步扩展源资料盘点、片段结构、组装器、清单解析、交叉引用校验、质量检查和测试。

### 5.5 `/study-revise [subjectId]`：资料包修订

**启动检查**：要求 Pi 空闲且有模型。可以继续已有 draft，或从 active 创建 revision+1 的修订草稿。

**当前流程**：

```text
创建/读取 draft
 → study_plan_profile_revision 规划最小修改范围
 → 必要时向用户澄清
 → study_revise_profile_draft 生成受控补丁
 → 应用层展示实际差异并等待确认
 → 原子应用到 draft
 → study_review_profile_draft 独立质量审查
 → 系统更新 quality_report.md
 → 用户选择启用、继续修改或保留 draft
```

**安全边界**：智能体补丁不能随意修改 `profile.json` 和 `quality_report.md`；质量报告由系统根据审查输出写入。启用时检查 `revisionOf`，防止陈旧草稿覆盖新 active。

**适配方式**：第2版新增可变路径白名单和影响分析；datasets、隐藏测试和参考实现默认不允许智能体修改，只能由维护者放入后重新校验。

## 6. 当前领域模型与能力边界

| 类型 | 当前保存内容 | 当前用途 | 尚不具备 |
|---|---|---|---|
| `Profile`（资料包） | subjectId、version、revision、status、slot、固定路径 | 定位和校验 active/draft 资料 | schemaVersion、capabilities 和第2版目录 |
| `StudySession`（学习会话） | sessionId、subjectId、状态、模式、范围、题数、时间 | 当前一次 TUI 学习循环 | profileRevision、诊断、路径、活动、sessionVersion |
| `ReviewQuestion`（复习题） | 题型、题干、选项、答案、解析、来源 | 当前问答活动 | 代码任务、Rubric、隐藏测试 |
| `Attempt`（作答） | 题目、用户答案、历史提交、判题、知识点、讨论摘要 | 每题证据和会话归档 | 统一代码提交结果和评测器版本 |
| `GradeResult`（判题结果） | 是否正确、答案、解析、知识链、建议 | 语义短答判定 | 不能代替代码测试结果 |
| `SessionSummary`（总结） | 总题数、准确率、强弱项、建议 | 会话结束报告 | 知识点级置信度和路径版本 |
| `LearningProfile`（长期画像） | 累计题数/正确率、强弱项、未验证主题、建议 | 跨会话摘要 | 不是 `KnowledgeState`，不能直接驱动精细路径 |
| `SessionEvidence`（会话证据） | 由 session/attempt 推导的确定性统计和安全摘录 | 总结和画像候选输入 | 尚未包含代码测试、Rubric 和提示独立性 |

### 6.1 题型预留不等于可用

`QuestionType` 声明了 `choice | multi_choice | judgment | short_answer`，但当前生成工作流、资料构建/修订校验和 TUI 没有完整开放 `multi_choice`。因此当前可演示题型是单选、判断和简答；多选题只能标为类型预留，不得列为已有用户能力。

### 6.2 学习模式

当前 `ReviewMode` 有：

- `practice`：直接练习；
- `card_practice`：先进行卡片回忆再练习；
- `chapter_study`：阅读章节或小节材料后练习。

系统推荐路径、快速实战和查漏补缺是[六周新增]的统一路径入口，不是当前第四、第五种 `ReviewMode`。它们应通过适配层映射到新路径状态，不能修改旧枚举后让旧测试失效。

### 6.3 判题职责

`study_grade_answer` 是语义判题工作流图，不是一个业务智能体角色。选择题和判断题优先由代码确定性判断；代码任务由 `CodeEvaluationPort` 的测试和评分规则判定；模型不得覆盖确定性失败。

## 8. Profile 第1版结构和生命周期

### 8.1 固定清单

`profile-schema.ts` 当前要求：`profile.json` 可解析；`subjectId/name/status/slot/version/createdAt/updatedAt` 为非空字符串；`revision` 为正整数；`status` 与 `slot` 只能是 `active` 或 `draft` 且必须相同；固定路径必须匹配 canonical 值。

必须存在：`subject.md`、`knowledge_index.json`、`source_map.json`、`quality_report.md`、`cards/`、`chapters/`、`exam_points/`。校验器检查知识索引含 `chapters` 对象，来源映射含 `sources[]` 和 `mappings{}`。

### 8.2 生命周期

```text
源材料
 → ProfileBuildJobRepository 保存构建检查点
 → createDraftProfile() 创建规范 draft
 → 构建/修订控制器受控写入
 → 结构和质量门禁
 → 用户确认 enableDraft()
 → 原 active 移入 archived
 → draft 原子切换为 active
```

- 新建资料包已有 active 时必须改走修订草稿，不能直接覆盖。
- `createRevisionDraft()` 复制 active，revision+1，并用 `revisionOf` 记录来源版本。
- `enableDraft()` 检查 `revisionOf` 是否仍匹配当前 active，陈旧草稿必须拒绝。
- 启用过程使用准备目录和备份目录；任一步失败会尽力恢复 active/draft。
- `archived` 是旧版本目录，不写入 manifest 的 `status`。

### 8.3 第2版贯穿改造

第2版适配必须同时修改：清单解析、固定/条件目录校验、领域只读目录、源资料盘点、构建片段、组装器、构建检查点、修订白名单、交叉引用检查、质量审查提示和测试样例。若只修改 `ProfilePaths`，会出现“能读但不能构建/修订/启用”的半成品。

## 9. 三类仓储和事务边界

### 9.1 `ProfileFamilyRepository`：资料包家族仓储

**已有能力**：初始化示例、列出 active、读取 active/draft、列修订候选、创建新 draft、复制修订 draft、受控读写 draft、列文件、原子应用变更、启用和丢弃。

**事务保证**：新建、批量修改和启用使用临时/备份目录与重命名；失败时恢复原目录。active 没有公开写入方法。

**边界**：目录级原子重命名不等于数据库多事务；程序崩溃遗留的隐藏临时目录需要启动清理策略时另行设计。当前只扫描 Markdown/JSON 内容，不支持二进制数据集修订。

### 9.2 `PrivateMemoryRepository`：私有学习记录仓储

**已有能力**：创建 pending batch、保存 Attempt、更新 running session、完成/中断会话、加载/列出 pending、读取画像、原子保存画像并归档所选批次。

**事务保证**：单文件原子写；画像写入和批次归档按受控顺序执行，失败时保留未消费批次以便重试。

**边界**：当前没有 `sessionVersion` 比较写入、路径快照、活动提交、知识状态和多智能体运行产物；Web/TUI 同时写时缺少完整冲突协议。

### 9.3 `ProfileBuildJobRepository`：构建任务仓储

**已有能力**：创建/加载/列任务、列未完成任务、保存/加载分片、更新状态。状态为 `extracting | draft_ready | enabled | kept_draft | discarded | failed`。

**恢复范围**：可以从已保存源清单和 fragments 继续构建，不需要重做全部模型调用。

**边界**：只负责资料构建任务，不负责学习会话、Profile 修订工作流或多智能体实时审核。

### 9.4 网页访问规则

React 前端不得直接调用任何仓储或读取文件路径。本地 Node 后端通过统一应用入口调用仓储，并只返回裁剪后的安全数据。TUI 同样逐步改为调用统一应用入口，避免两端形成两套写入规则。

## 11. 隔离执行器和串行能力

`createIsolatedGraphExecutor()` 的当前事实：

1. 从 `ExtensionCommandContext` 取得模型、工作目录、认证存储、模型注册表和取消信号。
2. 每次工作流图执行创建新的内存 `AgentSession`，完成后在 `finally` 中释放 host。
3. `defaultTools: []`，智能体默认不能调用宿主文件/终端工具。
4. 参数通过 `background` 传入，调用边界为 `delegate`。
5. extension 配置 `rootMaxSteps: 10` 和 `agentRunTimeoutMs: 300_000`。
6. 运行生命周期写入 JSONL 轨迹。

按当前SDK源码与能力探测结果，现有集成尚未提供并行分叉/汇合、内建角色消息总线、同一扩展并发根执行保证或跨进程检查点恢复。因此当前可直接实现的多智能体适配必须由应用层串行编排，每个角色等待上一阶段完成、校验并持久化后才能运行。SDK作为项目内组件可以继续修改；若把检查点或其他通用能力下沉到SDK，必须先冻结接口契约、补充探测与回归测试，再同步修订03、09号文档，不能仅修改SDK后继续沿用旧能力声明。详细角色设计见03号文档。

## 12. TUI 和应用控制器边界

`StudyTuiGateway`（学习终端交互网关）接受安全视图模型：题目视图不包含正确答案、解析和来源内部字段；材料视图只接受 Profile 材料；卡片回忆支持先回忆、查看正文或直接练习；题目支持首次作答、订正和明确放弃。

当前 `StudySessionController` 同时负责：选择流程、会话生命周期、工作流调用、TUI 状态、作答循环、讨论、持久化和总结。这对当前 Pi 命令可用，但网页不能直接复用绑定的 `ctx.ui` 和长阻塞循环。

[六周新增] 保留旧控制器公共行为和测试，逐步抽取短应用用例：创建会话、提交诊断、构建路径、打开节点、运行/提交活动、上下文提问、推进路径、完成和恢复。TUI 与 Web 只做显示/输入适配。

```text
会话 / 诊断 / 路径 / 活动 / 提问 / 总结 / 恢复用例
          │                 │                 │
          ↓                 ↓                 ↓
确定性领域规则    ReviewOrchestrator    CodeEvaluationPort
                  串行智能体审核          代码评测接口
          └─────────────────┬─────────────────┘
                            ↓
资料包仓储 / 私有记录仓储 / 审核运行仓储 / 运行轨迹
```

核心原则：网页和终端共享同一应用用例、领域规则和持久化；只有显示方式不同。React 不访问文件、工作流 SDK 或模型。

## 17. 公共接口及责任

### 17.1 `LearningRuntimeFacade`：统一应用入口

固定暴露17个短用例：`startSession`、`recoverSession`、`completeSession`、`saveDiagnosticDraft`、`submitDiagnosticAnswer`、`completeDiagnostic`、`buildPath`、`confirmPath`、`getNextStep`、`replanPath`、`openActivity`、`saveActivityDraft`、`prepareActivityRun`、`submitActivity`、`getActivityAttempt`、`recoverActivity`、`askContextQuestion`。旧名称仅为兼容别名。

它负责输入结构校验、`sessionVersion`、调用用例、错误映射和安全输出；不负责直接渲染 UI。

### 17.2 `ModelExecutionPort`：模型执行接口

当前隔离执行器绑定 Pi 的 `ExtensionCommandContext`，Web 后端不能直接创建它。Pi 适配器用现有隔离执行器实现此接口；Web 适配器通过 OpenAI 兼容配置提供模型能力。两端必须调用相同工作流图和编排器。

### 17.3 `ProfileReader`：资料包只读接口

按 subjectId/revision 返回不可变 Profile snapshot、知识点、来源和活动。运行中的会话不能因 active 更新而读取到新内容。

### 17.4 `PathEngine`：确定性路径生成器

输入目标、先修图、知识状态、活动和预算；输出合法路径和 reasonCodes。智能体只解释，不修改节点。

### 17.5 `CodeEvaluationPort`：代码评测接口

准备任务并运行 preview/submit；输出安全结果和 `learner_error/evaluator_error`。隐藏测试和评分规则只在服务端解析。

### 17.6 `ReviewOrchestrator`：串行审核编排器

逐步等待 Generator（生成者）→Hunter（核查者）→条件 Defender（辩护者）→Judge（裁判者），负责重试、检查点、降级和持久化；工作流图本身不决定下一角色。

## 18. Web 和 TUI 适配

### 18.1 Pi 端

保留命令注册、`ctx.isIdle()`、模型检查和 `createIsolatedGraphExecutor(ctx)`。旧 TUI 流程先由 facade 适配，确保现有测试不因一次性重构大面积失效。

### 18.2 Web 端

React（网页界面库）与 Vite（网页构建工具）通过本地 Node.js HTTP 层调用统一应用入口。Node.js 层持有模型密钥、仓储和评测器；浏览器只接收安全视图。六周优先 REST JSON（基于 HTTP 的 JSON 接口）加轮询，不为状态动画引入消息队列。

### 18.3 代码活动交接

TUI 显示任务说明和一次性 localhost 深链，Web 校验 session/activity/revision 后打开编辑器。提交写入共享仓储；TUI 重新读取 `ActivityAttempt` 摘要。URL 不包含绝对路径、隐藏测试或答案。

## 19. 数据版本、一致性和恢复

| 标识 | 作用 | 写入规则 | 冲突/失败行为 |
|---|---|---|---|
| `profileRevision` | 固定会话使用的资料版本 | session 创建时快照 | 旧会话继续旧版本，新会话用新 active |
| `sessionVersion` | 防 Web/TUI 双写 | 每个写请求比较后递增 | 不一致返回 `session_conflict` 和最新安全视图 |
| `attemptId` | 正式提交幂等 | 相同编号只写一次 | 重复请求返回原结果 |
| `evidenceVersion` | 标识知识状态输入版本 | 正式证据写入后递增 | 路径只基于明确版本重算 |
| `pathVersion` | 保存不可变路径快照 | 每次确认/重算新建 | 已完成节点不被新版本移动 |
| `generationRunId` | 关联多智能体审核阶段 | 同一运行单角色串行 | 从最后完整阶段恢复，baseline不覆盖 |

文件写入遵循“写临时文件→校验→原子替换”。模型超时使用同版本固定资源；评测器异常不产生学习者负证据；磁盘写失败时界面不能显示“已保存”。

## 20. 安全、隐私、运行观测和成本

### 20.1 文件和资料安全

- `assertSafeRelativePath/resolveInside` 防止路径穿越；
- Profile 扫描拒绝符号链接；
- active 只读，draft 受控写；
- datasets、隐藏测试、参考实现需哈希校验并禁止智能体修改；
- 浏览器不接收绝对路径。

### 20.2 模型和学习数据

- API Key 只保存在环境变量/模型注册表，不进入 React、URL 或运行轨迹；
- 智能体只接收所需来源片段和脱敏学习状态；
- 私有学习记录不随 Profile 分发；
- 真实试测使用匿名编号和最小采集。

### 20.3 Python 运行边界

Pyodide（浏览器内 Python 运行环境）的 Worker 用于公开“运行”；Node.js 子进程用于正式“提交”和隐藏测试。Node.js 方案限制工作目录、环境变量、时间、输出和临时文件，但只承诺本地受信任测试环境，不宣称生产级恶意代码隔离。

### 20.4 运行观测

记录 runId、stage、角色、模型、提示版本、耗时、token、cacheHit、fallback、状态和错误码。不得记录密钥、隐藏测试、参考实现或完整私人答案。普通问答不能无条件走完整审核链，否则延迟和成本失控。

## 21. 明确延期和不采用的架构

六周不实现：FastAPI、Milvus、Neo4j、消息队列、负载均衡、并行智能体网络、云数据库、登录、多租户、任意 CSV 上传、运行时任意 `pip install`、生产级恶意代码沙箱、模型训练/微调和第二个完整领域。

CIDPP只评价教学内容质量，用于检查讲义、题面和反馈的清晰性、完整性、深度、实用性与针对性；学习者五维评分已按D01正式进入上层画像，但不能替换 `LearningProfile` 或 `KnowledgeState`，也不能驱动路径。GoA/OAEO仅作为离线资料初始化/修订调研，不进入实时学习链。

## 22. 六周实施顺序

```text
第1周：核对事实、运行第1版回归、冻结Profile/接口/错误码
 → 第2周：第2版资料包、来源锚点、固定诊断和路径基础
 → 第3周：知识状态、统一路径、五类活动、统一应用入口及Node.js正式评测原型
 → 第4周：React网页骨架、串行多智能体、任务缓存和恢复检查点
 → 第5周：React主流程、Pyodide公开预览、TUI深链、重复性/超时门禁和端到端
 → 第6周：全量回归、效果评测、演示和提交冻结
```

依赖门禁：资料包和传输数据契约未冻结不得并行开发页面；路径合法性未通过不得让智能体解释替代；执行器未通过答案隔离不得接入生成任务；Web 未使用共享 facade 不得进入端到端验收。
