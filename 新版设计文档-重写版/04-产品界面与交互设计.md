> **D01/D09 已冻结**：诊断完成后展示五维“分数+置信度+证据卡”；无证据维度显示 `unverified`，不显示0分。P0代码编辑器为普通 `textarea`，保证编辑、Tab、草稿恢复、公开运行、正式提交和反馈；Monaco高级能力延期。页面只调用统一 Facade，不读写仓储。

> 版本：MVP-1.1  
> 状态：`[MVP]`，Web 是新增适配层；Pi/TUI 为当前可复用入口。  
> 目标读者：产品、前端、应用层、Agent、测试和演示人员。

## 1. 设计目标与真实边界

### 1.1 我最终要体验什么

我是一个第一次接触本资料包的学习者。我打开本地网页后，能够完成：

```text
选择资料包和目标
→ 填写背景并完成固定诊断
→ 看懂“为什么给我这条路径”
→ 阅读一张适合我的学习卡片
→ 做自测/客观题/代码补全/独立实操
→ 在页面内运行和提交 Python
→ 看到错误类别、来源和下一步
→ 刷新或关闭后恢复
→ 查看学习总结并继续下一节点
```

页面只负责展示和收集输入，不负责判断掌握度、构造非法路径、修改 Rubric 或访问隐藏测试。所有业务动作通过统一 `LearningRuntimeFacade` 进入 application/domain/repository/Graph 链路；`StudyAppFacade` 仅是旧文档兼容别名。

### 1.2 当前代码与新增 Web 的关系

| 能力 | 当前 `pi-study-helper` | 本文设计 |
|---|---|---|
| Pi Extension | `[已有]` `src/extension/index.ts` 注册五个命令 | 保留，增加 Web 深链/状态提示 |
| TUI | `[已有]` `StudyTuiGateway` 显示材料、题目、选择、输入 | 作为同一 facade 的另一个 adapter |
| 学习模式 | `[已有]` `practice/card_practice/chapter_study` | 映射到统一入口；路径模式为 `[MVP]` 新增 |
| Profile | `[已有]` active/draft/v1 canonical 文件 | Web 只读 active catalog；v2 字段为 `[MVP]` |
| Session/Attempt | `[已有]` 本地 private memory、pending batch、恢复 | 新增 sessionVersion/path/activity checkpoint |
| Graph | `[已有]` 出题、判题、讨论、总结等 Graph | Web 不直接调用 Graph，由 facade/Orchestrator 调用 |
| React/HTTP | `[缺口]` 当前不存在 | `[MVP]` 本地 Node HTTP + React/Vite |
| Python 执行 | `[缺口]` 当前不存在 | `[MVP]` Pyodide 预览 + Node 受限提交 |

六周不做登录、多用户云端、公网部署、任意 CSV 上传、任意 `pip install`、移动端独立应用和生产级恶意代码沙箱。代码执行面向本地受信任测试者。

## 2. 我打开网页后看到的整体布局

### 2.1 AppShell 草图

桌面宽度下所有学习页采用三栏；开始页/总结页可以隐藏左栏。

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Pi Study Helper  | Pandas 数据清洗 · revision 2 | 保存中/已保存 退出 │
├───────────────┬──────────────────────────────────────┬───────────────┤
│ 我的路径       │ 当前学习内容                         │ 提问与依据     │
│               │                                      │               │
│ ✓ 读取 CSV     │ 标题、卡片、题目或代码编辑器         │ [提问输入框]    │
│ ● 检查表结构   │                                      │ 选中内容依据    │
│ ○ 处理缺失值   │                                      │ 来源锚点列表    │
│ ○ 处理重复值   │                                      │ Agent 状态      │
│ ○ 类型清洗     │                                      │               │
│ ○ 最终实操     │                                      │               │
├───────────────┴──────────────────────────────────────┴───────────────┤
│ [返回] [保存状态] [查看路径理由] [下一步] [结束并总结]                  │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 全局区域的职责

- **顶栏**：显示 `subjectId/name`、`profileRevision`、当前模式、`sessionId` 的短标识、保存状态和恢复入口；不显示 API Key、完整 trace 或隐藏测试。
- **路径栏**：只显示当前不可变 `LearningPath` snapshot 的节点状态；点击已完成节点只读查看，不能绕过先修直接解锁后置节点。
- **主区域**：一次只展示一个活动，避免同时出现多个待提交动作；所有 Agent/执行器状态使用步骤条。
- **问答栏**：当前节点/选中文本/运行错误的上下文提问；问题是 soft evidence，不直接改变 mastery。
- **底部操作栏**：返回、保存、下一步、结束等动作。正在保存或提交时禁用重复点击。

### 2.3 响应式规则

MVP 只保证桌面窗口（建议 1280×800 及以上）和窄桌面；窄屏时顺序变为主区域→问答栏→路径栏，不承诺移动端独立 App。焦点顺序必须与视觉顺序一致，颜色不能作为唯一状态含义。

## 3. 页面、路由和切换总表

| 路由 | 我在这里做什么 | 进入条件 | 成功后 | 中断/失败 |
|---|---|---|---|---|
| `/` | 选资料包、目标、入口，恢复会话 | 无 | `/diagnostic`、`/path` 或 `/learn` | 留在开始页展示错误 |
| `/diagnostic/:sessionId` | 填问卷、答固定诊断、保存进度 | 新 session 或未完成诊断 | `/path/:sessionId` | 保留已答，进入 `/recover` |
| `/path/:sessionId` | 查看画像、确认/调整路径 | 诊断完成或章节入口 | `/learn/:sessionId/:nodeId` | 路径不可行则回到选择 |
| `/learn/:sessionId/:nodeId` | 阅读卡片、查看来源、提问 | 有合法 path snapshot | `/activity` 或下一个节点 | fallback 卡片/恢复 |
| `/activity/:sessionId/:activityId` | 自测、客观题、Run、Submit | 活动属于当前节点 | `/learn` 或下一节点 | 保留草稿，显示错误 |
| `/summary/:sessionId` | 看总结、证据和继续选项 | 主动结束或完成目标 | 新 session/原路径继续 | 可返回未完成节点 |
| `/recover/:sessionId` | 选择 checkpoint、处理冲突 | 发现 pending/checkpoint | 原页面 | 放弃未提交草稿，不删已保存证据 |
| `/review/:generationRunId` | 演示时查看 Agent 串行审核 | 具有演示权限/本地 token | 返回来源页面 | 仅显示脱敏失败摘要 |

路由切换由 `RouteGuard` 校验 session、revision、节点和活动归属；前端不能仅通过改 URL 解锁页面。

```text
│                                                         │
│ 我的目标：[完成订单数据清洗任务        ] 可用时间：[45] 分钟 │
│ [开始学习]                              [恢复上次会话]     │
└─────────────────────────────────────────────────────────┘
```

我只能看到校验通过的 active Profile。draft、损坏包、哈希不匹配或 capability 不足的包显示为不可选择，并有“查看原因”按钮。

### 4.2 我点击开始后的技术链

1. `startSession({requestId, subjectId, mode, goalId, availableMinutes, chapterId?})`；创建前没有`sessionId/sessionVersion`，客户端也不能指定revision。
2. facade 读取并校验当前 active Profile，由服务端固定 `profileRevision`、内容摘要和新 `sessionId`，返回初始`sessionVersion=1`。
3. 章节模式可直接进入合法章节；系统推荐进入诊断。快速实战和查漏补缺保留为未来入口，不在六周Pandas版本展示。
4. 返回 `SessionView {sessionId, sessionVersion, profileRevision, entry, diagnosticRequired}`。

如果当前 Pi 有未完成批次，开始页显示数量和最后更新时间，调用 `/study-recover` 对应的 repository 查询；不会自动覆盖旧 session。

### 4.3 失败和状态

- Profile 无效：`invalid_profile`，我仍留在开始页，可选择其他 active 包。
- 本地服务未启动：`offline`，显示“启动本地服务后重试”，不生成假 session。
- session 创建写盘失败：`storage_error`，不显示“已开始”，保留表单内容。

## 5. 第二站：我是第一次完成诊断

### 5.1 背景问卷页

我看到进度“第 1/2 步”和四个低负担问题：Python 经验、Pandas 经验、偏好的学习方式、可用时间。每题提供“不确定”，不强迫我编造答案。

点击“下一步”只保存问卷快照，不直接把自评当作掌握证据；点击“稍后完成”会保存 `diagnosticDraft` 并回到开始页。

**技术**：`saveDiagnosticDraft` 写入 private memory/checkpoint；字段受枚举和范围校验，未知值为 `unknown`，不能用 0 代替。

### 5.2 固定诊断题页

我看到一题一页、进度、题型、跳过和上一题按钮。选择题沿用当前 TUI 的安全展示原则：题干和选项显示在 ViewModel，正确答案不下发到前端。短代码题可以使用普通 textarea；MVP 不把 `multi_choice` 预留类型误写成已支持 UI。

```text
诊断 · 4/10                         [已自动保存]
知识点：检查 DataFrame 结构（仅供提示，不显示答案）
题目：读取数据后，怎样确认列名和数据类型？
[ 输入答案/选择选项                         ]
[上一题] [跳过，标记证据不足] [保存并退出]
```

提交调用 `submitDiagnosticAnswer`，确定性代码写 `Evidence(kind=diagnostic)`；不会因我提问或跳过直接扣 mastery。诊断 Agent 在全部固定题完成后才生成摘要。

### 5.3 诊断完成页

我看到三种清晰状态，而不是一张看起来很精确的雷达图：

| 状态 | 页面文案 | 下一步 |
|---|---|---|
| 已证明 | “已有直接证据” | 允许在满足阈值时压缩先修 |
| 正在学习 | “有部分证据，还需要练习” | 安排学习活动 |
| 未验证 | “目前没有足够证据” | 插入短验证节点 |

页面同时显示“系统如何得出结论”的证据折叠面板。Diagnosis Agent 失败时用确定性模板，不阻塞进入路径确认。

```text
│ 已证明 2 · 正在学习 2 · 未验证 2      证据置信度：中       │
│ 说明：这是基于固定诊断的暂时判断，可在学习中更新。         │
├─ 推荐路径（预计 38 分钟）────────────────────────────────┤
│ 1 ✓ 读取 CSV       基础       你已有证据                  │
│ 2 ● 检查表结构     基础       目标必需                     │
│ 3 ○ 处理缺失值     基础       诊断未验证                   │
│ 4 ○ 处理重复值     综合       处理订单任务需要             │
│ 5 ○ 最终验证       综合       目标核心                     │
├─────────────────────────────────────────────────────────┤
│ [查看为什么这样安排] [选择其他侧重点] [确认这条路径]        │
└─────────────────────────────────────────────────────────┘
```

“按章节学习”或“系统推荐”只改变 PathEngine 的入口约束，不创建第二套路由引擎。快速实战、查漏补缺属于未来扩展；用户不能选择尚未满足先修的后置节点。

### 6.2 确认与切换

- 确认：`confirmPath(pathId, sessionVersion)` 保存不可变 `path snapshot`，进入第一个 `available` 节点。
- 返回问卷：创建新 `diagnosticVersion`，旧路径保留为历史，不原地改写。
- 重新选择目标：调用同一 `buildPath`，返回新 `pathVersion` 和 `changeReasons`。
- 浏览器返回：如果没有未保存写操作，返回上一个页面；有草稿则先显示确认对话框。

### 6.3 不可行路径

如果时间预算不足，页面明确显示“将压缩可选活动，不删除核心目标”；如果用户请求跳过先修，显示缺失知识点和“先做 5 分钟验证”按钮。`path_infeasible` 由服务端产生，前端不自行修补节点顺序。

## 7. 第四站：我进入学习工作台 `/learn`

### 7.1 我看到什么

主卡片按固定结构显示：本节目标、为什么现在学习、前置概念、核心解释、示例、常见错误、无分自测、来源依据、活动入口。右侧路径栏显示当前节点，底部有“标记已读”但没有“直接标记掌握”。

我可以：

1. 展开来源锚点，看到来源文件、章节/片段和版本；
2. 选中一句话提问；
3. 先做无分自测；
4. 进入客观题或代码活动；
5. 暂停并恢复。

### 7.2 卡片生成和审核状态

页面顶部用一个串行步骤条展示：`生成 → Hunter 核查 →（争议时）Defender → Judge → 已发布`。步骤条不能同时显示多个 Agent 正在运行，避免暗示并行。普通已缓存卡片显示“来自审核缓存”；fallback 显示“使用资料包固定版本”。

卡片只收到允许的 `sourceAnchorIds` 和 `leakagePhase` 内容；前端不接收 system prompt、hidden test、参考答案或完整模型 trace。

### 7.3 上下文提问

我可以在卡片中选中文字，右侧输入“这一步为什么要先检查 dtype？”。服务端构造 `ContextAnchor(card_section/selected_text/path_node)`，只把安全摘录和来源白名单交给 Agent。回答带来源；资料中没有依据时显示“当前资料没有足够依据”，而不是让模型自由扩展。

提问只记录 soft evidence（例如需要例子、偏好解释），不会直接降低 mastery，也不会跳过活动。

## 8. 我完成无分自测和客观题

### 8.1 无分自测

页面显示问题和“先在脑中回答”，点击“显示参考要点”只显示讲义中的允许内容；无分自测不能产生计分证据。点击“开始客观题”进入 `/activity`。学习后客观题优先由AI生成，审核通过后使用代码比较答案；动态生成失败时使用少量固定兜底题。

### 8.2 客观题

我看到题号、难度、题干和选项；提交后显示正确/错误、简短解释、来源和“再试一次/看卡片/下一步”。确定性题型由代码判定；短答才调用现有 `gradeAnswer` Graph。每次提交创建兼容旧 `Attempt` 的 `ActivityAttempt` 摘要，重复 `attemptId` 返回原结果。

错误状态必须区分：

- `learner_error`：答案错误或格式不符合要求；
- `evaluator_error`：系统没有成功完成判定，应重试且不降低画像；
- `agent_fallback`：解释生成失败，但固定反馈仍可用。

只读区域、编辑标记和允许库由 Profile 固定；前端不能自行修改 hiddenTestRef、Rubric 或 allowedLibraries。

### 9.2 Run 与 Submit

- **运行**：Pyodide Worker + 公开测试；显示 stdout/stderr、首个错误和提示；不更新 mastery。
- **提交**：Node 受限 Python 子进程（或已通过调研的外部评测器）执行公开/隐藏测试和 Rubric；写入一次 `ActivityAttempt` 与 `ActivityEvidence`。

页面始终显示“运行结果不计入掌握度”“正式提交使用评测器版本 X”。隐藏测试只反馈维度/类别，不显示断言、输入行或参考实现。

### 9.3 执行失败界面

```text
结果：评测器暂时不可用（不是你的代码错误）
原因：测试服务超时
[重试提交] [保存代码并回到卡片] [使用固定练习]
```

语法错误、运行错误、测试失败、超时、输出过大属于 `learner_error`；评测器崩溃、测试读取失败、环境缺库属于 `evaluator_error`。本地 Node 方案只面向受信任演示者，不宣称强恶意代码隔离。

## 10. 我完成独立实操任务

### 10.1 进入任务

我看到业务背景、输入数据字段、输出 contract、验收标准、预计时间和允许 API 范围。实操中可以问“这个错误代表哪一类问题”，但不能要求系统直接给完整答案。脚手架按学习者状态逐步退坡，最终提交必须使用统一 Rubric。

### 10.2 提交反馈

提交结果按 Rubric 分项显示：读取/列、缺失、重复、类型/格式、验证不变量、可读性。隐藏测试失败只显示“重复键仍不唯一”等类别。通过阈值、blocking test 和下一步由服务器返回；前端不自行判断是否完成。

## 12. 从 Pi/TUI 进入 Web

当前 TUI 已能通过 `StudyTuiGateway` 展示卡片、材料、选择题/判断题/简答题、讨论和总结。代码活动在 TUI 中不伪装成可执行：

1. TUI 显示任务标题、预计时间和“需要打开本地 Web 执行器”；
2. Controller 创建一次性 localhost 深链，携带短期 token、`sessionId`、`activityId` 和 `profileRevision`；
3. Web facade 校验 token/session/revision 后打开 Activity 页面；
4. Web 提交后写入共享 private memory；
5. TUI 重新读取 `ActivityAttempt` 摘要并显示下一步。

TUI 不把绝对文件路径、API Key、hiddenTestRef 放入 URL。深链失效时显示“请从 Web 开始页恢复”，不自动新建 session。

## 13. 导航、自动保存和恢复

### 13.1 浏览器返回规则

- 只读页面可直接返回；
- 未提交答案/代码草稿存在时，弹出“保存本地草稿并离开/继续编辑”；
- 已提交结果不允许通过返回重新覆盖；
- 路径确认后返回诊断会创建新版本，不修改已确认 snapshot。

### 13.2 保存分层

| 数据 | 保存位置 | 时机 |
|---|---|---|
| UI route、未提交代码 | 内存 + `localStorage` | 输入节流 500ms |
| 问卷/诊断答案 | private memory/checkpoint | 每题提交/离开页 |
| Attempt/Evidence | private memory | Submit 成功后原子写 |
| Path snapshot | private memory | 确认/重算后 |
| Profile/测试 | Profile family | 维护者启用前，Web 只读 |
| Agent trace/artifact | review repository | 每阶段完成后 |

### 13.3 CAS 和刷新

创建会话只携带`requestId`；创建后的每个写请求携带`requestId/sessionId/sessionVersion/profileRevision`。服务端版本不一致返回 `409 session_version_conflict` 及最新安全view；前端显示“另一窗口已有更新”，让用户选择加载最新或保留本地草稿文本。刷新后只恢复最后完整checkpoint，不能把半个模型输出当作已完成。

## 14. 技术实现架构

### 14.1 运行时组成

```text
React/Vite SPA
  → 本地 Node HTTP adapter（REST JSON，MVP 可轮询）
  → LearningRuntimeFacade（统一入口）
  → application use cases / ReviewOrchestrator
  → domain policy + Profile catalog + repositories
  → Pi adapter 或 Web ModelExecutionPort
```

当前 `createIsolatedGraphExecutor(ctx)` 绑定 `ExtensionCommandContext`，不能直接在浏览器调用。MVP 抽象 `ModelExecutionPort`：Pi 适配器继续使用隔离 AgentSession；Web 适配器读取 OpenAI 兼容配置；两者共享 DTO、审查状态机和持久化。

### 14.2 前端模块

```text
src/web/
├─ app/AppShell.tsx, routes.tsx
├─ state/SessionProvider.tsx, reducers.ts
├─ api/study-facade-client.ts
├─ pages/StartPage.tsx, DiagnosticPage.tsx, PathPage.tsx
├─ pages/LearnPage.tsx, ActivityPage.tsx, SummaryPage.tsx
├─ components/PathSidebar.tsx, EvidencePanel.tsx, ErrorBanner.tsx
└─ workers/pyodide.worker.ts
```

组件只消费安全 DTO；所有 API 请求通过 client 加入 `requestId`、`sessionVersion`、超时和取消信号。页面状态使用 reducer/有限状态机，不用多个布尔值组合出非法状态。

### 14.3 DTO 与 API

```ts
interface SessionView {
  sessionId: string; sessionVersion: number; profileRevision: number;
  mode: "chapter"|"recommended";
  status: "diagnostic"|"path"|"learning"|"activity"|"completed"|"interrupted";
}
interface ActivityView {
  activityId: string; kind: string; title: string; prompt: string;
  publicTests?: PublicTestSummary[]; allowedLibraries: string[];
  leakagePhase: "learning"|"preSubmit"|"inProgress"|"postComplete";
  sourceRefs: SafeSourceRef[]; sessionVersion: number;
}
```

最小 API：

| 方法 | 作用 | 不允许客户端提交 |
|---|---|---|
| `POST /api/sessions` | 创建 session | profile 文件内容、任意 revision |
| `POST /api/sessions/:id/diagnostic` | 提交固定诊断 | 正确答案、权重、阈值 |
| `POST /api/sessions/:id/path` | 确认/重算路径 | 非法节点、先修图 |
| `GET /api/sessions/:id/next` | 获取安全下一步 | hidden assets |
| `POST /api/activities/:id/run` | 公开预览 | hiddenTestRef、Rubric |
| `POST /api/activities/:id/submit` | 正式评测 | 测试文件、参考答案、库白名单 |
| `POST /api/sessions/:id/question` | 上下文提问 | system prompt、来源外任意上下文 |
| `POST /api/sessions/:id/recover` | 恢复 checkpoint | 新 revision 强制重判旧 attempt |

响应统一带 `requestId`、`sessionVersion`、`status`、`errorCode?` 和安全 DTO。MVP 优先 REST + 轮询；SSE 只有在第 5 周验证必要时才作为 `[候选]`，不能为实时多 Agent 并行而引入复杂消息系统。

## 15. 安全、隐私和答案隔离

- API Key 只在 Node 后端环境变量中，绝不进入 React bundle/localStorage/URL。
- 浏览器只能收到脱敏卡片、来源片段、公开测试摘要和安全错误；不收到 hidden tests、参考实现、完整 Agent prompt 或绝对路径。
- ContextAnchor 只保存 artifactId、版本、范围和短摘录；DOM 坐标不能作为唯一定位。
- 用户代码运行于 Pyodide Worker 或 Node 受限子进程；禁网、禁运行时安装白名单外依赖、限制时间/输出/临时目录。Node 方案不承诺抵御恶意本机代码。
- URL 只允许安全的 session/activity/token 标识，所有 ID 使用安全字符校验；不允许路径穿越。

## 16. 可访问性与性能预算

- 所有输入有 label、键盘焦点和错误关联；按钮在请求期间禁用；状态同时使用文字和图标。
- 首屏静态壳目标 <3 秒（本地机器）；普通 facade 请求 p95 <2 秒（不含模型/评测）；Run 反馈目标 <5 秒；Submit 超时上限由任务 manifest 声明。
- 长卡片分页或折叠，stdout/stderr 截断；Worker 初始化失败不阻塞阅读卡片。
- 颜色、动画和 Agent 状态条不能成为唯一信息来源。

## 19. 页面状态与切换的统一规则

前端实现必须先定义状态机，再实现组件，不能由多个 `isLoading/isError/isDone` 布尔值自由组合。每个页面都使用以下状态集合：

```text
idle → loading → ready
                 ├→ saving → ready
                 ├→ submitting → result
                 ├→ validation_error
                 ├→ learner_error
                 ├→ evaluator_error
                 ├→ agent_fallback
                 ├→ conflict
                 └→ offline
```

统一规则如下：

1. `loading` 时显示骨架屏或明确的“正在读取资料”，不能显示空白页面。
2. `saving`/`submitting` 时禁用相同操作按钮，允许取消尚未发出的请求，但不能取消已经产生的正式提交。
3. 错误状态必须保留用户最后一次安全输入，并提供下一步按钮；不能因为请求失败清空答案或代码。
4. `learner_error` 显示可学习的反馈；`evaluator_error` 显示重试/稍后提交，不修改画像。
5. `agent_fallback` 必须标明“使用固定资料版本”，不伪装成实时生成结果。
6. `conflict` 必须显示服务端版本时间和本地草稿状态，由用户选择加载最新或另存文本。
7. 任意页面刷新后先读取 session snapshot，再恢复本地草稿；没有通过版本校验前不能渲染可提交按钮。

## 20. 每个页面的完整界面规格

本节是实现验收清单。每个页面均按“我看到什么 → 我能做什么 → 系统怎么处理 → 去哪里 → 出错怎么办”描述。

### 20.1 开始页和 Profile 空状态

**我看到什么**：页面顶部是产品名和“本地演示模式”提示；中部是 Profile 卡片；卡片下方是学习入口单选、目标输入、可用时间；底部是“开始学习”和“恢复会话”。

**空状态**：没有 active Profile 时显示安装/准备说明、当前数据根目录的安全短名称和“请由维护者运行 Profile 构建”提示；不暴露绝对路径，不提供任意上传按钮。

**切换**：

| 我点击 | 系统动作 | 下一页 |
|---|---|---|
| 开始学习 | 校验 profile、创建 session | 诊断或路径 |
| 查看包详情 | 读取安全 manifest/能力 | 返回开始页 |
| 恢复会话 | 列 pending batch/checkpoint | 恢复页 |
| 关闭 | 清除临时 UI 状态，不删 private memory | 仍在开始页 |

### 20.2 诊断问卷、固定题和结果页

诊断拆成三个可恢复子页面，而不是一个长表单：

```text
/diagnostic/:id/background
        ↓ 下一步（保存 draft）
/diagnostic/:id/questions/:index
        ↓ 全部完成
/diagnostic/:id/result
        ↓ 确认
/path/:id
```

**问卷页**显示题目总数、当前页、必填/可跳过标识和“不确定”。保存的是 `backgroundDraft`，不产生 mastery。

**题目页**显示题干、选项/答案框、上一题、跳过、保存并退出；一次只提交一个答案。跳过时二次确认“跳过会把该知识点标为未验证”，但不阻止继续。

**结果页**显示证据数量、三类知识状态、置信度解释和“查看证据”折叠区。不能显示未经验证的精确百分比或雷达图；Diagnosis Agent 超时显示确定性模板。

### 20.3 路径确认和路径变更对话框

路径页分为：路径摘要、节点列表、侧重点选择、时间预算提示和确认栏。每个节点至少显示 `nodeId` 的安全短标签、知识点、预计时间、难度、先修状态、活动数量和 `reasonCode` 的人话解释。

当我调整模式或时间时，页面先显示“正在重新计算”，旧路径保持可见但标记为 `recalculating`；新路径通过 schema 和先修检查后才替换。若计算失败，保留旧路径并提供“继续旧路径/返回开始页”。

点击后置锁定节点时，弹出：

```text
还不能直接进入“最终验证”
还缺少：处理缺失值、处理重复值的直接证据
[先做一个 5 分钟验证] [返回路径]
```

### 20.4 学习卡片页的展开层级

卡片不是一整段 Markdown 直接塞入页面，固定为四层：

1. **摘要层**：目标、为什么现在学、预计时长。
2. **理解层**：概念、分步说明、示例、常见误区。
3. **验证层**：无分自测、活动入口、提示等级。
4. **依据层**：claim、sourceAnchor、原始文件/片段定位、Profile revision。

我点击“依据”才展开来源；我选中文字才出现“提问”。卡片加载时先显示固定 skeleton；Agent 审核期间按钮显示阶段文本；fallback 时保留同样的四层结构，保证页面不跳变。

### 20.5 客观题和短答结果页

客观题页面底部固定三个区域：提交区、判定区、下一步区。提交前不显示答案；提交后判定区显示结果、解释、来源和错误标签。

```text
[题干与选项]
                         [提交答案]
────────────────────────────────
结果：答错（错误标签：missing_strategy）
依据：sourceAnchor ...
下一步：查看缺失值卡片 / 再试一次 / 继续
```

短答调用现有 `gradeAnswer` Graph；Graph 无结果时显示“暂时无法判定”，不把模型拒答写成 gave_up。提交结果通过 `Attempt` adapter 兼容现有 private memory。

### 20.6 代码补全页的三种状态

代码页必须明确区分：

1. **编辑状态**：显示只读区域、可编辑区域、库版本、题目要求和本地草稿保存提示。
2. **运行状态**：显示 Worker/Node 后端、运行计时、公开测试进度、stdout/stderr；取消只终止预览。
3. **提交结果状态**：显示最终状态、Rubric 摘要、错误类别、评测器版本、下一步；不显示隐藏测试。

按钮规则：Run 可以重复；Submit 在同一 `attemptId` 完成前不可重复；提交完成后“再提交”必须创建新 `attemptId`，不能覆盖旧证据。

### 20.7 独立实操页的分区

独立实操比代码补全多显示四个区域：业务目标、数据字典、验收清单、提交历史。页面左侧是任务说明，中央是编辑器，右侧是“允许提问/提示等级”和 Rubric 进度，底部是运行/提交。

我可以逐项勾选“已检查列名/已处理缺失/已验证唯一性”，但这些勾选只是 UI 辅助，不产生证据；只有正式评测结果才能更新状态。

### 20.8 总结页的继续分支

总结页不能只有一个“结束”按钮，至少提供：

| 操作 | 处理 |
|---|---|
| 继续当前路径 | 从第一个 `available` 或 `needs_review` 节点进入 |
| 做迁移任务 | 使用同一知识点但不同合成数据，标记迁移活动 |
| 复习薄弱点 | 由 PathEngine 生成 gap-review 后缀 |
| 重新诊断 | 新建 diagnostic version，旧证据保留 |
| 结束会话 | 原子写 summary、session status=completed |

### 20.9 Agent 审核详情页

审核页只用于演示和调试授权，不是普通学习者必经页面。它显示一个竖向时间线：

```text
生成草稿 ✓
  ↓
Hunter 核查 ✓（2 个低风险提示）
  ↓
Defender 未触发（无实质争议）
  ↓
Judge 定稿 ✓
  ↓
固定规则门禁 ✓ → 已发布
```

每步显示开始/结束时间、耗时、状态、问题数量、改动摘要和安全来源数；不显示 system prompt、完整用户答案、隐藏测试和模型秘钥。严格按串行顺序刷新，不同时显示多个运行中的 Agent。

### 20.10 恢复页和冲突页

恢复页按时间倒序显示 checkpoint：会话、Profile revision、最后完成阶段、未提交草稿是否存在、最近错误。

如果 Profile revision 不一致：

```text
本会话使用 Profile v2.0，当前 active 已是 v2.1
已保存的旧活动不会用新测试重判。
[继续旧版本会话] [以新版本开始新会话] [导出草稿文本]
```

如果是 CAS 冲突：显示服务端更新时间和本地修改时间，不自动合并代码或 Agent 产物。

## 21. 页面切换与数据流时序

### 21.1 开始到路径

```text
我点击开始
 → React 校验表单
 → POST /api/sessions
 → Facade 锁定 Profile revision
 → private memory 创建 session/checkpoint
 → 返回 SessionView
 → RouteGuard 决定 diagnostic/path
```

### 21.2 提交活动到下一步

```text
我点击 Submit
 → 前端生成 requestId/attemptId
 → Facade 校验 activity 归属和 sessionVersion
 → CodeEvaluationPort 执行
 → 返回 ActivityResult
 → 仅成功/learner_error 写 ActivityAttempt/Evidence
 → EvidenceAggregator 更新 KnowledgeState
 → PathEngine 只重排未完成后缀
 → 返回新 pathVersion 和 changeReasons
 → 页面显示结果并允许下一步
```

### 21.3 页面关闭和恢复

```text
页面关闭/刷新
 → localStorage 恢复 route + 未提交草稿
 → GET session snapshot/checkpoint
 → 校验 profileRevision/sessionVersion
 → 有完整 checkpoint 才恢复提交能力
 → 否则进入恢复页
```

## 23. 空、加载、失败和恢复界面文案

| 情况 | 页面必须显示 | 可用操作 | 禁止文案 |

| 无 Profile | “当前没有可学习的 active 资料包” | 查看准备说明 | “系统错误” |
| 模型超时 | “生成超时，已保留固定版本/可重试” | 重试、继续固定版本 | “你操作太慢” |
| 路径不可行 | 缺少先修/预算原因 | 做验证、返回选择 | 静默跳过先修 |
| 代码语法错 | 错误位置和概念提示 | 修改、再运行 | 展示参考答案 |
| 评测器异常 | “不是你的代码错误” | 重试、保存、fallback | “提交失败，请重写代码” |
| 磁盘写失败 | “进度未确认保存” | 重试、导出草稿 | “已保存” |
| CAS 冲突 | 新旧版本时间 | 加载最新、保留草稿 | 自动覆盖 |
| 审核 fallback | “使用固定审核版本” | 查看依据、继续 | 假装实时生成成功 |

## 24. 产品走查验收清单

走查人员按三个角色执行：第一次学习者、从 TUI 转 Web 的学习者、发榜公司评测人员。每项必须截图或录屏留证：

1. 开始页能看懂资料包状态，不能上传任意文件。
2. 诊断每题可保存、跳过和恢复，结果能看到证据不足。
3. 路径页能解释节点安排，不能绕过先修。
4. 学习卡片能展开来源和进入活动，审核状态严格串行。
5. 客观题、代码补全、独立实操三种界面均显示完整的提交前/提交中/提交后状态。
6. Run 与 Submit 的后端、计分和错误语义清楚可见。
7. 总结页提供继续、迁移、复习、重新诊断和结束分支。
8. 刷新、关闭、双窗口冲突和 Profile revision 变化都有安全页面。
9. Web/TUI 使用同一 session 和 evidence，不产生两套进度。
10. 浏览器 DOM、网络响应、日志和 URL 均不含 API Key、hidden tests、参考答案或绝对路径。
