# 子图调用边界：call、compose 与 delegate

Graph 可以把另一张 Graph 当作节点调用。调用边界决定父图和子图共享执行会话、逻辑工作实例与工作记忆的程度。

如果只记住一句话：默认使用 call；需要“以图代点”并共享父级工作记忆时使用 compose；需要独立会话和最强隔离时使用 delegate。

术语定义见[领域语言](glossary.md)。数据通道见[上下文与状态](context-and-state.md)。

## 边界对比

| 边界 | 执行会话 Session | AgentInstance | 父级 frames | Mechanism state | 子图返回方式 | 典型用途 |
| --- | --- | --- | --- | --- | --- | --- |
| call（默认） | 复用 | 新建 | 不可见 | 新建 | 状态和结果 | 独立子任务、可复用业务模块 |
| compose | 复用 | 复用 | 可见 | 复用 | 临时帧段归约后返回 | 用一张图实现父节点内部流程 |
| delegate | 新建 | 新建 | 不可见 | 新建 | 隔离执行所得的状态和结果 | 强隔离、长任务、独立压缩生命周期 |

三种边界都只通过明确输入启动子图。Runtime 不会把调用方的局部变量作为子图输入；虽然普通 JavaScript 节点函数仍可能捕获闭包或模块变量，但不应依赖这种方式传递图间业务数据。

## Call：同一会话中的独立工作区

call 是 graph node 的默认边界。它复用当前执行会话，但创建新的 AgentInstance。

这意味着：

- 子图与父图使用同一个模型和消息承载环境。
- 子图看不到父图的 frames。
- 子图拥有新的 Mechanism state 和 scratch。
- 父子双方只通过调用输入和最终结果交换业务数据。

### 场景：复用“审查并给出结论”子图

多个父图都需要调用同一套审查流程，但不希望审查子图依赖父图的完整历史。使用 call 可以把待审查内容作为输入传给子图，只接收最终审查结果。

call 适合边界清晰的业务模块，也是没有特殊共享需求时的首选。

## Compose：在当前工作区中以图代点

compose 复用当前 Session 和 AgentInstance。子图可以读取父图已经完成的 frames，并在同一工作记忆上继续工作。

为了避免子图内部阶段永久展开到父图，compose 会把子图运行期间新增的工作记忆视为临时帧段。子图返回时，这段记忆会归约为父级 graph node 的一次完成结果：可以提供自定义 `fold`，省略时使用 SDK 的默认归约；失败时则回滚临时帧段。

### 场景：把复杂节点拆成内部小图

父图中有一个“修改文档”阶段，内部实际包含读取、计划、编辑和复查。调用方仍希望把它视为一个节点，同时内部步骤需要看到父图已有的审查结论。

此时 compose 比 call 更合适：它共享父级工作记忆，但退出后只留下归约结果，不让内部四个阶段永久污染父图结构。

compose 不是普通的子任务隔离方式。只有确实需要共享父级 frames 或把图作为节点实现细节时才使用它。

## Delegate：新的独立执行会话

delegate 创建新的执行会话和新的 AgentInstance。独立 host 内部产出 `GraphRunResult`，父图中的 graph node 最终只接收其中的状态和业务结果；父子双方不共享工作记忆。

这意味着：

- 模型消息、工具运行和上下文压缩具有独立生命周期。
- 父级 transcript 和 frames 不会进入子会话。
- 子图需要单独配置可用工具和执行 host。
- 子图的取消与资源释放由 delegate host 管理。

### 场景：运行时间较长的独立任务

例如父图需要委托一个可能经历大量 Agent turn 的代码迁移任务。该任务需要独立上下文压缩，不应让中间对话进入父会话。

使用 delegate 可以获得最强隔离。代价是需要创建独立 host，并显式传递所需工具、配置和输入。

delegate 提供隔离边界，但当前 SDK 不因此自动获得多 Agent 通讯或 fork/join 并行编排能力。

## 如何选择

按顺序回答以下问题：

1. 子图是否必须使用独立 AgentSession？如果是，选择 delegate。
2. 子图是否必须读取父图已完成的 frames，或需要作为父节点的内部实现展开？如果是，选择 compose。
3. 否则选择 call。

```text
需要独立 Session？ ──是──→ delegate
       │否
       ↓
需要共享父级工作记忆？ ──是──→ compose
       │否
       ↓
      call
```

## 输入、工作记忆与返回

无论使用哪种边界，父图都应明确构造子图 background。不要依赖闭包或模块变量偷偷传递业务状态。

返回时：

- call 和 delegate 不把子图 frames 带回父图，只返回稳定结果。
- compose 使用临时帧段，并在返回时归约为父 graph node 的 Completion；可以自定义 `fold`，也可以使用默认归约。
- 父图是否把子图结果保存为长期工作记忆，仍由父图离开该 graph node 时选中的 Edge 决定。

## 上下文压缩的考虑

call 和 compose 共享当前 Session。为了避免压缩摘要混合父图与活动子图的内部对话，共享会话中的嵌套调用会限制压缩跨越调用边界。

需要独立、长时间运行并拥有自身压缩生命周期的子图，应优先使用 delegate。

## 常见误选

### 为了“代码复用”一律使用 compose

代码复用不等于共享工作记忆。只需要输入和输出契约时，call 更清晰。

### 认为 call 会创建新 Session

call 只创建新的 AgentInstance，仍复用当前 Session。需要物理会话隔离时使用 delegate。

### 认为 delegate 等于并行图

delegate 允许独立 host 承载任务，但当前图运行仍沿单一路径等待其结果；它不是 fork/join 调度器。

### 让 compose 子图的内部 frames 永久留在父图

compose 的临时帧段会在退出时归约，而不是永久留在父图。父图只应看到 graph node 对外提交的完成结果和随后由 Edge 保存的工作记忆。
