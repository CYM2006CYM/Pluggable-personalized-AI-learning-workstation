# 多 Agent 通讯研究提案

> **状态：未实现。**
>
> 本文不属于 Loop Graph SDK 0.1.x 的能力承诺。文中的通讯模型、接口形态和数据结构均为研究提案，不能作为当前公共 API 或实现事实引用。

## 研究问题

当前 SDK 在一次图运行中串行推进单一节点路径。delegate 可以把子图放入独立 Session，但父图仍等待稳定 GraphRunResult；它不提供多个 Agent 的寻址、持续通讯、共享任务或 fork/join 调度。

如果未来引入多 Agent，需要先回答：

1. 多个 Agent 是同一图中的并行分支，还是多个独立图运行？
2. 谁拥有 Agent 身份和生命周期？
3. 消息是否进入模型上下文，何时被折叠或丢弃？
4. 通讯失败是业务结果、节点失败还是图失败？
5. 如何避免共享 Session、工具和状态造成跨 Agent 泄漏？
6. 调度、背压、取消和审计由哪个层次负责？

在这些问题有真实场景和实现约束前，不应把 ROS2、消息总线或共享状态接口写进当前 Graph、Node、AgentInstance 或 Mechanism 类型。

## 当前设计提供的基础

以下现有能力可能成为未来工作的基础，但不等同于多 Agent 通讯：

- GraphRunRequest/GraphRunResult 提供稳定调用边界。
- delegate 提供独立 Session 与 AgentInstance。
- traceSink 提供生命周期审计出口。
- Mechanism 提供节点内横切 Hook 和受控工具门禁。
- ContextFrame 提供显式、可折叠的工作记忆。

未来方案必须复用这些边界，而不是通过全局变量或裸 transcript 共享绕过它们。

## 候选能力分层

研究时可把能力分成三个层次，避免一次引入完整通讯平台。

### 1. 调度层

负责创建、等待、取消和汇总多个独立图运行。第一版更可能是受限 fork/join，而不是任意动态图调度。

需要定义：并发上限、父级取消传播、部分失败、超时以及结果归约。

### 2. 交换层

负责 Agent 之间显式交换消息或请求。候选原语可以包括：

- 单向消息
- 请求/回复
- 长任务进度

每条消息必须有来源、目标、关联 ID、时间和大小预算。是否进入模型上下文必须显式决定。

### 3. 共享数据层

负责多个运行读取或更新外部共享状态。它应被视为独立存储服务，而不是把可变对象挂到 AgentInstance 上。

需要定义一致性、冲突、权限、过期和审计语义。

## 不应提前承诺的设计

- 不假设所有 Agent 共享同一个 Session。
- 不把 publish/subscribe 等工具直接加入当前 Node 类型。
- 不把订阅列表、pending goals 或共享状态塞进当前 AgentInstance。
- 不把 Mechanism state 升级成跨 Agent 业务数据库。
- 不把 delegate 描述为已经支持并行协作。
- 不把通讯消息自动加入 ContextFrame 或模型上下文。

## 必须继承的安全边界

任何未来方案都应保持：

- AgentInstance 与 Session 分离。
- 业务数据通过显式协议交换。
- Runtime 控制身份不由模型正文声明。
- 消息和结果有大小、超时和取消边界。
- 已关闭任务的内部 transcript 不泄漏给其他 Agent。
- 共享状态不绕过权限和审计。
- 单个 Agent 的图上状态迁移仍由 Router 和 Edge 决定。

## 建议的验证顺序

在设计公共 API 前，先用独立实验验证：

1. 两个 delegate host 并发运行，Session 和结果不串线。
2. 父级取消可以可靠终止所有子 host。
3. 有界消息通道不会把一个 Agent transcript 暴露给另一个。
4. 一个子任务失败时，父级能够选择 fail-fast 或收集部分结果。
5. 审计记录可以解释谁在何时向谁发送了什么，而不依赖模型自报。

只有出现真实业务案例并通过这些验证后，才应提出正式 ADR 和公共类型。
